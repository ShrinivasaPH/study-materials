Section 01

What is NoSQL?

NoSQL stands for "Not Only SQL". It is a class of database management systems that store and retrieve data in ways other than tabular relations used in relational (SQL) databases. NoSQL databases are designed for large-scale, distributed data where flexibility, speed, and horizontal scalability matter more than strict schema enforcement.

Core Philosophy
NoSQL = Schema-Flexibility + Horizontal Scale + High Throughput
Schema-Flexibility

Each record can have different fields — no rigid table structure required.

Horizontal Scale

Add more machines (nodes) to scale out, rather than upgrading one big server.

High Throughput

Optimized for millions of reads/writes per second with low latency.

Section 02

The Four Pillars of NoSQL

Each NoSQL type solves a different class of problem. Choosing the right one is half the battle.

📄

Document Store

Stores semi-structured JSON/BSON documents. Each document is self-describing and can nest sub-documents and arrays.

{
  "_id": "usr_42",
  "name": "Priya",
  "orders": [
    { "id": "o1", "total": 299 }
  ]
}
  • MongoDB
  • CouchDB
  • Firestore
  • DynamoDB
⚡

Key-Value Store

The simplest NoSQL model. A giant, distributed hash-map. Lightning-fast lookups by key. Values are opaque blobs.

# SET / GET in Redis
SET session:42 "user_data"
GET session:42
TTL session:42 → 3600s
  • Redis
  • DynamoDB
  • Riak
  • Memcached
📊

Column-Family Store

Organizes data into rows with dynamic column families. Ideal for write-heavy analytical workloads and time-series data.

Row Key → Column Families
user:42 →
  info: { name, email }
  activity: { last_login, views }
  prefs: { theme, lang }
  • Cassandra
  • HBase
  • Bigtable
  • ScyllaDB
🕸️

Graph Database

Models data as nodes (entities) and edges (relationships). Native traversal queries are magnitudes faster than SQL JOINs for connected data.

// Cypher (Neo4j)
(Priya)-[:FOLLOWS]->(Ravi)
(Priya)-[:LIKES]->(Post:42)
(Ravi)-[:AUTHORED]->(Post:42)
  • Neo4j
  • Amazon Neptune
  • ArangoDB
  • TigerGraph
⏱️

Time-Series DB

Optimized for sequential, time-stamped data. Efficient compression, downsampling, and range queries over time windows.

# InfluxDB line protocol
cpu,host=server01
  usage=72.3 1704067200000000000
memory,host=server01
  free=4096 1704067200000000000
  • InfluxDB
  • TimescaleDB
  • Prometheus
  • QuestDB
Section 03

The CAP Theorem

Proposed by Eric Brewer (2000), the CAP Theorem states that a distributed system can guarantee at most two out of three properties simultaneously.

CAP Theorem
∀ distributed system S: |{C, A, P} ∩ guarantee(S)| ≤ 2
C — Consistency

Every read receives the most recent write or an error. All nodes see the same data at the same time.

A — Availability

Every request receives a (non-error) response — not necessarily the most recent data.

P — Partition Tolerance

The system continues operating even if network messages are dropped between nodes.

CA
Consistent + Available
No partition tolerance. Works on a single machine or tightly-coupled cluster. Example: PostgreSQL, MySQL
CP
Consistent + Partition Tolerant
May reject requests during partitions to maintain consistency. Example: MongoDB, HBase, Redis
AP
Available + Partition Tolerant
Always responds but data might be stale (eventual consistency). Example: Cassandra, CouchDB, DynamoDB

Section 04

Real-Life Use Cases

Every NoSQL type solves a concrete class of problems. Here's where each shines — with the problem it solved.

Document DB 🛍️

E-Commerce Product Catalog

Amazon, Flipkart — millions of products, each with wildly different attributes (a shirt has size/color, a laptop has RAM/GPU).

Problem SQL tables can't handle 50 different attributes per product type without nullable columns or complex EAV patterns.
Solution MongoDB stores each product as a document with its own schema — shirts have {"size":"L"}, laptops have {"ram":"16GB"}. No wasted nulls.
Key-Value 🔐

Session Management

Web apps like Netflix, Swiggy — millions of users logged in simultaneously with active sessions.

Problem Querying a SQL DB for session data on every API call adds 20-50ms latency. Under 10M concurrent users, this breaks.
Solution Redis stores session:userID → {cart, preferences} in RAM. Sub-millisecond reads. TTL auto-expires stale sessions.
Column-Family 📱

WhatsApp Message History

100 billion messages per day — write-heavy, time-ordered, accessed by conversation ID.

Problem SQL can't sustain 10M writes/sec. Sharding is painful. Reads by conversation ID need full table scans without perfect indexing.
Solution Cassandra (used by WhatsApp/Discord) partitions by chat_id — all messages in a conversation are co-located. Appends are O(1).
Graph DB 👥

LinkedIn — "People You May Know"

Finding friends-of-friends 3 hops away across 900M users.

Problem SQL multi-join for 3-hop traversal on 900M users generates billions of intermediate rows. Query time: minutes or crashes.
Solution Neo4j traverses relationships in milliseconds. Finding 2nd-degree connections is pointer-following — no joins needed.
Key-Value 🎮

Gaming Leaderboards

Real-time ranking of 5M players in a live multiplayer game.

Problem Updating and querying a sorted rank table in SQL under 10,000 score updates/sec requires constant re-indexing.
Solution Redis Sorted Sets (ZADD/ZRANK) maintain a live, auto-sorted leaderboard. Rank lookup is O(log N). Top-100 query is O(100).
Document DB 🏥

Electronic Health Records

Patient records with vastly different structure — cardiologist notes vs. dermatology reports vs. lab results.

Problem A fixed SQL schema forces all specialties into a single table — 90% of columns are NULL for any given patient visit.
Solution MongoDB stores each visit as a rich nested document — labs embedded in the record, no JOINs needed to retrieve a full patient history.

mongodb_aggregation.py
▶ Open Colab 
# MongoDB Aggregation Pipeline
# Pipeline = [ Stage1, Stage2, Stage3, ... ]
# Data flows left→right through each stage

from pymongo import MongoClient

client = MongoClient("mongodb://localhost:27017/")
orders = client["ecommerce"]["orders"]

pipeline = [
    # Stage 1 — $match: Filter (like WHERE)
    {"$match": {
        "status": "delivered",
        "order_date": {"$gte": "2024-01-01"}
    }},

    # Stage 2 — $group: Aggregate (like GROUP BY)
    {"$group": {
        "_id": "$category",          # Group by category
        "total_revenue": {"$sum": "$amount"},
        "avg_order":    {"$avg": "$amount"},
        "order_count":  {"$count": {}}
    }},

    # Stage 3 — $sort: Order results (like ORDER BY)
    {"$sort": {"total_revenue": -1}},  # -1 = descending

    # Stage 4 — $limit: Top N (like LIMIT)
    {"$limit": 5},

    # Stage 5 — $project: Select fields (like SELECT)
    {"$project": {
        "category": "$_id",
        "total_revenue": {"$round": ["$total_revenue", 2]},
        "avg_order": 1,
        "order_count": 1,
        "_id": 0
    }}
]

results = orders.aggregate(pipeline)
for r in results:
    print(r)

# ── Pipeline Analogy ────────────────────────────────────────────────────────
# SQL Equivalent:
# SELECT category, ROUND(SUM(amount),2) AS total_revenue,
#        AVG(amount) AS avg_order, COUNT(*) AS order_count
# FROM orders
# WHERE status = 'delivered' AND order_date >= '2024-01-01'
# GROUP BY category
# ORDER BY total_revenue DESC
# LIMIT 5;
mongodb_indexing.py
▶ Open Colab 
# ── MongoDB Indexing ───────────────────────────────────────────────────────
# Without index: O(n) — full collection scan
# With index:    O(log n) — B-tree traversal

from pymongo import MongoClient, ASCENDING, DESCENDING, TEXT

col = MongoClient()["ecommerce"]["products"]

# ── Single field index ─────────────────────────────────────────────────────
col.create_index([("price", ASCENDING)])
# Now: col.find({"price": {"$lt": 1000}}) → O(log n)

# ── Compound index (most selective field first) ─────────────────────────────
col.create_index([("category", ASCENDING), ("price", ASCENDING)])
# Covers: col.find({"category":"electronics","price":{"$lt":50000}})

# ── Text index (full-text search) ──────────────────────────────────────────
col.create_index([("name", TEXT), ("description", TEXT)])
col.find({"$text": {"$search": "gaming laptop"}})

# ── Unique index ───────────────────────────────────────────────────────────
col.create_index("sku", unique=True)

# ── TTL index (auto-expire documents) ─────────────────────────────────────
sessions = MongoClient()["app"]["sessions"]
sessions.create_index("created_at", expireAfterSeconds=3600)
# Documents with created_at older than 1 hour are auto-deleted!

# ── View all indexes ───────────────────────────────────────────────────────
print(list(col.list_indexes()))

# ── Explain query performance ──────────────────────────────────────────────
explanation = col.find({"price": {"$lt": 1000}}).explain()
print("Plan:", explanation["queryPlanner"]["winningPlan"])
# Look for "IXSCAN" (index) vs "COLLSCAN" (slow full scan)

# ── COUNTER (Atomic increment) ─────────────────────────────────────────────
r.set("page_views:home", 0)
r.incr("page_views:home")                  # Atomic +1
r.incrby("page_views:home", 100)           # Atomic +N
print(r.get("page_views:home"))            # → 101

# ── HASH: Object store (like a row in a table) ─────────────────────────────
r.hset("user:42", mapping={
    "name":  "Priya Sharma",
    "email": "priya@example.com",
    "score": "1500"
})

print(r.hget("user:42", "name"))          # → Priya Sharma
print(r.hgetall("user:42"))               # → All fields
r.hset("user:42", "score", "1600")       # Update single field
r.hincrby("user:42", "score", 50)         # Increment numeric field

# ── KEY NAMING CONVENTION ──────────────────────────────────────────────────
# Pattern:  object_type:id:field
# Examples: user:42, session:abc123, product:99:views
# Use ":"  as namespace separator (Redis convention)

import fakeredis
r = fakeredis.FakeRedis(decode_responses=True)

# ── LIST: Message queue / activity feed ────────────────────────────────────
# LPUSH = push to LEFT (head), RPUSH = push to RIGHT (tail)
r.rpush("feed:42", "Priya liked your post")
r.rpush("feed:42", "Ravi commented: Great!")
r.rpush("feed:42", "You have 5 new followers")

# Get latest 10 items (0 = first, -1 = last)
print(r.lrange("feed:42", -10, -1))

# BLPOP = blocking pop (consumer waits for messages)
# msg = r.blpop("task_queue", timeout=5)

# ── SET: Unique membership (tags, followers) ───────────────────────────────
r.sadd("tags:post:1", "python", "nosql", "database")
r.sadd("tags:post:2", "python", "ml", "ai")

# Set operations: SINTER, SUNION, SDIFF
print(r.sinter("tags:post:1", "tags:post:2"))  # → {'python'}
print(r.sunion("tags:post:1", "tags:post:2"))  # → all unique tags
print(r.sismember("tags:post:1", "python"))    # → True

# ── SORTED SET: Leaderboard ────────────────────────────────────────────────
# ZADD key score member — auto-sorted by score
r.zadd("leaderboard", {
    "Priya": 9850,
    "Ravi":  9200,
    "Meera": 9750,
    "Arjun": 10100
})

# Top 3 players (highest score first)
top3 = r.zrevrange("leaderboard", 0, 2, withscores=True)
print("🏆 Top 3:", top3)

# Rank of a player (0-indexed)
rank = r.zrevrank("leaderboard", "Priya")
print(f"Priya's rank: #{rank + 1}")

# ── BITMAP: Efficient boolean flags ───────────────────────────────────────
# Track which users opened email (1 bit per user!)
r.setbit("email_opened:campaign_1", 42, 1)   # user 42 opened
r.setbit("email_opened:campaign_1", 99, 1)   # user 99 opened
print(r.bitcount("email_opened:campaign_1"))   # → 2 total opens
redis_caching_pattern.py
▶ Open Colab 
import fakeredis, json, time

r = fakeredis.FakeRedis(decode_responses=True)

# ── Cache-Aside Pattern (most common) ─────────────────────────────────────
# Logic: Read from cache → miss? → query DB → populate cache → return
#
#   ┌──────────────────────────────────────────────────────────────┐
#   │  App → CACHE HIT? → YES → Return                             │
#   │          ↓ NO                                                 │
#   │       Query DB → Store in Cache (TTL) → Return               │
#   └──────────────────────────────────────────────────────────────┘

def fake_db_query(user_id: int) -> dict:
    """Simulates slow DB query (200ms)"""
    time.sleep(0.2)
    return {"id": user_id, "name": "Priya", "plan": "pro"}

def get_user(user_id: int) -> dict:
    cache_key = f"user:{user_id}"

    # 1. Try cache first
    cached = r.get(cache_key)
    if cached:
        print("⚡ Cache HIT")
        return json.loads(cached)

    # 2. Cache miss — query DB
    print("🐌 Cache MISS — querying DB")
    data = fake_db_query(user_id)

    # 3. Store in cache with TTL (5 minutes)
    r.setex(cache_key, 300, json.dumps(data))
    return data

# First call — hits DB (slow)
start = time.time()
user  = get_user(42)
print(f"First call:  {(time.time()-start)*1000:.0f}ms → {user}")

# Second call — hits cache (fast!)
start = time.time()
user  = get_user(42)
print(f"Second call: {(time.time()-start)*1000:.0f}ms → {user}")

# ── Rate Limiting Pattern ──────────────────────────────────────────────────
def is_rate_limited(user_id: int, limit: int = 5, window: int = 60) -> bool:
    key    = f"rate:{user_id}"
    count  = r.incr(key)
    if count == 1:
        r.expire(key, window)       # Set window on first request
    return count > limit

for i in range(7):
    limited = is_rate_limited(42)
    print(f"Request {i+1}: {'❌ BLOCKED' if limited else '✅ OK'}")

cassandra_messages.py

# pip install cassandra-driver
# For Colab: pip install cassandra-driver aiohttp
# Use DataStax Astra free tier or local Docker:
# docker run -p 9042:9042 cassandra

from cassandra.cluster import Cluster
from cassandra import ConsistencyLevel
import uuid, datetime

cluster = Cluster(['127.0.0.1'])
session = cluster.connect()

# ── Keyspace (≈ Database in Cassandra) ─────────────────────────────────────
session.execute("""
    CREATE KEYSPACE IF NOT EXISTS messaging
    WITH REPLICATION = {
        'class': 'NetworkTopologyStrategy',
        'datacenter1': 3           -- 3 replicas per datacenter
    }
""")
session.set_keyspace('messaging')

# ── Table Design: Query-First Principle ────────────────────────────────────
# Cassandra forces you to design tables around your READ queries.
# Primary Key = (Partition Key, Clustering Columns)
# Partition Key → determines which node stores the row
# Clustering Columns → sort order within a partition

session.execute("""
    CREATE TABLE IF NOT EXISTS messages_by_chat (
        chat_id     UUID,
        sent_at     TIMESTAMP,
        message_id  UUID,
        sender_id   UUID,
        content     TEXT,
        media_url   TEXT,
        PRIMARY KEY ((chat_id), sent_at, message_id)
    )
    WITH CLUSTERING ORDER BY (sent_at DESC)   -- Latest messages first
    AND  default_time_to_live = 7776000        -- Auto-delete after 90 days
""")

# ── INSERT ─────────────────────────────────────────────────────────────────
chat_id = uuid.uuid4()
session.execute("""
    INSERT INTO messages_by_chat
    (chat_id, sent_at, message_id, sender_id, content)
    VALUES (%s, %s, %s, %s, %s)
""", (chat_id, datetime.datetime.utcnow(),
        uuid.uuid4(), uuid.uuid4(), "Hey! How are you?"))

# ── QUERY: Last 20 messages in a chat ──────────────────────────────────────
# This is O(1) — directly hits the correct partition!
rows = session.execute(
    "SELECT * FROM messages_by_chat WHERE chat_id=%s LIMIT 20",
    (chat_id,)
)

for row in rows:
    print(f"[{row.sent_at}] {row.content}")

# ── Consistency Levels ─────────────────────────────────────────────────────
# ONE   → Fastest, least consistent (1 replica responds)
# QUORUM → Majority of replicas (N/2 + 1) — balanced
# ALL   → Slowest, most consistent (all replicas respond)

from cassandra.query import SimpleStatement
stmt = SimpleStatement(
    "SELECT * FROM messages_by_chat WHERE chat_id=%s",
    consistency_level=ConsistencyLevel.QUORUM
)

neo4j_social_graph.py
▶ Open Colab 
# pip install neo4j
# Free sandbox: https://sandbox.neo4j.com

from neo4j import GraphDatabase

driver  = GraphDatabase.driver("bolt://localhost:7687", auth=("neo4j", "password"))
session = driver.session()

# ── CREATE nodes and relationships ─────────────────────────────────────────
# Notation: (Node:Label {properties})-[:RELATIONSHIP]->(Node)

session.run("""
    CREATE (priya:User  {name: 'Priya',  city: 'Bengaluru', score: 9850})
    CREATE (ravi:User   {name: 'Ravi',   city: 'Mumbai',    score: 9200})
    CREATE (meera:User  {name: 'Meera',  city: 'Bengaluru', score: 9750})
    CREATE (python:Tag  {name: 'Python'})
    CREATE (nosql:Tag   {name: 'NoSQL'})
    CREATE (post1:Post  {title: 'Intro to NoSQL', likes: 142})

    CREATE (priya)-[:FOLLOWS]->(ravi)
    CREATE (priya)-[:FOLLOWS]->(meera)
    CREATE (ravi)-[:FOLLOWS]->(meera)
    CREATE (priya)-[:AUTHORED]->(post1)
    CREATE (priya)-[:TAGGED]->(python)
    CREATE (post1)-[:HAS_TAG]->(nosql)
    CREATE (ravi)-[:LIKED]->(post1)
    CREATE (meera)-[:LIKED]->(post1)
""")

# ── MATCH: Find friends-of-friends (2 hops) ───────────────────────────────
result = session.run("""
    MATCH (me:User {name: 'Priya'})-[:FOLLOWS*2]->(fof:User)
    WHERE fof <> me AND NOT (me)-[:FOLLOWS]->(fof)
    RETURN DISTINCT fof.name AS suggestion, fof.city AS city
""")
print("People You May Know:")
for row in result:
    print(f"  {row['suggestion']} — {row['city']}")

# ── Shortest Path between two users ───────────────────────────────────────
path = session.run("""
    MATCH p = shortestPath(
        (a:User {name:'Priya'})-[:FOLLOWS*]-(b:User {name:'Meera'})
    )
    RETURN length(p) AS hops, [n IN nodes(p) | n.name] AS path
""")
for row in path:
    print(f"Shortest path ({row['hops']} hops): {' → '.join(row['path'])}")

# ── Recommendation: Posts liked by people you follow ──────────────────────
recs = session.run("""
    MATCH (me:User {name:'Priya'})-[:FOLLOWS]->(friend)-[:LIKED]->(post:Post)
    WHERE NOT (me)-[:LIKED]->(post)
    RETURN post.title AS title, COUNT(friend) AS mutual_likes
    ORDER BY mutual_likes DESC
    LIMIT 5
""")
for r in recs:
    print(f"📌 {r['title']} — liked by {r['mutual_likes']} friends")

# ── Cypher Cheatsheet ─────────────────────────────────────────────────────
# MATCH    → Find patterns (like SELECT + JOIN)
# WHERE    → Filter
# CREATE   → Insert nodes/relationships
# MERGE    → Create if not exists (upsert)
# SET      → Update properties
# DELETE   → Remove nodes/relationships
# RETURN   → Output results
# WITH     → Pipeline intermediate results (like CTEs)

Section 08

Brief History of NoSQL

1998

The Term "NoSQL" is Born

Carlo Strozzi coins "NoSQL" for his lightweight relational database that didn't use SQL. The meaning later shifted to "Not Only SQL".

2004

Google Bigtable Paper

Google publishes the Bigtable paper — a distributed storage system for structured data. Blueprint for column-family stores like HBase and Cassandra.

2007

Amazon Dynamo Paper

Amazon's Dynamo paper introduces consistent hashing and eventual consistency. Influences Cassandra, Riak, and the AP system design philosophy.

2009

MongoDB & Cassandra Launch

The modern NoSQL era begins. MongoDB launches as a document store; Cassandra open-sourced by Facebook. Redis also debuts this year.

2012

NewSQL Emerges

Google Spanner paper — globally distributed, ACID-compliant, scale-out SQL. Bridges the gap between NoSQL scale and SQL consistency.

2024+

Multi-Model Databases

ArangoDB, Couchbase, and AWS DynamoDB evolve to support multiple NoSQL paradigms in one system. The future is polyglot persistence.
