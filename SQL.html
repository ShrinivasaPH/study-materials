01 / Foundations
Introduction to SQL

SQL (Structured Query Language) is the standard language used to interact with relational databases. Created in the 1970s at IBM by Donald D. Chamberlin and Raymond F. Boyce, SQL was designed to manipulate and retrieve data stored in relational database management systems (RDBMS). It became an ANSI standard in 1986 and ISO standard in 1987.

💡
What SQL Does: SQL lets you Create, Read, Update, and Delete data (CRUD) in a relational database using English-like declarative statements. You tell the database what you want — not how to get it.

SQL Command Categories

🏗️
DDL

Data Definition Language. Defines schema structure — CREATE, ALTER, DROP, TRUNCATE.

✏️
DML

Data Manipulation Language. Modifies data — INSERT, UPDATE, DELETE, MERGE.

🔍
DQL

Data Query Language. Retrieves data — SELECT is the only command here.

🔒
DCL

Data Control Language. Controls access — GRANT, REVOKE permissions.

💾
TCL

Transaction Control Language. Manages transactions — COMMIT, ROLLBACK, SAVEPOINT.

Why Learn SQL?

  • Over 50% of data engineering and analyst roles list SQL as a required skill
  • Works across MySQL, PostgreSQL, SQLite, SQL Server, BigQuery, Snowflake, and more
  • Foundation for Business Intelligence, ETL pipelines, and data science workflows
  • Extremely stable — SQL written in the 1990s still runs today

Databases & RDBMS

A relational database organises data into tables (relations) composed of rows (records) and columns (attributes). Tables relate to each other through keys — this model was proposed by Edgar F. Codd in 1970 and remains dominant today.

Key Concepts

TableA collection of related data in rows and columnsemployees, orders, products
Row (Record)A single instance of data in a tableOne employee's details
Column (Field)An attribute of the entityname, salary, department
Primary KeyUnique identifier for each row — NOT NULL, UNIQUEemployee_id INT PRIMARY KEY
Foreign KeyA column referencing the primary key of another tabledepartment_id → departments.id
IndexData structure that speeds up queries on a columnINDEX ON employees(email)
SchemaThe blueprint/structure of a databaseAll tables and their relationships

CREATE TABLE products (
    product_id   INT              PRIMARY KEY,    -- Whole number ID
    sku          CHAR(10)         NOT NULL,       -- Fixed-length code
    name         VARCHAR(200)     NOT NULL,       -- Variable text

04 / Core SQL
DDL — Schema Design

Data Definition Language (DDL) defines the structure of your database. These statements are auto-committed in most databases — they can't be rolled back in a transaction.

CREATE TABLE

The foundation of any database. Define tables, columns, types, and constraints.

CREATE TABLE with Constraints
-- Creating a complete employees table with all constraint types
CREATE TABLE departments (
    dept_id    INT           PRIMARY KEY AUTO_INCREMENT,
    dept_name  VARCHAR(100)  NOT NULL UNIQUE,
    location   VARCHAR(50)   DEFAULT 'Remote'
);

CREATE TABLE employees (
    emp_id      INT             PRIMARY KEY AUTO_INCREMENT,
    first_name  VARCHAR(50)    NOT NULL,
    last_name   VARCHAR(50)    NOT NULL,
    email       VARCHAR(150)   NOT NULL UNIQUE,
    salary      DECIMAL(10,2)  CHECK (salary > 0),
    dept_id     INT             NOT NULL,
    hire_date   DATE            DEFAULT (CURDATE()),

    FOREIGN KEY (dept_id) REFERENCES departments(dept_id)
        ON DELETE RESTRICT
        ON UPDATE CASCADE
);

ALTER TABLE

Modify an existing table structure — add/drop/rename columns, add constraints, change data types.

ALTER Examples
-- Add a new column
ALTER TABLE employees
    ADD COLUMN phone VARCHAR(20);

-- Change data type of a column
ALTER TABLE employees
    MODIFY COLUMN phone VARCHAR(30) NOT NULL;

-- Rename a column (MySQL 8+, PostgreSQL)
ALTER TABLE employees
    RENAME COLUMN phone TO mobile_number;

-- Drop a column
ALTER TABLE employees
    DROP COLUMN mobile_number;

-- Add an index
ALTER TABLE employees
    ADD INDEX idx_email (email);

-- DROP removes the table entirely (with all data)
DROP TABLE IF EXISTS temp_employees;

-- TRUNCATE removes all rows but keeps structure (faster than DELETE *)
TRUNCATE TABLE employees;
⚠️
DROP vs TRUNCATE vs DELETE: DROP removes the table entirely. TRUNCATE removes all rows (can't be rolled back, resets AUTO_INCREMENT). DELETE removes specific rows with a WHERE clause and can be rolled back within a transaction.

05 / Core SQL
DML — Manipulate Data

Data Manipulation Language (DML) is how you write, modify, and remove actual data rows. Unlike DDL, DML statements can be wrapped in transactions and rolled back.

INSERT
UPDATE
DELETE
UPSERT
INSERT Variations
-- 1. Single row insert (explicit columns recommended)
INSERT INTO employees (first_name, last_name, email, salary, dept_id)
VALUES ('Priya', 'Sharma', 'priya@company.com', 85000.00, 2);

-- 2. Multiple rows in one statement (efficient!)
INSERT INTO employees (first_name, last_name, email, salary, dept_id)
VALUES
    ('Arjun',  'Mehta', 'arjun@company.com',  92000, 1),
    ('Sneha',  'Patel', 'sneha@company.com',  78000, 3),
    ('Rohit',  'Kumar', 'rohit@company.com',  67000, 2);

-- 3. INSERT … SELECT — copy data from another table
INSERT INTO employees_archive (first_name, last_name, email, salary, dept_id)
SELECT first_name, last_name, email, salary, dept_id
FROM   employees
WHERE  hire_date < '2020-01-01';
UPDATE Patterns
-- 1. Update a single column for specific rows
UPDATE employees
SET    salary = 90000
WHERE  emp_id = 7;

-- 2. Update multiple columns at once
UPDATE employees
SET    salary   = salary * 1.10,    -- 10% raise
       dept_id  = 3
WHERE  dept_id = 2
  AND  hire_date < '2022-01-01';

-- 3. UPDATE with a subquery (conditional update)
UPDATE employees
SET    salary = salary * 1.15
WHERE  dept_id IN (
    SELECT dept_id FROM departments
    WHERE  location = 'Bengaluru'
);

-- ⚠️ Always test with SELECT first, then replace with UPDATE!
DELETE Safely
-- 1. Delete specific rows
DELETE FROM employees
WHERE emp_id = 42;

-- 2. Delete with multiple conditions
DELETE FROM employees
WHERE is_active = FALSE
  AND hire_date < '2019-01-01';

-- 3. DELETE using a subquery
DELETE FROM orders
WHERE customer_id NOT IN (
    SELECT customer_id FROM customers
);

-- Golden Rule: Always run SELECT with the same WHERE first!
SELECT * FROM employees WHERE emp_id = 42;  -- Verify first
UPSERT (INSERT … ON DUPLICATE KEY)
-- MySQL: INSERT … ON DUPLICATE KEY UPDATE
INSERT INTO employees (emp_id, first_name, salary)
VALUES (7, 'Priya', 95000)
ON DUPLICATE KEY UPDATE
    salary = VALUES(salary);

-- PostgreSQL: INSERT … ON CONFLICT DO UPDATE
INSERT INTO employees (emp_id, first_name, salary)
VALUES (7, 'Priya', 95000)
ON CONFLICT (emp_id)
DO UPDATE SET
    salary = EXCLUDED.salary;

-- Useful for: syncing data feeds, idempotent pipelines, cache tables

SELECT — Full Anatomy
-- Full SELECT statement with all major clauses
SELECT
    e.emp_id,
    CONCAT(e.first_name, ' ', e.last_name)  AS full_name,
    d.dept_name,
    e.salary,
    ROUND(e.salary * 0.1, 2)               AS bonus

FROM   employees e
JOIN   departments d ON e.dept_id = d.dept_id

WHERE  e.salary BETWEEN 50000 AND 150000
  AND  d.location = 'Bengaluru'
  AND  e.first_name LIKE 'A%'

ORDER BY e.salary DESC, full_name ASC
LIMIT  10 OFFSET 0;

-- Key operators in WHERE clauses:
-- =, !=, <, >, <=, >=          → Comparison
-- AND, OR, NOT                  → Logical
-- BETWEEN x AND y               → Range (inclusive)
-- IN (v1, v2, v3)               → Set membership
-- LIKE 'A%'  (% = any, _ = one) → Pattern match
-- IS NULL / IS NOT NULL         → Null checks

07 / Core SQL
Aggregates & Grouping

Aggregate functions compute a single result from a set of rows. Combined with GROUP BY, they power dashboards, reports, and analytics.

FunctionWhat It DoesNULL Behaviour
COUNT(*)Counts all rows including NULLsIncludes NULLs
COUNT(col)Counts non-NULL values in a columnSkips NULLs
SUM(col)Sum of all numeric valuesSkips NULLs
AVG(col)Average of non-NULL valuesSkips NULLs
MIN(col)Smallest valueSkips NULLs
MAX(col)Largest valueSkips NULLs
GROUP_CONCATConcatenates values in a group (MySQL)Skips NULLs
GROUP BY & HAVING
-- Department analytics: salary stats per department
SELECT
    d.dept_name,
    COUNT(e.emp_id)          AS headcount,
    ROUND(AVG(e.salary), 2) AS avg_salary,
    MIN(e.salary)            AS min_salary,
    MAX(e.salary)            AS max_salary,
    SUM(e.salary)            AS total_payroll

FROM       employees e
INNER JOIN departments d ON e.dept_id = d.dept_id

GROUP BY   d.dept_id, d.dept_name

-- HAVING filters AFTER aggregation (unlike WHERE which filters rows)
HAVING     COUNT(e.emp_id) >= 5           -- only departments with 5+ employees
  AND      AVG(e.salary) > 60000

ORDER BY   avg_salary DESC;
💡
WHERE vs HAVING: WHERE filters individual rows before aggregation. HAVING filters groups after aggregation. You cannot use aggregate functions in a WHERE clause.

JOINs

JOINs combine rows from two or more tables based on a related column. Understanding when to use each type is essential for correct query results.

INNER JOIN
Only matching rows from both tables
LEFT JOIN
All left rows + matching right
RIGHT JOIN
All right rows + matching left
FULL OUTER JOIN
All rows from both tables
CROSS JOIN
Every row × every row (Cartesian)

SELF JOIN

SELF JOIN — Employee & Manager
-- A table referencing itself (employees have a manager who is also an employee)
SELECT
    e.first_name                      AS employee,
    CONCAT(m.first_name, ' ', m.last_name) AS manager
FROM       employees e
LEFT JOIN  employees m ON e.manager_id = m.emp_id
ORDER BY   manager, employee;

09 / Intermediate
Subqueries & CTEs

Subqueries are queries nested inside another query. CTEs (Common Table Expressions) use the WITH clause to make complex queries readable and reusable.

📌 Scalar Subquery — returns a single value ▼
Scalar Subquery
-- Find employees earning more than the company average
SELECT first_name, last_name, salary
FROM   employees
WHERE  salary > (
    SELECT AVG(salary)
    FROM   employees          -- Scalar: returns ONE value
)
ORDER BY salary DESC;
📌 Correlated Subquery — references outer query ▼
Correlated Subquery
-- For each employee, find their salary vs their dept average
SELECT
    e.first_name,
    e.salary,
    (
        SELECT ROUND(AVG(e2.salary), 2)
        FROM   employees e2
        WHERE  e2.dept_id = e.dept_id   -- references outer e
    ) AS dept_avg_salary
FROM employees e;
📌 CTE (Common Table Expression) — WITH clause ▼
CTE — Readable Multi-step Logic
-- CTEs break complex queries into named, readable steps
WITH
dept_stats AS (
    SELECT
        dept_id,
        AVG(salary) AS avg_sal,
        MAX(salary) AS max_sal
    FROM employees
    GROUP BY dept_id
),
top_earners AS (
    SELECT e.emp_id, e.first_name, e.salary, e.dept_id
    FROM  employees e
    JOIN  dept_stats ds ON e.dept_id = ds.dept_id
    WHERE e.salary >= ds.avg_sal * 1.2   -- 20% above dept avg
)
SELECT t.first_name, t.salary, d.dept_name
FROM  top_earners t
JOIN  departments d ON t.dept_id = d.dept_id
ORDER BY t.salary DESC;
📌 Recursive CTE — hierarchical data ▼
Recursive CTE — Org Chart
-- Walk an org hierarchy from CEO downward
WITH RECURSIVE org_tree AS (
    -- Anchor: start with CEO (no manager)
    SELECT emp_id, first_name, manager_id, 0 AS level
    FROM   employees
    WHERE  manager_id IS NULL

    UNION ALL

    -- Recursive step: find each manager's direct reports
    SELECT e.emp_id, e.first_name, e.manager_id, ot.level + 1
    FROM   employees e
    JOIN   org_tree ot ON e.manager_id = ot.emp_id
)
SELECT REPEAT('  ', level) || first_name AS hierarchy
FROM   org_tree
ORDER BY level, first_name;

10 / Intermediate
Views & Indexes

Views

A view is a stored SQL query that acts like a virtual table. It simplifies complex queries, provides security by hiding columns, and ensures consistent logic across the organisation.

CREATE VIEW
-- Create a view for HR analytics (hides salary from general users)
CREATE VIEW v_employee_summary AS
SELECT
    e.emp_id,
    CONCAT(e.first_name, ' ', e.last_name) AS full_name,
    d.dept_name,
    e.hire_date,
    DATEDIFF(NOW(), e.hire_date) AS days_employed
FROM  employees e
JOIN  departments d ON e.dept_id = d.dept_id;

-- Use it just like a table
SELECT * FROM v_employee_summary WHERE days_employed > 1000;

-- Drop a view
DROP VIEW IF EXISTS v_employee_summary;

Indexes

An index is a separate data structure (usually a B-tree) that enables fast lookups on a column. Dramatically speeds up SELECT queries but slightly slows INSERT/UPDATE/DELETE.

Index Strategies
-- Single-column index (most common)
CREATE INDEX idx_emp_email    ON employees(email);
CREATE INDEX idx_emp_dept     ON employees(dept_id);
CREATE INDEX idx_emp_hire     ON employees(hire_date);

-- Composite index — column order matters!
-- Optimal for: WHERE dept_id=1 AND hire_date > '2022-01-01'
CREATE INDEX idx_dept_hire    ON employees(dept_id, hire_date);

-- Unique index (also enforces uniqueness constraint)
CREATE UNIQUE INDEX uidx_email ON employees(email);

-- EXPLAIN — analyse your query's use of indexes
EXPLAIN
SELECT * FROM employees WHERE email = 'priya@company.com';

-- Drop index
DROP INDEX idx_emp_email ON employees;

11 / Advanced
Window Functions

Window functions perform calculations across a set of rows related to the current row — without collapsing them into a group. They are the most powerful analytical tool in SQL.

🪟
Syntax: function_name() OVER (PARTITION BY col ORDER BY col ROWS/RANGE frame)
CategoryFunctionsUse Case
RankingROW_NUMBER(), RANK(), DENSE_RANK(), NTILE(n)Leaderboards, pagination, quartiles
OffsetLAG(col, n), LEAD(col, n), FIRST_VALUE(), LAST_VALUE()Period-over-period, prev/next row access
AggregateSUM(), AVG(), COUNT(), MIN(), MAX() OVER()Running totals, moving averages
Window Functions — Analytics Patterns
-- 1. ROW_NUMBER: rank employees by salary within department
SELECT
    first_name,
    dept_id,
    salary,
    ROW_NUMBER() OVER (
        PARTITION BY dept_id
        ORDER BY salary DESC
    ) AS rank_in_dept
FROM employees;

-- 2. RANK vs DENSE_RANK (handling ties)
-- RANK: gaps after ties (1,2,2,4)  DENSE_RANK: no gaps (1,2,2,3)
SELECT
    first_name, salary,
    RANK()       OVER (ORDER BY salary DESC) AS rank_val,
    DENSE_RANK() OVER (ORDER BY salary DESC) AS dense_rank_val
FROM employees;

-- 3. Running total of salary (cumulative sum)
SELECT
    hire_date,
    first_name,
    salary,
    SUM(salary) OVER (
        ORDER BY hire_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_total
FROM employees;

-- 4. LAG — month-over-month revenue comparison
SELECT
    month,
    revenue,
    LAG(revenue, 1) OVER (ORDER BY month) AS prev_revenue,
    revenue - LAG(revenue, 1) OVER (ORDER BY month) AS mom_change
FROM monthly_revenue;

-- 5. NTILE — divide into 4 salary quartiles
SELECT
    first_name, salary,
    NTILE(4) OVER (ORDER BY salary) AS salary_quartile
FROM employees;

12 / Advanced
Transactions & ACID

A transaction is a group of SQL statements that execute as a single unit of work — either all succeed (COMMIT) or all fail (ROLLBACK). This guarantees data integrity.

⚛️
Atomicity

All operations in a transaction succeed or none do. No partial updates.

✅
Consistency

A transaction brings the database from one valid state to another. Constraints always hold.

🏝️
Isolation

Concurrent transactions don't interfere. Controlled via isolation levels.

💾
Durability

Once committed, data survives crashes — written to persistent storage.

Bank Transfer — Classic Transaction Example
-- Transfer ₹50,000 from Account A to Account B atomically
START TRANSACTION;

UPDATE accounts
SET    balance = balance - 50000
WHERE  account_id = 'ACC-001';   -- Debit sender

UPDATE accounts
SET    balance = balance + 50000
WHERE  account_id = 'ACC-002';   -- Credit receiver

-- Verify balances are valid (application logic checks this)
-- If all good:
COMMIT;

-- If any error occurred:
-- ROLLBACK;  ← undoes all changes since START TRANSACTION

-- SAVEPOINT — partial rollback within a transaction
START TRANSACTION;
    INSERT INTO orders VALUES (...);
    SAVEPOINT sp1;
    INSERT INTO order_items VALUES (...);
    ROLLBACK TO sp1;   -- Undoes only order_items
COMMIT;

13 / Advanced
Query Optimization

Writing SQL that works is step one. Writing SQL that performs at scale is a professional skill. Here are the key principles and patterns.

⚡ Use EXPLAIN / EXPLAIN ANALYZE▼

Always run EXPLAIN before optimising. It shows the query plan — whether indexes are used, estimated rows, and join types.

EXPLAIN ANALYZE
SELECT * FROM employees
WHERE email = 'priya@company.com';
-- Look for: "Using index" vs "Full table scan"
-- Seq Scan = slow (no index), Index Scan = fast
⚡ SELECT only needed columns▼
-- ❌ Bad: pulls all columns, kills performance on wide tables
SELECT * FROM employees;

-- ✅ Good: only what you need
SELECT emp_id, first_name, salary FROM employees;
⚡ Avoid functions on indexed columns in WHERE▼
-- ❌ Bad: function on column defeats index
SELECT * FROM employees
WHERE  YEAR(hire_date) = 2022;

-- ✅ Good: use a range — index is used
SELECT * FROM employees
WHERE  hire_date BETWEEN '2022-01-01' AND '2022-12-31';
⚡ EXISTS vs IN for subqueries▼
-- EXISTS stops after finding first match (faster for large sets)
SELECT c.customer_name
FROM   customers c
WHERE  EXISTS (
    SELECT 1
    FROM   orders o
    WHERE  o.customer_id = c.customer_id
      AND  o.amount > 10000
);

E-Commerce — Top Selling Products this Month
WITH monthly_sales AS (
    SELECT
        p.product_id,
        p.name                      AS product_name,
        SUM(oi.quantity)            AS units_sold,
        SUM(oi.quantity * oi.unit_price) AS revenue
    FROM       order_items oi
    JOIN       orders o      ON oi.order_id   = o.order_id
    JOIN       products p    ON oi.product_id = p.product_id
    WHERE      o.order_date >= DATE_FORMAT(NOW(), '%Y-%m-01')
    GROUP BY   p.product_id, p.name
)
SELECT
    RANK() OVER (ORDER BY revenue DESC) AS rank_pos,
    product_name,
    units_sold,
    CONCAT('₹', FORMAT(revenue, 2))         AS revenue
FROM  monthly_sales
ORDER BY revenue DESC
LIMIT 10;
Analytics — Customer Cohort Retention
-- Find users active in Month 0 who returned in Month 1, 2, 3...
WITH cohorts AS (
    SELECT
        user_id,
        DATE_FORMAT(MIN(event_date), '%Y-%m') AS cohort_month
    FROM     user_events
    GROUP BY user_id
),
activity AS (
    SELECT
        ue.user_id,
        c.cohort_month,
        TIMESTAMPDIFF(MONTH, c.cohort_month, ue.event_date) AS month_num
    FROM user_events ue
    JOIN cohorts c ON ue.user_id = c.user_id
)
SELECT
    cohort_month,
    month_num,
    COUNT(DISTINCT user_id) AS active_users
FROM     activity
GROUP BY cohort_month, month_num
ORDER BY cohort_month, month_num;
HR — Salary Bands & Pay Gap Analysis
SELECT
    d.dept_name,
    e.gender,
    COUNT(*)                    AS headcount,
    ROUND(AVG(e.salary), 2)     AS avg_salary,
    MIN(e.salary)                AS min_salary,
    MAX(e.salary)                AS max_salary,
    -- Percentile salary using window function
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY salary)
        OVER (PARTITION BY dept_id, gender) AS median_salary
FROM  employees e
JOIN  departments d ON e.dept_id = d.dept_id
GROUP BY d.dept_name, e.gender, dept_id
ORDER BY d.dept_name, e.gender;
