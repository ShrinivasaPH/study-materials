🧬 The Complete LangChain Ecosystem — All Four Layers

Build. Orchestrate.
Observe. Deploy.
The full AI stack.

Somewhere between a raw language model and a finished AI product, there are four layers of engineering to master. First, you need to connect the LLM to your data, tools, and memory — that's LangChain. Then you need to orchestrate workflows where the AI loops, branches, waits for approvals, and collaborates with other agents — that's LangGraph. Then you need to observe every call, evaluate output quality, and version your prompts like production code — that's LangSmith. Finally, you need to deploy everything as a scalable, documented REST API with streaming support — that's LangServe.

This guide is a deep, story-first walkthrough of all four — not a quick-reference API dump. You'll understand why each tool exists, what problem it solves that the others can't, and how they compose into a single cohesive production stack. Every section comes with working, Google Colab-ready code. By the end, you'll be able to build, debug, evaluate, and ship production-grade AI applications with confidence.

01 Foundation

Twelve building blocks across
the entire ecosystem

The four frameworks in this guide — LangChain, LangGraph, LangSmith, and LangServe — each introduce their own set of core concepts. But they're designed to work together as layers: LangChain provides the atomic pieces; LangGraph assembles those pieces into workflows that can loop, branch, and pause; LangSmith wraps around everything to make it visible and measurable; and LangServe sits at the outermost layer, exposing finished workflows as documented REST APIs the outside world can call. Before you write any code, it pays to understand what each piece is for and which layer it belongs to.

The crucial insight is this: each tool solves exactly one problem, and none of them try to solve the others'. LangChain doesn't do orchestration. LangGraph doesn't do observability. LangSmith doesn't run your chains. LangServe doesn't build your logic. This clean separation of concerns is what makes the ecosystem so powerful — you can use as much or as little of each tool as your application needs.

🤖
LLM / Chat Model
The brain of the operation. A Chat Model is a unified wrapper around any large language model — OpenAI's GPT-4o, Google's Gemini, Anthropic's Claude, or a local Ollama model. They all speak the same LangChain language, so you can swap one for another without rewriting your app.
ChatOpenAI · ChatGoogleGenerativeAI
📝
Prompt Template
Prompts are the instructions you give the LLM, and templates let you write them once with variable placeholders, then reuse them with different values. Think of a Prompt Template like a well-designed form — the structure is fixed, the content changes each time.
ChatPromptTemplate
⛓️
Chain (LCEL)
A Chain is what you get when you connect primitives together using the | pipe operator. Output from the left becomes input for the right. This is the LangChain Expression Language (LCEL) — elegant, readable, and powerful. Chains are the main unit of logic in any LangChain app.
RunnableSequence
🧠
Memory
LLMs are stateless — send them two messages and they won't remember the first. Memory modules solve this by recording conversation history and injecting it automatically into every new prompt. Without memory, every message is a first meeting. With memory, it's a conversation.
ConversationBufferMemory
🗂️
Vector Store
A vector store holds your documents as mathematical embeddings — high-dimensional number arrays that capture semantic meaning. When you search, it finds documents whose meaning is close to your query, not just documents that share the same words. This is what powers RAG applications.
FAISS · ChromaDB · Pinecone
🔧
Tools & Agents
An Agent is an LLM that can decide what to do next. Give it a set of Tools — functions like "search the web", "run SQL", "calculate math" — and the agent will reason about which tool to use, call it, observe the result, and keep going until it has a final answer. Agents bring autonomy to your apps.
AgentExecutor · @tool
📤
Output Parser
LLMs return raw text. Output Parsers transform that text into exactly the format your code needs — a plain string, a Python dict, a JSON object, or a fully validated Pydantic model. Parsers are the bridge between the natural language world and the structured world of your application.
StrOutputParser · JsonOutputParser
🕸️
StateGraph LangGraph
LangGraph's central concept. A StateGraph defines your AI workflow as a real directed graph with a shared typed state dict. Every node reads from and writes to this state. Unlike LCEL chains, graphs can have cycles — a node can loop back to an earlier node based on what happened. This is how you build agents that retry, verify, and self-correct.
StateGraph · START · END
🔭
Run / Trace LangSmith
Every LLM call, tool invocation, and chain execution is automatically logged as a "Run" in LangSmith the moment you set four environment variables. Runs nest into a tree — the full trace of one end-to-end request. You can see exactly what went in, what came out, how long it took, and how many tokens it used — for every step, at every depth.
Trace · Dataset · Evaluator
🚀
add_routes / RemoteRunnable LangServe
LangServe is the deployment layer of the stack. add_routes(app, chain, path="/chat") takes any LCEL chain or LangGraph workflow and instantly exposes it as a documented REST API with five endpoints: invoke, batch, stream, stream_log, and an interactive playground UI. On the client side, RemoteRunnable wraps a deployed endpoint so it behaves exactly like a local chain — same .invoke(), .stream(), .batch() interface.
add_routes · RemoteRunnable · FastAPI

03 LangChain

LLMs & Chat Models

Every LangChain application begins with a model. But here's the brilliant part — you don't have to think about which model you're using when writing your chain logic. LangChain provides a unified interface that works identically whether you're calling GPT-4o, Gemini Flash, or a locally-running Llama model via Ollama. You can swap the model in one line without touching anything else.

When you create a Chat Model, you pass in a few key parameters. The model name determines which version of the AI you get (more capable usually means more expensive). The temperature controls creativity — set it to 0 when you need reliable, repeatable answers (like answering a factual question), or bump it up to 0.9 when you want creative, varied outputs (like writing taglines or brainstorming ideas).

You communicate with a Chat Model through messages. There are three kinds: SystemMessage is like giving the AI its job description at the start — "You are a helpful assistant who only answers in bullet points." HumanMessage is what the user says. AIMessage is what the AI says back, and you can include past AI messages when building conversation history manually.

text = response.content # ← the actual string you want
model="gpt-4o-mini"Cheaper, fast GPT-4 level model. Best for learning and prototyping.
model="gemini-2.0-flash"Google's fast model. Has a generous free tier — ideal for getting started without a credit card.

04 LangChain

Prompt Templates —
the art of talking to AI

The quality of your output is almost entirely determined by the quality of your prompt. This is so important that there's an entire field called "prompt engineering" dedicated to it. LangChain's Prompt Template system makes prompt engineering practical and maintainable by separating your prompt structure from your runtime data.

Think about how a well-run restaurant works. The chef doesn't invent a new menu from scratch every day — they have recipes with clearly defined ingredients and steps. When a customer orders, the kitchen fills in the specific details (which cut of steak, which sauce) but follows the same recipe. A Prompt Template is your recipe. The recipe defines the structure; the {variables} are the ingredients that change per request.

There are two main types. PromptTemplate is for single-message prompts — simpler tasks where you just need to fill in some slots. ChatPromptTemplate is for multi-turn, multi-role conversations — it lets you specify a system message, optional few-shot examples, and the human input, all as a structured list of (role, content) pairs. In almost all real applications, you'll use ChatPromptTemplate.

A key reason to use templates even when you could use f-strings: templates integrate directly into LCEL chains using the | pipe operator. They automatically handle the conversion of your input dict into properly formatted messages that the LLM expects.

❌ Without Template — Don't do this
# Fragile. Hard to reuse. No role separation.
topic = "recursion"
lang = "Python"

# Directly building strings is messy
prompt = f"Explain {topic} in {lang}"
llm.invoke(prompt)

# Problems:
# - No system role, no personality
# - Can't pipe into a chain
# - Hard to test variations
# - Easy to forget a variable
✅ With Template — Do this
# Reusable. Testable. Composable.
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a {role} expert."),
    ("human",  "Explain {topic} in {lang}.")
])

# Chains directly with | pipe
chain = prompt | llm | parser

# Benefits:
# ✓ Clear role separation
# ✓ Plugs into any LCEL chain
# ✓ Easy to test and swap
# ✓ Variables validated at runtime

from langchain_openai import ChatOpenAI

simple = PromptTemplate.from_template(
    "Write a catchy tagline for a {product} company named {name}."
)
chain1 = simple | llm | parser

Your explanations are clear, concise, and use simple analogies.

05 LangChain

Chains & LCEL — composing
your AI pipeline

Here's where LangChain starts to feel genuinely elegant. The LangChain Expression Language (LCEL) introduces one deceptively simple idea: any two compatible components can be connected with the | pipe operator. If you've ever used Unix pipes in the terminal (cat file.txt | grep "error" | wc -l), you already understand the concept intuitively. The output of one step flows as the input to the next.

The power of this approach becomes clear when you realise that every step in the chain follows the same contract — it's a Runnable, which means it has .invoke(), .stream(), and .batch(). So a chain of three components is itself a Runnable, which means you can pipe chains into other chains. You can compose arbitrarily complex logic from simple pieces.

Beyond simple linear pipelines, LCEL supports branching with RunnableParallel — run two chains on the same input simultaneously and merge their outputs. You can inject custom Python functions anywhere with RunnableLambda. And you can pass the original input through unchanged at any point with RunnablePassthrough. These three utilities cover the vast majority of real-world logic you'll need to express.

⛓️ LCEL Composition Rules
chain = A | B | C → equivalent to → C(B(A(input)))
parallel = RunnableParallel(branch1=A, branch2=B)
with_fn = RunnableLambda(my_python_function)
A = PromptTemplateInput: dict of variables → Output: ChatPromptValue (list of messages)
B = LLMInput: ChatPromptValue → Output: AIMessage object
C = OutputParserInput: AIMessage → Output: str / dict / Pydantic model
chain.invoke({})Runs all steps in sequence. Returns the final output type (str if using StrOutputParser).
chain.stream({})Streams tokens. The StrOutputParser handles partial tokens — you get readable chunks in real-time.
03a_basic_chain.py
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from langchain_core.output_parsers import StrOutputParser

# ── Define each piece separately, then compose with | ──
prompt = ChatPromptTemplate.from_template(
    "Summarise the following in exactly 2 sentences:\n\n{text}"
)
llm    = ChatOpenAI(model="gpt-4o-mini", temperature=0)
parser = StrOutputParser()

# The chain — each | hands off output to the next stage
summarise_chain = prompt | llm | parser

# ── invoke: run once, get a string back ───────────────
result = summarise_chain.invoke({
    "text": """
    LangChain is an open-source framework that makes it easy to build
    applications powered by large language models. It provides
    standardised interfaces for models, prompts, memory, retrievers,
    and agents — so developers can focus on logic, not boilerplate.
    """
})
print("Summary:", result)

# ── stream: print tokens as they arrive ───────────────
print("\nStreaming: ", end="")
for chunk in summarise_chain.stream({"text": "Explain neural networks"}):
    print(chunk, end="", flush=True)
print()

# ── batch: process many inputs in parallel ────────────
texts = [
    {"text": "Explain RAG in LangChain"},
    {"text": "Explain Agents in LangChain"},
    {"text": "Explain Embeddings in LangChain"}
]
summaries = summarise_chain.batch(texts)
for i, s in enumerate(summaries, 1):
    print(f"\n{i}. {s}")
03b_parallel.py
from langchain_core.runnables import RunnableParallel, RunnablePassthrough
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from langchain_core.output_parsers import StrOutputParser

llm    = ChatOpenAI(model="gpt-4o-mini")
parser = StrOutputParser()

# RunnableParallel sends the SAME input to multiple chains at once.
# Both chains run concurrently — you get both outputs in one dict.

pros_chain = (
    ChatPromptTemplate.from_template("List exactly 3 advantages of {tech}.")
    | llm | parser
)
cons_chain = (
    ChatPromptTemplate.from_template("List exactly 3 disadvantages of {tech}.")
    | llm | parser
)
summary_chain = (
    ChatPromptTemplate.from_template("Write a 1-sentence overview of {tech}.")
    | llm | parser
)

analysis = RunnableParallel(
    pros    = pros_chain,
    cons    = cons_chain,
    summary = summary_chain,
    topic   = RunnablePassthrough()  # passes input dict unchanged
)

result = analysis.invoke({"tech": "Docker containers"})

print("=== OVERVIEW ===")
print(result["summary"])
print("\n=== ADVANTAGES ===")
print(result["pros"])
print("\n=== DISADVANTAGES ===")
print(result["cons"])
03c_lambda.py
from langchain_core.runnables import RunnableLambda
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from langchain_core.output_parsers import StrOutputParser

# RunnableLambda wraps ANY Python function so it can live inside a | pipe chain.
# This is how you inject pre-processing or post-processing logic.

def preprocess_input(data: dict) -> dict:
    """Normalize and enrich the input before it reaches the prompt."""
    return {
        "topic": data["topic"].strip().title(),
        "level": data.get("level", "beginner")
    }

def add_metadata(response: str) -> dict:
    """Enrich the final output with computed metadata."""
    words = len(response.split())
    sentences = response.count(".") + response.count("!") + response.count("?")
    return {
        "explanation": response,
        "word_count": words,
        "sentences": sentences,
        "reading_time_sec": round(words / 3)  # ~180 wpm average
    }

# Chain: preprocess → prompt → llm → parse → enrich
chain = (
    RunnableLambda(preprocess_input)
    | ChatPromptTemplate.from_template(
        "Explain {topic} to a {level}. Be thorough."
      )
    | ChatOpenAI(model="gpt-4o-mini")
    | StrOutputParser()
    | RunnableLambda(add_metadata)

06 LangChain

Memory — giving your AI
a sense of continuity

Here's a fundamental and often surprising fact about language models: they have no memory whatsoever between API calls. Every single call is completely independent. If you tell GPT-4 your name on Monday and ask "what's my name?" on Tuesday — even a millisecond later — it will have absolutely no idea. The model doesn't remember the previous call at all. It only knows what's in its context window right now.

This creates an obvious problem for chatbots and conversational applications. The solution that LangChain's memory system provides is elegant: instead of the model magically "remembering" things, it gives the model its own conversation history as part of every new prompt. Each time you send a new message, the system automatically retrieves the past messages and injects them into the prompt's context. The LLM isn't remembering — it's being reminded. The effect from the user's perspective is seamless.

LangChain manages this through session IDs. Each unique user or conversation gets its own session ID, and LangChain uses that ID to look up the correct history from a store. In development, you use InMemoryChatMessageHistory — history lives in RAM and disappears when your program exits. In production, you'd swap this for a Redis or PostgreSQL-backed store that persists across server restarts.

The component that ties it all together is RunnableWithMessageHistory. It wraps any chain and automatically handles loading history before the chain runs and saving the new messages after it finishes. From your application's perspective, you just call .invoke() with a session ID in the config — everything else is handled for you.

07 LangChain

Vector Stores & RAG —
teaching AI about your data

Imagine you've just hired a brilliant new employee — a world-class expert in everything. There's just one catch: they've never seen a single document from your company. They don't know your products, your pricing, your policies, or your customers. You could spend months retraining them on your company's data, but that would cost a fortune and take forever. Or — and this is RAG — you could just hand them the relevant documents before every question they need to answer.

Retrieval-Augmented Generation (RAG) is exactly this pattern. When a user asks a question, you first search your document library for the most relevant pieces, then hand those pieces to the LLM along with the question. The LLM doesn't need to "know" your data at training time — it can read it on demand, just like your employee reading a page from the company handbook before answering a customer.

The magic ingredient is embeddings. An embedding is a numerical representation of text — typically a list of 768 or 1536 floating-point numbers — that captures semantic meaning. The remarkable property: texts with similar meanings have embeddings that are close together in this high-dimensional space. "The dog barked loudly" and "the canine made a loud noise" will have very similar embeddings, even though they share almost no words. This is what makes RAG's search so powerful — it finds meaning, not keywords.

The process has two phases. The indexing phase (done once): load documents → split into chunks → embed each chunk → store in a vector database. The querying phase (done every question): embed the question → find the k nearest chunks → inject them as context → generate an answer. This is the full RAG pipeline, and it powers everything from customer support bots to enterprise document search.

08 LangChain

Agents & Tools —
the AI that can act

Chains are powerful, but they follow a fixed script. Every time you run a chain, it executes the same sequence of steps in the same order. This is great for predictable tasks, but what about tasks where you don't know in advance exactly what steps are needed? What if the answer requires searching the web, then doing a calculation, then querying a database, and the order depends on what each step returns?

This is what Agents are built for. An agent is an LLM with access to a set of Tools and the autonomy to decide which tool to use and when. At each step, the agent reasons about what it knows, decides on an action (which tool to call and with what input), observes the result, and then reasons again. It repeats this loop until it has enough information to give a final answer — or until a maximum iteration limit is hit.

The strategy most agents use is called ReAct (Reason + Act), from a 2022 research paper by Yao et al. The model is prompted to generate its reasoning as text ("I need to find the current price of X, then calculate Y% of it") before taking any action. This makes agents more transparent, easier to debug, and less likely to make mistakes — because the model has to justify its reasoning before it commits to an action.

Tools are just Python functions with a decorator. The @tool decorator tells LangChain "this function is available to the agent." Critically, the function's docstring is what the LLM reads to decide when to use it. Write clear, specific docstrings — they're the API documentation the agent consults when deciding what to do.

🔁 ReAct Agent Loop
while not_done:
thought = LLM(question + observations_so_far) ← reasoning step
action = LLM(thought) ← which tool? what input?
obs = tool[action.name](action.input) ← execute tool
not_done = LLM(obs) ← do I have enough info yet?
return Final Answer ← only when LLM decides it's done
ThoughtThe agent's internal monologue. Helps the model reason step-by-step before acting.
ActionA structured output: tool name + input string. The LLM chooses from available tools.
ObservationThe return value from the tool. Fed back to the LLM for the next reasoning step.
max_iterations=5Safety limit. If the agent hasn't finished after 5 loops, it stops and returns what it has.

# ── Define custom tools using the @tool decorator ────────────
# The docstring IS what the LLM reads to decide when to call a tool.
# Write it clearly — it is the tool's API contract with the agent.

@tool
def calculate(expression: str) -> str:
    """Evaluates a mathematical expression safely.

    Use for arithmetic, algebra, or trigonometry.
    Valid inputs: '2 ** 10', 'sqrt(144)', 'pi * 5**2', '(100 * 1.08) ** 3'
    Do NOT use for symbolic algebra or calculus problems.
    """
    try:
        safe_env = {k: getattr(math, k) for k in dir(math) if not k.startswith("_")}
        result = eval(expression, {"__builtins__": {}}, safe_env)
        return f"Result: {result}"
    except Exception as e:
        return f"Error evaluating expression: {e}"

@tool
def count_words(text: str) -> str:
    """Counts the total number of words in a piece of text.

    Use when the user wants to know how long a text is in words.
    Input: the full text string to count.
    """
    count = len(text.split())
    return f"The text has {count} words."

09 LangChain

Internals & Patterns —
how it actually works

You've now used LangChain's main building blocks. But understanding why they work the way they do — the underlying mechanics — will make you dramatically more effective when things go wrong, when performance matters, or when you need to do something the documentation doesn't quite cover. This section goes one layer deeper.

The entire LangChain system is built on a single interface called the Runnable protocol. Every object in a chain — prompts, LLMs, parsers, retrievers, custom functions — implements this protocol. That's what makes the | pipe operator work: it doesn't call anything yet, it just creates a new RunnableSequence object that records the composition. Nothing executes until you call .invoke(). This lazy evaluation is what allows LangChain to support streaming, batching, and async out of the box without any special syntax.

Streaming is one of the most practically important features for production. Instead of waiting 8 seconds for a long LLM response to complete before showing anything to the user, .stream() yields tokens as the model generates them. The | operator propagates streaming automatically — if the LLM streams, the output parser streams too, and the final consumer receives a generator. Similarly, .batch() runs multiple inputs through the same chain with automatic concurrency, dramatically reducing total latency for bulk workloads.

The most powerful advanced pattern is RunnablePassthrough.assign(). It lets you add computed keys to a dictionary mid-chain — the dictionary passes through with all its original keys intact, plus new keys you compute inline. This is the idiomatic way to build RAG chains: forward the original question through unchanged while also computing the retrieved context in parallel. Understanding this pattern unlocks a huge range of chain architectures that would otherwise require awkward workarounds.

⚙️ The Runnable Protocol — Full Interface
any_runnable.invoke(input) ← synchronous, one input, one output
any_runnable.stream(input) ← yields chunks as they arrive (generator)
any_runnable.batch([in1, in2, ...]) ← parallel, list in → list out
any_runnable.ainvoke(input) ← async variant (use with await)
any_runnable.astream(input) ← async generator for real-time UI
A | B = RunnableSequence(A, B)The pipe creates a lazy sequence — nothing runs yet. The sequence itself is a Runnable, so it supports all 5 methods above. You can pipe sequences together indefinitely.
RunnablePassthrough.assign(key=fn)Pass the input dict through unchanged AND add a new key computed by fn. The canonical way to add retrieved context to a RAG chain without losing the original question.
RunnableParallel({k1: r1, k2: r2})Run multiple Runnables on the SAME input concurrently. Returns a dict. Much faster than running sequentially when operations are independent (e.g. two different retrievers).
chain.with_retry(stop_after_attempt=3)Automatically retry on failure. Wraps the chain with exponential backoff. Essential for production chains that call external APIs which occasionally time out.
chain.with_fallbacks([backup_chain])If the primary chain raises an exception, run the fallback instead. Use to gracefully degrade: try GPT-4o first, fall back to GPT-4o-mini if the API is overloaded.

18 LangGraph

LangGraph — when chains
aren't enough

Everything you've built so far with LCEL chains follows a fundamentally linear pattern: input goes in one end, passes through a sequence of steps, and output comes out the other. This is beautiful for a huge range of tasks — but there's a class of problem that linear chains simply cannot model. What if, after generating an answer, you need to check that answer and potentially redo a step? What if two agents need to hand work back and forth, each waiting for the other to finish? What if a user needs to approve the AI's plan before it executes?

All of these scenarios require cycles — the ability to loop back to a previous step based on what happened at a later step. LCEL chains are Directed Acyclic Graphs (DAGs): they can branch, but they can never loop. LangGraph lifts this restriction entirely. It lets you define your AI workflow as a genuine graph — with nodes that represent units of work, edges that define how control flows between nodes, and crucially, the ability to have cycles, branches, and waiting states built right into the structure.

The central concept in LangGraph is State. Every node in the graph reads from and writes to a shared state dictionary, typed and validated by a TypedDict. Think of the state as a whiteboard in the middle of the room — every agent, every tool call, every human approval step reads from it and writes back to it. The graph moves from node to node, and at each step the state evolves. This makes the entire workflow inspectable, resumable, and debuggable in a way that's impossible with raw chains.

LangGraph also introduces conditional edges: instead of always flowing from node A to node B, you can define a routing function that looks at the current state and decides which node to go to next — or whether to end. This is how you implement "if the answer quality is too low, regenerate; otherwise, proceed." It's how you build a ReAct agent loop manually, with full control over every iteration. And it's how you implement human-in-the-loop patterns, where the graph pauses and waits for a human to review before resuming.

🕸️ LangGraph Mental Model
graph = StateGraph(MyState) ← typed shared state dict
graph.add_node("step_name", fn) ← node = any Python function
graph.add_edge("A", "B") ← always go from A to B
graph.add_conditional_edges("A", router_fn) ← decide at runtime
app = graph.compile() ← compile into a runnable
result = app.invoke(initial_state) ← runs like any LCEL chain
StateGraphThe graph builder. Parameterised by your state TypedDict — enforces what's in and out of every node.
Node functionTakes state dict as input, returns a partial state dict with only the keys it wants to update.
START / ENDSpecial sentinel strings. Add an edge from START to your first node, and to END when you want the graph to finish.
Conditional edgeA Python function that returns the name of the next node as a string. This is how branching and looping work.
checkpointerOptional: attach a checkpointer (e.g. MemorySaver) to make the graph resumable across interruptions — essential for HITL.
📖
The key intuition. Imagine a hiring pipeline: screen résumé → if qualified, schedule interview → if interview passes, make offer → if offer declined, loop back to interview pool → else, done. This is a graph with cycles. A linear chain can't model the "loop back" step. LangGraph can. Any workflow where the next step depends on what the previous step returned — especially if it can cycle — is a LangGraph use case.

from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, START, END
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage, SystemMessage

# ── Step 1: Define the State ──────────────────────────
# State is a TypedDict that ALL nodes share and can read/update.
# Every key you add here is available to every node in the graph.
class WritingState(TypedDict):
    topic:     str         # initial input, set once
    draft:     str         # first draft written by the writer node
    critique:  str         # feedback from the critic node
    final:     str         # polished output from the editor node

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.7)

# ── Step 2: Define Node Functions ─────────────────────
# Each node is just a Python function.
# Input:  the current full state dict
# Output: a PARTIAL dict with only the keys this node updates

def writer(state: WritingState) -> dict:
    """Writes a first draft on the given topic."""
    response = llm.invoke([
        SystemMessage(content="You are a skilled blog writer. Write clearly and engagingly."),
        HumanMessage(content=f"Write a concise 3-paragraph blog post about: {state['topic']}")
    ])
    print("✏️  Writer finished draft.")
    return {"draft": response.content}   # only update the "draft" key

def critic(state: WritingState) -> dict:
    """Reads the draft and provides specific, actionable critique."""
    response = llm.invoke([
        SystemMessage(content="You are a demanding editor. Give 3 specific improvement suggestions."),
        HumanMessage(content=f"Critique this blog post:\n\n{state['draft']}")
    ])
    print("🔍  Critic finished review.")
    return {"critique": response.content}

def editor(state: WritingState) -> dict:
    """Rewrites the draft incorporating the critic's feedback."""
    response = llm.invoke([
        SystemMessage(content="You are a final editor. Rewrite to address all critique points."),
        HumanMessage(content=(
            f"Original draft:\n{state['draft']}\n\n"
            f"Critique to address:\n{state['critique']}\n\n"
            "Rewrite the post to be excellent."
        ))
    ])
    print("✨  Editor finished final version.")
    return {"final": response.content}

# ── Step 3: Build the Graph ───────────────────────────
builder = StateGraph(WritingState)

# Register nodes
builder.add_node("writer", writer)
builder.add_node("critic", critic)
builder.add_node("editor", editor)

# Define the linear flow: START → writer → critic → editor → END
builder.add_edge(START,    "writer")
builder.add_edge("writer", "critic")
builder.add_edge("critic", "editor")
builder.add_edge("editor", END)

# Compile into a runnable (works like any LCEL chain after this)
app = builder.compile()

# ── Step 4: Run the Graph ─────────────────────────────
initial_state = {"topic": "Why Python is still the best language for AI in 2025"}
result = app.invoke(initial_state)

print("\n" + "="*60)
print("FINAL POLISHED POST:")
print("="*60)
print(result["final"])

# You can inspect every state key:
print("\n--- CRITIQUE THAT WAS APPLIED ---")
print(result["critique"])

from typing import TypedDict, Annotated, Sequence
import operator
from langgraph.graph import StateGraph, START, END
from langgraph.prebuilt import ToolNode
from langchain_openai import ChatOpenAI
from langchain_core.messages import BaseMessage, HumanMessage
from langchain_core.tools import tool
from langchain_community.tools import WikipediaQueryRun
from langchain_community.utilities import WikipediaAPIWrapper
import math

# ── Tools ─────────────────────────────────────────────
@tool
def calculate(expression: str) -> str:
    """Evaluate a math expression. Input example: '50000 * (1.12 ** 10)'"""
    try:
        safe = {k: getattr(math, k) for k in dir(math) if not k.startswith("_")}
        return str(eval(expression, {"__builtins__": {}}, safe))
    except Exception as e:
        return f"Error: {e}"

wiki = WikipediaQueryRun(api_wrapper=WikipediaAPIWrapper(doc_content_chars_max=800))
tools = [calculate, wiki]

# ── State: uses Annotated[list, operator.add] for message accumulation ──
# This tells LangGraph: when a node updates "messages", ADD to the list,
# don't overwrite it. This is the standard pattern for agent message history.
class AgentState(TypedDict):
    messages: Annotated[Sequence[BaseMessage], operator.add]

llm_with_tools = ChatOpenAI(model="gpt-4o-mini").bind_tools(tools)

# ── Nodes ─────────────────────────────────────────────
def call_llm(state: AgentState) -> dict:
    """The LLM node: looks at messages, decides to call a tool or produce final answer."""
    response = llm_with_tools.invoke(state["messages"])
    return {"messages": [response]}  # appended via operator.add

# ToolNode is a prebuilt LangGraph node that executes whatever tool
# calls are present in the last AIMessage's tool_calls field.
tool_node = ToolNode(tools)

# ── Conditional Edge: should we call a tool or stop? ──
def should_continue(state: AgentState) -> str:
    last_msg = state["messages"][-1]
    if hasattr(last_msg, "tool_calls") and last_msg.tool_calls:
        return "tools"   # go to tool_node
    return END           # LLM produced a final text answer, stop

# ── Build the cyclic agent graph ──────────────────────
builder = StateGraph(AgentState)
builder.add_node("llm",   call_llm)
builder.add_node("tools", tool_node)

builder.add_edge(START,   "llm")
builder.add_conditional_edges(
    "llm",
    should_continue,            # routing function
    {"tools": "tools", END: END}  # map return values → next nodes
)
builder.add_edge("tools", "llm")   # ← THE CYCLE: after tool, back to LLM

app = builder.compile()

# ── Run ───────────────────────────────────────────────
question = "What is 50000 * (1.12 ** 10)? Also look up compound interest on Wikipedia."
result = app.invoke({"messages": [HumanMessage(content=question)]})

print("\n--- Agent Final Answer ---")
print(result["messages"][-1].content)

# Print the full trace to see every tool call and response
print("\n--- Full Message Trace ---")
for msg in result["messages"]:
    role = type(msg).__name__
    content_preview = str(msg.content)[:80]
    print(f"[{role}]: {content_preview}...")

from typing import TypedDict, Literal
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage

# ── Human-in-the-loop pattern ─────────────────────────
#
# The graph generates an email draft, then PAUSES and waits
# for a human to approve or reject before sending.
# This is achieved with an "interrupt_before" checkpoint.
# The graph state is saved to MemorySaver so it can resume
# exactly where it left off after the human responds.

class EmailState(TypedDict):
    recipient:    str
    subject:      str
    draft:        str
    human_review: str   # "approve" or "reject"
    sent:         bool

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.4)

def draft_email(state: EmailState) -> dict:
    """Node 1: generate an email draft."""
    response = llm.invoke([
        HumanMessage(content=(
            f"Write a professional email to {state['recipient']} "
            f"with subject: {state['subject']}. Keep it under 150 words."
        ))
    ])
    print(f"\n📧 Draft ready for your review:\n{response.content}")
    return {"draft": response.content}

def send_email(state: EmailState) -> dict:
    """Node 3: actually send (or simulate sending) the email."""
    print(f"\n✅ Email sent to {state['recipient']}!")
    return {"sent": True}

def discard_email(state: EmailState) -> dict:
    """Node 3 (alt): human rejected — discard."""
    print("\n🗑️  Email discarded by reviewer.")
    return {"sent": False}

def human_decision(state: EmailState) -> str:
    """Routing function: reads human_review key and routes to send or discard."""
    if state.get("human_review") == "approve":
        return "send"
    return "discard"

# ── Build graph ───────────────────────────────────────
builder = StateGraph(EmailState)
builder.add_node("draft",   draft_email)
builder.add_node("send",    send_email)
builder.add_node("discard", discard_email)

builder.add_edge(START,    "draft")
builder.add_conditional_edges("draft", human_decision, {"send": "send", "discard": "discard"})
builder.add_edge("send",    END)
builder.add_edge("discard", END)

# MemorySaver persists graph state across .invoke() calls (within the same process).
# interrupt_before=["draft"] pauses the graph BEFORE executing "draft" — but here
# we use the human_review key in state as a manual gate instead (simpler for learning).
memory = MemorySaver()
app = builder.compile(checkpointer=memory)

Architecture Patterns —
the five graph shapes

Just as design patterns in software engineering give names to recurring solutions — Factory, Observer, Strategy — LangGraph workflows tend to follow a small set of recurring graph shapes. Recognising which shape fits your problem is the first step to architecting a clean solution. The five patterns below cover the vast majority of real-world LangGraph applications.

The Sequential Pipeline is the simplest: nodes connected in a straight line, no branching, no loops. It's equivalent to an LCEL chain expressed as a graph — useful when you need shared state across steps but don't need cycles. The Router pattern adds a single conditional edge that dispatches to one of several specialist nodes based on the input — think of an intent classifier that routes "write code", "answer question", or "search the web" to different agents. The Evaluator-Optimizer Loop adds a cycle: generate → evaluate quality → if not good enough, go back and regenerate with the critique as additional context. This is the pattern for self-improving agents.

The Supervisor-Worker pattern has a central "supervisor" node that delegates subtasks to specialised worker agents and aggregates their results. The supervisor decides which worker to call next based on the current state of the task. Finally, the Map-Reduce pattern processes multiple inputs independently and merges the results — parallelize 10 document summaries, then combine them into one final report. This pattern uses LangGraph's Send API to dynamically spawn parallel branches at runtime.

🗺️ The Five Graph Shapes
Sequential = A → B → C → END ← like LCEL but with shared state
Router = START → classifier → [A | B | C] → END ← conditional dispatch
Eval-Loop = generate → evaluate → [improve → generate | END] ← self-critique cycle
Supervisor = supervisor ⇄ [worker_A | worker_B | worker_C] ← multi-agent delegation
Map-Reduce = fan_out → [parallel tasks] → fan_in → summarise ← dynamic parallel branches
When to use Eval-LoopWhen output quality matters more than speed. Typical for long-form writing, code generation, complex reasoning. Cap iterations with a counter in state to prevent infinite loops.
When to use SupervisorWhen your task naturally decomposes into specialist sub-problems. Example: a research agent that has a web-search worker, a calculator worker, and a citation-formatter worker.
Send API (Map-Reduce)langgraph.constants.Send lets a node emit multiple messages to the same downstream node in parallel at runtime — without knowing the number of messages at compile time.

Self-Critique Agent —
the evaluator-optimizer loop

One of the most powerful ideas in modern AI engineering is having the model evaluate its own output and improve it iteratively. This sounds circular, but it works remarkably well in practice: the model's ability to critique text is significantly better than its first-attempt generation, because critique is a simpler cognitive task than open-ended creation. The pattern is: generate → critique → rewrite incorporating the critique → critique again → repeat until quality threshold is met or iteration limit is reached.

This LangGraph workflow implements a blog post writer that iterates until a quality score of 8/10 or higher is achieved (or 3 attempts are used). The state tracks the current draft, the list of critiques from each iteration, and a counter to prevent infinite loops. The conditional edge reads the quality score from the last critique and routes back to the writer node if improvements are still needed.

import os
os.environ["OPENAI_API_KEY"] = "sk-your-key"

from typing import TypedDict, List, Annotated
from pydantic import BaseModel, Field
from langgraph.graph import StateGraph, START, END
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage

# ── State ─────────────────────────────────────────────
class WritingState(TypedDict):
    topic:      str
    draft:      str
    critiques:  List[str]   # history of all critiques across iterations
    score:      int          # quality score from last critique (1-10)
    iteration:  int          # how many write→critique cycles have run

# ── Critique schema ───────────────────────────────────
class Critique(BaseModel):
    score:          int       = Field(description="Quality 1-10. 8+ means publish-ready.", ge=1, le=10)
    strengths:      List[str] = Field(description="What works well")
    improvements:   List[str] = Field(description="3 specific things to improve")
    rewrite_instructions: str = Field(description="One clear instruction for the next draft")

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.7)
eval_llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

# ── Nodes ─────────────────────────────────────────────
def write_draft(state: WritingState) -> dict:
    """Write or rewrite the draft, incorporating previous critique if any."""
    iteration = state.get("iteration", 0)
    critiques  = state.get("critiques", [])

    if iteration == 0:
        instruction = f"Write a compelling 3-paragraph blog post about: {state['topic']}"
    else:
        last_critique = critiques[-1]
        instruction = f"""Rewrite this blog post about '{state['topic']}' to address this critique:
{last_critique}

Current draft:
{state['draft']}"""

    response = llm.invoke([HumanMessage(content=instruction)])
    print(f"✏️  Iteration {iteration + 1}: draft written ({len(response.content)} chars)")
    return {"draft": response.content, "iteration": iteration + 1}

def critique_draft(state: WritingState) -> dict:
    """Evaluate the current draft and produce a structured critique."""
    critique: Critique = eval_llm.with_structured_output(Critique).invoke([
        HumanMessage(content=f"""Evaluate this blog post about '{state['topic']}' as a demanding editor.
Be strict — only give 8+ if it's genuinely excellent and ready to publish.

{state['draft']}""")
    ])

    critique_text = (
        f"Score: {critique.score}/10. "
        f"To improve: {'; '.join(critique.improvements)}. "
        f"Instruction: {critique.rewrite_instructions}"
    )
    print(f"🔍  Critique score: {critique.score}/10")
    return {"critiques": state.get("critiques", []) + [critique_text], "score": critique.score}

# ── Routing function ──────────────────────────────────
def should_rewrite(state: WritingState) -> str:
    """Continue rewriting if score < 8 AND we haven't hit the iteration limit."""
    if state["score"] >= 8 or state["iteration"] >= 3:
        return END         # good enough, or limit reached
    return "write"       # loop back to writer with critique

builder.add_node("critique", critique_draft)
builder.add_edge(START,    "write")
builder.add_edge("write",  "critique")
builder.add_conditional_edges("critique", should_rewrite, {"write": "write", END: END})

Multi-Agent Research Pipeline —
the supervisor pattern

Some tasks are genuinely too complex for a single agent. A research question like "What are the pros and cons of microservices vs monoliths for a startup?" benefits from specialist agents working in parallel: one that searches Wikipedia for background theory, one that searches for recent real-world case studies, and one that synthesises all the gathered information into a coherent report. This is the Supervisor-Worker pattern.

The Supervisor node holds the master state and decides which worker to invoke next — or whether enough information has been gathered to produce the final report. Workers are isolated; they each receive a subtask and return their findings, never talking to each other directly. The Supervisor accumulates findings in state and delegates until it judges the task complete. This architecture scales gracefully: adding a new specialist is just adding a new node and updating the Supervisor's routing logic.

import os
os.environ["OPENAI_API_KEY"] = "sk-your-key"

from typing import TypedDict, List, Literal, Optional
from pydantic import BaseModel
from langgraph.graph import StateGraph, START, END
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage
from langchain_community.tools import WikipediaQueryRun
from langchain_community.utilities import WikipediaAPIWrapper

wiki = WikipediaQueryRun(api_wrapper=WikipediaAPIWrapper(doc_content_chars_max=1500))

# ── State ─────────────────────────────────────────────
class ResearchState(TypedDict):
    question:       str
    background:     str        # filled by researcher_A
    pros_cons:      str        # filled by researcher_B
    examples:       str        # filled by researcher_C
    final_report:   str        # filled by synthesiser
    next_worker:    str        # supervisor decides who goes next

# ── Routing schema ────────────────────────────────────
class SupervisorDecision(BaseModel):
    next: Literal["researcher_A", "researcher_B", "researcher_C", "synthesiser"]
    reason: str

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

# ── Supervisor ────────────────────────────────────────
def supervisor(state: ResearchState) -> dict:
    """Decide which worker to run next based on what's already been done."""
    status = []
    if not state.get("background"):  status.append("background research not done")
    if not state.get("pros_cons"):   status.append("pros/cons analysis not done")
    if not state.get("examples"):    status.append("real-world examples not gathered")
    if not status:                      status.append("all research complete — ready to synthesise")

    decision: SupervisorDecision = llm.with_structured_output(SupervisorDecision).invoke([
        HumanMessage(content=f"""You manage a research team. Based on the status below, decide who to assign next.

Available workers:
- researcher_A: gathers background/theory (run first)
- researcher_B: analyses pros and cons (run second)  
- researcher_C: finds real-world examples (run third)
- synthesiser: writes the final report (only when all 3 researchers are done)

Current status: {'; '.join(status)}
Research question: {state['question']}""")
    ])
    print(f"🎯 Supervisor → {decision.next}: {decision.reason}")
    return {"next_worker": decision.next}

# ── Workers ───────────────────────────────────────────
def researcher_A(state: ResearchState) -> dict:
    """Gather background theory from Wikipedia."""
    print("🔬 Researcher A: gathering background...")
    wiki_results = wiki.run(state["question"])
    response = llm.invoke([HumanMessage(content=
        f"Using this Wikipedia context, write a 2-paragraph background on '{state['question']}':\n{wiki_results}"
    )])
    return {"background": response.content}

def researcher_B(state: ResearchState) -> dict:
    """Analyse pros and cons using existing background."""
    print("📊 Researcher B: analysing pros and cons...")
    response = llm.invoke([HumanMessage(content=
        f"Given this background:\n{state['background']}\n\nList 3 pros and 3 cons regarding: '{state['question']}'"
    )])
    return {"pros_cons": response.content}

def researcher_C(state: ResearchState) -> dict:
    """Find real-world examples."""
    print("🌍 Researcher C: finding examples...")
    response = llm.invoke([HumanMessage(content=
        f"Give 3 concrete real-world company examples relevant to: '{state['question']}'. Be specific about company names and outcomes."
    )])
    return {"examples": response.content}

def synthesiser(state: ResearchState) -> dict:
    """Write the final research report from all gathered material."""
    print("📝 Synthesiser: writing final report...")
    response = llm.invoke([HumanMessage(content=f"""Write a structured research report answering: '{state['question']}'

Use this material:
BACKGROUND: {state['background']}
PROS/CONS: {state['pros_cons']}
EXAMPLES: {state['examples']}

Format: Executive Summary, Background, Analysis, Real-World Evidence, Conclusion.""")])
    return {"final_report": response.content}

# ── Routing function ──────────────────────────────────
def route_to_worker(state: ResearchState) -> str:
    return state["next_worker"]

# ── Build graph ───────────────────────────────────────
builder = StateGraph(ResearchState)
for name, fn in [("supervisor", supervisor), ("researcher_A", researcher_A),
                   ("researcher_B", researcher_B), ("researcher_C", researcher_C),
                   ("synthesiser", synthesiser)]:
    builder.add_node(name, fn)

builder.add_edge(START, "supervisor")
for worker in ["researcher_A", "researcher_B", "researcher_C"]:
    builder.add_edge(worker, "supervisor")   # workers always report back to supervisor
builder.add_conditional_edges("supervisor", route_to_worker, {
    "researcher_A": "researcher_A", "researcher_B": "researcher_B",
    "researcher_C": "researcher_C", "synthesiser": "synthesiser"
})
builder.add_edge("synthesiser", END)
app = builder.compile()

# ── Run ───────────────────────────────────────────────
result = app.invoke({
    "question":     "Microservices vs monolith architecture: what's right for an early-stage startup?",
    "background":   "", "pros_cons": "", "examples": "",
    "final_report": "", "next_worker": ""
})
print(f"\n{'='*60}\nFINAL RESEARCH REPORT\n{'='*60}")
print(result["final_report"])

A/B Prompt Testing —
measure what you change

Every experienced prompt engineer has had this experience: you tweak a prompt, it seems better on the examples you tested manually, you deploy it, and three days later you notice the outputs have quietly gotten worse on a different type of input. This is the core problem with gut-feel prompt iteration — your informal testing set is too small and too biased toward inputs you've already seen.

Systematic A/B testing with LangSmith solves this. The workflow is: build a representative Dataset of 20-50 test cases covering the full range of your input distribution; define one or more evaluators that score correctness, helpfulness, or whatever you care about; run both the old prompt (Variant A) and the new prompt (Variant B) against the full dataset; compare their scores statistically. This gives you a number — prompt B is 15% better on the metric you care about — instead of a feeling.

            print(f"   [{r.id}] {str(r.error)[:100]}")

25 LangServe

LangServe — your chain
becomes an API in minutes

You've built a chain. It works in a notebook. Now someone asks: "Can we put this in a product?" The traditional answer involves writing a Flask or FastAPI server, defining request/response schemas, handling CORS, writing a Swagger UI, and wiring up streaming endpoints — probably a full day of work. LangServe collapses all of that into about 10 lines of code.

LangServe is a library built on top of FastAPI that takes any LCEL chain and automatically exposes it as a production-ready REST API. You get five endpoints for free: /invoke (one input, one output), /batch (multiple inputs), /stream (server-sent events for token streaming), /stream_log (streaming with intermediate step visibility), and /playground (an interactive UI you can open in a browser and test your chain immediately, no frontend required).

The architecture is straightforward: you create a FastAPI app, call add_routes(app, your_chain, path="/endpoint"), and run it with Uvicorn. LangServe handles schema generation, input validation, CORS, and the playground UI automatically. You can serve multiple chains from the same server by calling add_routes multiple times with different paths. The result is a proper microservice that any frontend, mobile app, or other backend service can consume.

On the consumer side, RemoteRunnable is the mirror image. It wraps a deployed LangServe endpoint in a standard LangChain Runnable interface — meaning you can call .invoke(), .stream(), and .batch() on it exactly as if the chain were running locally. This makes it trivial to compose a client application that orchestrates multiple deployed chains, or to test a deployed endpoint from a Jupyter notebook without writing any HTTP code.

🚀 LangServe in Four Lines
from fastapi import FastAPI
from langserve import add_routes
app = FastAPI()
add_routes(app, your_chain, path="/chat") ← that's it
POST /chat/invokeSynchronous. Sends {"input": {...}} and waits for the full {"output": ...} response. The standard endpoint for most integrations.
POST /chat/batchSends {"inputs": [{...}, {...}]} and returns {"outputs": [...]}. Runs all inputs concurrently server-side.
POST /chat/streamServer-Sent Events. Each event is a token chunk. Perfect for streaming UIs — the browser receives tokens as they're generated.
GET /chat/playgroundAn interactive browser UI built into LangServe. Paste inputs, run the chain, see outputs and intermediate steps. Zero frontend code needed.
RemoteRunnable("http://host/chat")A drop-in client that calls the LangServe API but feels exactly like a local chain. Supports .invoke(), .stream(), .batch(), and async variants.
📖
Where LangServe fits in the stack. Think of it as the deployment layer that sits on top of the other three tools: you build chains with LangChain, orchestrate them with LangGraph, observe them with LangSmith, and then expose them to the world with LangServe. A complete production AI feature might use all four — a LangGraph workflow running inside a LangServe endpoint, with every invocation traced in LangSmith.

REST API Deployment —
four chains, one server

This example shows a production-style LangServe server that exposes four different chains from a single FastAPI application. Each chain lives at its own URL path. The server includes a custom lifespan function that warms up models on startup, proper CORS configuration for browser clients, and input type annotations that LangServe uses to generate accurate OpenAPI documentation.

This is the standard pattern for deploying a suite of AI features as a microservice: one server process, multiple chains, each independently callable. You can run this locally with uvicorn server:app --reload, open http://localhost:8000/docs for the auto-generated Swagger UI, or navigate to /translate/playground for the interactive LangServe playground.

import os
os.environ["OPENAI_API_KEY"] = "sk-your-key"

from contextlib import asynccontextmanager
from typing import List
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
from langserve import add_routes
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnableParallel, RunnablePassthrough
from pydantic import BaseModel, Field

# ── Models ────────────────────────────────────────────
llm      = ChatOpenAI(model="gpt-4o-mini", temperature=0)
llm_warm = ChatOpenAI(model="gpt-4o-mini", temperature=0.7)
parser   = StrOutputParser()

# ── Chain 1: Translation ──────────────────────────────
class TranslateInput(BaseModel):
    text:     str  = Field(description="Text to translate")
    language: str  = Field(description="Target language (e.g. French, Hindi, Japanese)")

translate_chain = (
    ChatPromptTemplate.from_messages([
        ("system", "You are a professional translator. Translate accurately, preserving tone."),
        ("human",  "Translate to {language}: {text}")
    ])
    | llm | parser
)

# ── Chain 2: Summariser ───────────────────────────────
class SummariseInput(BaseModel):
    text:       str = Field(description="Document to summarise")
    max_words:  int = Field(default=100, description="Maximum words in summary")

summarise_chain = (
    ChatPromptTemplate.from_messages([
        ("system", "You are a precise summariser. Never exceed the word limit."),
        ("human",  "Summarise in {max_words} words or fewer:\n\n{text}")
    ])
    | llm | parser
)

# ── Chain 3: Sentiment + keywords (parallel output) ───
class AnalyseInput(BaseModel):
    text: str = Field(description="Text to analyse")

analyse_chain = RunnableParallel(
    sentiment = (
        ChatPromptTemplate.from_template("Give one word sentiment (positive/negative/neutral): {text}")
        | llm | parser
    ),
    keywords = (
        ChatPromptTemplate.from_template("List 3 keywords as comma-separated values from: {text}")
        | llm | parser
    ),
    summary = (
        ChatPromptTemplate.from_template("Summarise in one sentence: {text}")
        | llm | parser
    ),
)

# ── Chain 4: Creative brainstorm ─────────────────────
class BrainstormInput(BaseModel):
    topic:  str = Field(description="Topic to brainstorm about")
    count:  int = Field(default=5, description="Number of ideas to generate")

brainstorm_chain = (
    ChatPromptTemplate.from_messages([
        ("system", "You are a creative strategist. Generate bold, specific, non-obvious ideas."),
        ("human",  "Generate exactly {count} creative ideas for: {topic}")
    ])
    | llm_warm | parser
)

# ── FastAPI app + CORS ───────────────────────────────
@asynccontextmanager
async def lifespan(app: FastAPI):
    print("🚀 LangServe API starting up...")
    yield
    print("👋 Shutting down.")

app = FastAPI(
    title="LangChain API",
    description="Multi-chain AI service powered by LangChain + LangServe",
    version="1.0.0",
    lifespan=lifespan
)

app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],   # restrict in production to your frontend domain
    allow_methods=["*"],
    allow_headers=["*"],
)

# ── Register all chains ───────────────────────────────
add_routes(app, translate_chain.with_types(input_type=TranslateInput),   path="/translate")
add_routes(app, summarise_chain.with_types(input_type=SummariseInput),   path="/summarise")
add_routes(app, analyse_chain.with_types(input_type=AnalyseInput),       path="/analyse")
add_routes(app, brainstorm_chain.with_types(input_type=BrainstormInput), path="/brainstorm")

# Each chain is now available at:
#   POST http://localhost:8000/{path}/invoke
#   POST http://localhost:8000/{path}/batch
#   POST http://localhost:8000/{path}/stream
#   GET  http://localhost:8000/{path}/playground   ← interactive UI
#   GET  http://localhost:8000/docs                 ← OpenAPI Swagger UI

if __name__ == "__main__":

Full-Stack Deployment —
RAG chain + LangGraph + LangSmith

This final example pulls the entire ecosystem together. A LangGraph self-critique workflow (from §21) is deployed as a LangServe endpoint, with LangSmith tracing every invocation automatically. This is what a complete, production-grade AI feature looks like: the logic lives in LangGraph, the deployment lives in LangServe, and the observability lives in LangSmith.

The critical insight here is how seamlessly the tools compose. The LangGraph app is just a Runnable — it has .invoke() and .stream(). LangServe accepts any Runnable, so you pass the compiled graph directly to add_routes. LangSmith is already watching because the environment variables are set. No adapter code, no glue layer. The same architectural principle that makes LCEL chains composable with | makes the whole ecosystem composable at the deployment level.

full_stack_server.py
# Install: pip install langchain langchain-openai langgraph langserve fastapi uvicorn langsmith pydantic

import os
# LangSmith — traces every LangGraph node invocation automatically
os.environ["LANGCHAIN_TRACING_V2"] = "true"
os.environ["LANGCHAIN_API_KEY"]     = "ls__your_key"
os.environ["LANGCHAIN_PROJECT"]     = "full-stack-demo"
os.environ["OPENAI_API_KEY"]        = "sk-your-key"

from typing import TypedDict, List
from pydantic import BaseModel, Field
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
from langserve import add_routes
from langgraph.graph import StateGraph, START, END
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.7)

# ── LangGraph: self-critique writer ───────────────────
class WriterState(TypedDict):
    topic:     str
    draft:     str
    critique:  str
    iteration: int

def write(state):
    prompt = (
        f"Write a 2-paragraph blog post about: {state['topic']}"
        if state["iteration"] == 0
        else f"Rewrite this draft to address: {state['critique']}\n\nDraft:\n{state['draft']}"
    )
    draft = llm.invoke([HumanMessage(content=prompt)]).content
    return {"draft": draft, "iteration": state["iteration"] + 1}

def critique(state):
    crit = ChatOpenAI(model="gpt-4o-mini", temperature=0).invoke([
        HumanMessage(content=f"Rate this post 1-10 and give ONE improvement. Start response with the number.\n\n{state['draft']}")
    ]).content
    return {"critique": crit}

def should_stop(state):
    try:
        score = int(state["critique"].split()[0].replace(".", ""))
        if score >= 8 or state["iteration"] >= 3: return END
    except: pass
    return "write"

builder = StateGraph(WriterState)
builder.add_node("write",   write)
builder.add_node("critique", critique)
builder.add_edge(START,    "write")
builder.add_edge("write",  "critique")
builder.add_conditional_edges("critique", should_stop, {"write": "write", END: END})
writer_graph = builder.compile()
# writer_graph is a Runnable — pass it directly to add_routes

# ── Simple LangChain chain ─────────────────────────────
class QAInput(BaseModel):
    question: str = Field(description="Your question")

qa_chain = (
    ChatPromptTemplate.from_messages([
        ("system", "You are a helpful assistant. Answer concisely."),
        ("human",  "{question}")
    ])
    | llm | StrOutputParser()
)

# ── FastAPI app ───────────────────────────────────────
app = FastAPI(title="LangChain Ecosystem API", version="1.0.0")
app.add_middleware(CORSMiddleware, allow_origins=["*"], allow_methods=["*"], allow_headers=["*"])

# LangGraph graph deployed at /writer — automatically traced by LangSmith
add_routes(app, writer_graph,                              path="/writer")
add_routes(app, qa_chain.with_types(input_type=QAInput),  path="/qa")

# Endpoints available:
#   POST /writer/invoke   — runs the full LangGraph cycle, returns final state
#   POST /writer/stream   — streams state updates as graph nodes complete
#   POST /qa/invoke       — simple Q&A
#   GET  /writer/playground
#   GET  /qa/playground
#   GET  /docs            — OpenAPI UI with both chains documented

    import uvicorn
