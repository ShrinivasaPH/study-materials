What is Git?
Version control, demystified

Imagine you're writing a novel. You save draft1.docx, then draft1_FINAL.docx, then draft1_FINAL_v2_USE_THIS_ONE.docx… Sound familiar? That's manual version control — chaotic, error-prone, and a nightmare for collaboration.

Git is a distributed version control system that elegantly solves this by tracking every change to your files over time. Think of it as a time machine + a collaboration superpower, all bundled into a command-line tool.

📸 Snapshots, Not Diffs
Unlike other VCS tools, Git stores snapshots of your entire project at each commit. If a file hasn't changed, it just references the previous version — blazing fast.
🌐 Distributed
Every developer has a full copy of the repository, including all history. No single point of failure. Work offline, sync later.
⚡ Blazingly Fast
Almost all operations are local. Branching takes milliseconds, not minutes. Git was built for the Linux kernel — it had to be fast.
🎉
Fun FactLinus Torvalds created Git in 2005 over a single weekend — out of frustration with existing tools. He famously named it "git" (British slang for a stupid person) saying "I name all my projects after myself."

Setup & Configuration
Get Git ready for you

Before you do anything with Git, tell it who you are. This identity gets stamped onto every commit you make — your personal watermark on the codebase.

bash — First-Time Setup
# Your identity — used in every commit git config --global user.name "Shrinivasa PH" git config --global user.email "you@example.com" # Your preferred editor (e.g., VS Code, vim, nano) git config --global core.editor "code --wait" # Default branch name (modern best practice) git config --global init.defaultBranch main # View all your config settings git config --list # Or check a specific one: git config user.name Shrinivasa PH
💡
Tip — Config ScopesGit config has three scopes: --system (all users on machine), --global (your user account), and --local (just this repo). Local overrides global overrides system. Use --local inside a work repo to set a different email than your personal one.

Initialising a Repository

bash
# Start a brand new repo in the current folder git init Initialized empty Git repository in /your/project/.git/ # OR: clone an existing remote repo git clone https://github.com/user/repo.git git clone https://github.com/user/repo.git my-local-name # custom folder name # Check your repo status anytime git status
📁
What is .git?The .git folder is the heart of your repo — it stores all the history, objects, refs, and config. Never manually edit files inside it unless you really know what you're doing. Deleting it removes all version history.

File Lifecycle & Staging
add, status, diff, rm, mv

Git's staging area (also called the index) is one of its most powerful — and initially confusing — features. Think of it as a draft board. You decide exactly which changes go into the next commit, giving you surgical control over your history.

bash — Core Staging Commands
# Stage a specific file git add filename.py # Stage a directory git add src/ # Stage ALL changes (new, modified, deleted) git add -A # or: git add --all # Stage interactively — choose individual chunks (VERY POWERFUL) git add -p # presents each change as a "hunk" for y/n/s/e # Unstage a file (keep changes, just unstage) git restore --staged filename.py # Discard changes in working directory (DESTRUCTIVE — can't undo!) git restore filename.py
⚠️
Gotcha — git add .git add . stages all changes in the current directory and below. If you're in a subdirectory, files above it won't be staged. Use git add -A from any directory to get everything.
bash
git status # full output git status -s # short format (great for quick glance) M src/app.py # M = modified (staged) M src/utils.py # M = modified (unstaged) ?? temp.txt # ?? = untracked file A new_feature.py # A = newly staged file D old_file.py # D = deleted
bash
git diff # unstaged changes vs last commit git diff --staged # staged changes vs last commit git diff main feature # diff between two branches git diff HEAD~3 # changes since 3 commits ago git diff --stat # summary: files changed, insertions, deletions git diff --word-diff # show word-level changes (great for docs)
.gitignore
# Ignore compiled Python files __pycache__/ *.pyc *.pyo # Ignore virtual environments venv/ .env # Ignore secrets .env.local config/secrets.json # Ignore OS files .DS_Store Thumbs.db # Ignore build output dist/ build/ *.egg-info/
💡
TipUse gitignore.io to generate .gitignore files for any tech stack. Committed a file you now want to ignore? Run git rm --cached filename to stop tracking it without deleting it.

Commits & History
Snapshot your progress like a pro

A commit is a permanent snapshot of your staged changes, along with a message, timestamp, and a pointer to its parent. Think of commits as diary entries — brief, clear, and specific.

bash — Committing
# Standard commit (opens editor for message) git commit # Commit with inline message git commit -m "feat: add user authentication module" # Stage all tracked files AND commit in one step git commit -am "fix: handle null pointer in login flow" # Amend the last commit (fix message or add forgotten file) git commit --amend -m "feat: add user auth with email verification" # ⚠️ Don't amend commits you've already pushed!

Reading History

bash — git log
git log # full history git log --oneline # compact (one line per commit) git log --oneline --graph --all # visual branch graph git log --author="Shrinivasa" # filter by author git log --since="2 weeks ago" # filter by time git log -p src/models.py # show diffs for a specific file git log --grep="auth" # search commit messages git shortlog -sn # commit count by author # Sample output of --oneline --graph * a3f9b12 (HEAD → main) feat: dashboard v2 * | c9d1e45 (feature/dark-mode) add theme toggle * | 4b7a833 wip: dark mode styles |/ * 1de0c92 fix: resolve login bug

The Anatomy of a Good Commit Message

✅
The Conventional Commits formatUse: type(scope): short description. Types include: feat (new feature), fix (bug fix), docs, style, refactor, test, chore. Keep the summary under 72 characters. Use the body to explain why, not what.
✓ GOOD
commit message
feat(auth): add JWT refresh token support Tokens now auto-refresh 5min before expiry. Resolves flaky session timeouts in prod. Closes #142
✗ BAD
commit message
fixed stuff wip asdfgh update changes
😄
Fun FactThere are entire websites dedicated to terrible commit messages. The classic is "initial commit" followed immediately by "actually initial commit" followed by "ok NOW initial commit". We've all been there.

Branching
Parallel universes for your code

Branches are one of Git's killer features. A branch is just a lightweight pointer to a commit — creating one is nearly instant and costs almost no disk space. This lets you work on features, fixes, or experiments in total isolation.

The HEAD pointer tells Git "you are here" — which branch and commit you're currently on.

# Visual: your branch tree main ──A──B──C──────────────────────M ← (HEAD, main) \ / feature D──E──F──────────── ← merged in main ──A──B──C────────────────────────── ← production branch \ hotfix H ← (HEAD, hotfix) quick bug fix
bash — Branch Operations
# Create a new branch git branch feature/user-auth # Create AND switch to it (most common workflow) git switch -c feature/user-auth # (older syntax: git checkout -b feature/user-auth) # Switch to existing branch git switch main # List all branches (local) git branch # List all branches (local + remote) git branch -a # Rename a branch git branch -m old-name new-name # Delete a branch (only if merged) git branch -d feature/user-auth # Force delete (even if not merged) git branch -D feature/user-auth # See which branches are merged into main git branch --merged main
🏷️
Branch Naming ConventionsCommon patterns: feature/, fix/, hotfix/, release/, chore/. A good name tells you the purpose at a glance. Avoid generic names like test or my-branch.
🔀
Merging & Conflict Resolution
Bringing code streams together

Merging combines the history of two branches into one. Git is usually smart enough to do this automatically. When it isn't — because two people edited the same lines — you get a merge conflict, which you resolve manually.

A fast-forward merge happens when the target branch has no new commits since the feature branch was created. Git simply moves the pointer forward — no merge commit needed.

# Before: main ──A──B \ feature C──D ← HEAD # After: git switch main && git merge feature main ──A──B──C──D ← (HEAD, main, feature)
bash
git switch main git merge feature Updating b3a4c12..e9f1234 Fast-forward # Use --no-ff to force a merge commit even when FF is possible git merge --no-ff feature

When both branches have diverged, Git performs a 3-way merge using the two branch tips plus their common ancestor. This creates a merge commit with two parents.

# Both branches have new commits: main ──A──B──E──F \ feature C──D # After merge: a new "merge commit" M is created main ──A──B──E──F────M ← (merge commit) \ / feature C──D────

When Git can't auto-merge (same lines changed differently), it pauses and shows you the conflict to resolve manually.

conflict markers in a file
<<<<<<< HEAD (your current branch) return "Hello, World!" ======= return "Hello, Git!" >>>>>>> feature/greeting (incoming branch)
bash — Resolving Conflicts
# 1. Edit the file to keep what you want (remove markers) # 2. Stage the resolved file git add greeting.py # 3. Complete the merge git commit # Prefer a visual merge tool git mergetool # Abort and go back to before the merge git merge --abort
💡
VS Code TipVS Code has an excellent built-in merge conflict editor — just open the conflicting file and click "Accept Current Change", "Accept Incoming Change", or "Accept Both Changes" inline.

Remote Repositories
Collaborate with the world

A remote is a version of your repository hosted somewhere else (GitHub, GitLab, Bitbucket, your own server). The conventional name for the primary remote is origin.

bash — Remote Operations
# View remotes git remote -v origin https://github.com/you/repo.git (fetch) origin https://github.com/you/repo.git (push) # Add a remote git remote add origin https://github.com/you/repo.git git remote add upstream https://github.com/original/repo.git # for forks # Push to remote (first time) git push -u origin main # After -u is set, you can just: git push # Push a new branch git push -u origin feature/user-auth # Fetch (download changes, don't merge) git fetch origin # Pull (fetch + merge in one step) git pull git pull --rebase # pull with rebase (cleaner history) # Delete a remote branch git push origin --delete feature/user-auth

fetch vs pull — What's the difference?

📥 git fetch
Downloads all changes from remote but does not touch your working directory. Safe to run anytime. Let's you inspect what changed before integrating: git log origin/main..main
⚡ git pull
Does git fetch + git merge in one go. Convenient but can create surprise merge commits. Using --rebase keeps history linear.
🔐
SSH vs HTTPSSSH keys (once set up) let you push without typing your password every time. Set up once: ssh-keygen -t ed25519 -C "you@example.com" then add the public key to GitHub. Switch a repo: git remote set-url origin git@github.com:you/repo.git

Undoing Things
Git's many ways to say "oops"

One of Git's greatest superpowers is that almost nothing is truly lost. But there are different "levels" of undoing, each with different consequences. Choose wisely.

🧹 git restore — discard uncommitted changes ⌄
bash
# Discard changes in working directory (⚠️ UNRECOVERABLE) git restore file.py git restore . # all files # Unstage a file (keeps your changes, just removes from staging) git restore --staged file.py
⚠️
Warninggit restore file.py permanently deletes your working directory changes. There's no undo. Use with care!
🔄 git revert — safely undo a commit (creates new commit) ⌄

The safe undo for shared history. Creates a new commit that reverses the changes from a previous commit. History is preserved — great for commits already pushed to remote.

bash
git revert a3f9b12 # revert by commit hash git revert HEAD # revert the last commit git revert HEAD~3..HEAD # revert a range of commits git revert --no-commit HEAD # stage the revert, don't auto-commit
⏮️ git reset — move HEAD backward (use with care!) ⌄

Three modes — from safest to most destructive:

ModeStagedWorking DirUse When
--softKept stagedKeptUndo commit, keep all changes staged
--mixed (default)UnstagedKeptUndo commit + unstage, keep changes
--hardDiscardedDiscardedNuke everything — go back completely
bash
git reset --soft HEAD~1 # undo last commit, keep staged git reset HEAD~1 # undo last commit, unstage changes git reset --hard HEAD~3 # go back 3 commits, nuke everything # Emergency recovery: reflog to the rescue! git reflog # see EVERY HEAD change ever git reset --hard HEAD@{4} # jump to 4 actions ago
🛟
Reflog is your safety net!git reflog records every movement of HEAD — even after --hard reset. You have ~90 days to recover "lost" commits this way. It's saved many careers.
🔁
Rebase
Rewrite history for a cleaner story

Rebasing is like telling Git: "pretend my branch started from here instead of there." It replays your commits on top of another branch, creating new commits with the same changes but a rewritten history. The result is a beautifully linear commit tree — no messy merge commits.

# Before rebase (feature started at B): main ──A──B──C──D \ feature X──Y # After: git switch feature && git rebase main main ──A──B──C──D \ feature X'──Y' ← replayed on top of D
bash — Rebase
# Standard rebase (replay feature on top of main) git switch feature git rebase main # Interactive rebase — the ultimate history editor git rebase -i HEAD~5 # open editor for last 5 commits # In the interactive editor, you can: # pick — keep commit as-is # squash — combine with previous commit # fixup — squash, discard this commit's message # reword — keep commit, change message only # drop — delete commit entirely # edit — pause to amend the commit # Abort if things go sideways git rebase --abort # Continue after resolving a conflict mid-rebase git rebase --continue
⚠️
The Golden Rule of RebaseNever rebase commits that have been pushed to a shared branch. Rebasing rewrites commits — if others have based work on those commits, you'll create a mess. Rebase only local, private branches.
🔀 Merge
Preserves history exactly. Creates a merge commit. Shows how and when branches were integrated. Best for public branches.
🔁 Rebase
Rewrites history to be linear. No merge commits. Cleaner log. Best for local feature branches before PR.
🎲
Fun FactThe merge vs. rebase debate is one of the great holy wars of software development, right up there with tabs vs. spaces. Entire engineering teams have strict policies one way or another. Both are valid — pick a convention and stick to it.

Git Stash
Temporarily shelve your work

Ever been mid-feature when an urgent bug report comes in? You can't commit half-done work, but you need a clean slate. Stash is your desk drawer — temporarily save messy WIP changes and retrieve them later.

bash — Stash Operations
# Save current changes to stash git stash git stash -m "wip: user auth refactor" # with a label git stash -u # include untracked files # View your stash stack git stash list stash@{0}: On main: wip: user auth refactor stash@{1}: WIP on feature: some older work # Apply and remove the most recent stash git stash pop # Apply without removing (keeps it in stash) git stash apply stash@{1} # Drop a specific stash git stash drop stash@{0} # Clear all stashes git stash clear # Create a branch from a stash (the cleanest approach) git stash branch feature/from-stash stash@{0}
💡
stash pop vs stash applyUse pop when you're done with the stash. Use apply when you want to apply the same stash to multiple branches (it stays in the list). If pop creates a conflict, the stash stays in the list automatically.
🪝
Git Hooks
Automate your workflow

Hooks are scripts that run automatically at specific points in the Git workflow. They live in .git/hooks/ and can be written in any scripting language. They're perfect for enforcing quality gates — running tests before commit, linting code, validating messages, deploying on push.

HookWhen it firesCommon Use
pre-commitBefore commit is createdRun linters, tests, format code
commit-msgAfter commit message is writtenValidate conventional commits format
prepare-commit-msgBefore editor opensAuto-populate message with ticket ID
post-commitAfter commit completesNotifications, CI triggers
pre-pushBefore pushing to remoteRun full test suite
post-mergeAfter merge completesInstall dependencies automatically
pre-rebaseBefore rebase startsSafety checks on shared branches
bash — .git/hooks/pre-commit
#!/bin/sh # Run Python linter before every commit echo "🔍 Running flake8..." if flake8 .; then echo "✅ All good!" exit 0 else echo "❌ Linting failed. Fix errors before committing." exit 1 # Non-zero exit aborts the commit fi
bash — Enable a Hook
# Make it executable (required!) chmod +x .git/hooks/pre-commit # Skip hooks when needed (use sparingly) git commit --no-verify -m "wip: quick save"
📦
Husky for TeamsThe .git/hooks/ folder isn't committed to the repo — so your teammates won't get your hooks automatically. Use Husky (Node.js) or pre-commit (Python) to manage hooks as part of your project setup, committing them to the repo.

Pro Tips & Power Features
Level up your Git game
🔍 git bisect — binary search for a bug ⌄

Found a bug that wasn't there a week ago? git bisect does a binary search through your history to find the exact commit that introduced it. Incredibly powerful.

bash
git bisect start git bisect bad # current commit is broken git bisect good v1.2.0 # this version was fine # Git checks out a midpoint commit. Test it, then: git bisect good # or: git bisect bad # Repeat until Git identifies the culprit commit git bisect reset # return to HEAD when done # Automate with a test script! git bisect run ./run_tests.sh # fully automated!
🍒 git cherry-pick — grab a specific commit ⌄

Sometimes you only want one specific commit from another branch — not the whole branch. Cherry-pick lets you selectively apply any commit onto your current branch.

bash
# Apply a single commit git cherry-pick a3f9b12 # Apply a range of commits git cherry-pick a3f9b12..e7c4d21 # Cherry-pick without committing (just stage the changes) git cherry-pick -n a3f9b12
🏷️ git tag — mark release versions ⌄
bash
# Create lightweight tag git tag v1.0.0 # Create annotated tag (recommended for releases) git tag -a v1.0.0 -m "Release 1.0.0 — stable" # Tag a specific commit git tag -a v0.9.0 a3f9b12 # List tags git tag -l "v1.*" # Push tags to remote (not automatic!) git push origin v1.0.0 git push origin --tags # push ALL tags
⚡ Aliases — your personal Git shortcuts ⌄
bash — ~/.gitconfig aliases
git config --global alias.st status git config --global alias.co checkout git config --global alias.br branch git config --global alias.unstage "restore --staged" git config --global alias.last "log -1 HEAD --stat" git config --global alias.lg "log --oneline --graph --all --decorate" git config --global alias.undo "reset HEAD~1 --soft" # Now you can use: git st git lg git undo
🔎 git log tips — find anything in history ⌄
bash
# Find who last changed a specific line git blame src/auth.py git blame -L 40,60 src/auth.py # only lines 40–60 # Find when a string was added or removed (pickaxe) git log -S "password_hash" --oneline # Find commits that touched a function git log -L :get_user:src/users.py # Pretty custom log format git log --format="%C(yellow)%h%Creset %s %C(cyan)(%cr)%Creset %C(green)<%an>"
