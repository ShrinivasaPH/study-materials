What is Git?
Version control, demystified

Imagine you're writing a novel. You save draft1.docx, then draft1_FINAL.docx, then draft1_FINAL_v2_USE_THIS_ONE.docx… Sound familiar? That's manual version control — chaotic, error-prone, and a nightmare for collaboration.

Git is a distributed version control system that elegantly solves this by tracking every change to your files over time. Think of it as a time machine + a collaboration superpower, all bundled into a command-line tool.

📸 Snapshots, Not Diffs
Git stores snapshots of your entire project at each commit. If a file hasn't changed, it references the previous version — blazing fast.
🌐 Distributed
Every developer has a full copy of the repository, including all history. No single point of failure. Work offline, sync later.
⚡ Blazingly Fast
Almost all operations are local. Branching takes milliseconds. Git was built for the Linux kernel — it had to be fast.
🎉
Fun FactLinus Torvalds created Git in 2005 in a single weekend — out of frustration with existing tools. He named it "git" (British slang for a stupid person), saying "I name all my projects after myself."

Setup & Configuration
Get Git ready for you

Before you do anything with Git, tell it who you are. This identity gets stamped onto every commit you make — your personal watermark on the codebase.

bash — First-Time Setup
# Your identity — used in every commit git config --global user.name "Shrinivasa PH" git config --global user.email "you@example.com" # Your preferred editor (e.g., VS Code, vim, nano) git config --global core.editor "code --wait" # Default branch name (modern best practice) git config --global init.defaultBranch main # View all your config settings git config --list git config user.name Shrinivasa PH
💡
Tip — Config ScopesGit config has three scopes: --system (all users), --global (your account), and --local (just this repo). Local overrides global overrides system. Use --local inside a work repo to set a different email.

Initialising a Repository

bash
# Start a new repo in the current folder git init Initialized empty Git repository in /your/project/.git/ # Clone an existing remote repo git clone https://github.com/user/repo.git git clone https://github.com/user/repo.git my-local-name # Check repo status anytime git status
📁
What is .git?The .git folder is the heart of your repo — it stores all history, objects, refs, and config. Never manually edit it. Deleting it removes all version history.

File Lifecycle & Staging
add, status, diff, rm, mv

Git's staging area (the index) is one of its most powerful features. Think of it as a draft board — you decide exactly which changes go into the next commit, giving you surgical control over your history.

bash — Core Staging Commands
git add filename.py # stage a specific file git add src/ # stage a directory git add -A # stage ALL changes git add -p # interactive hunk-by-hunk staging (POWERFUL) # Unstage a file (keeps changes, just removes from staging) git restore --staged filename.py # Discard working directory changes (DESTRUCTIVE — can't undo!) git restore filename.py
⚠️
Gotcha — git add .git add . stages all changes in the current directory and below. Use git add -A from any directory to stage everything in the repo.
bash
git status # full output git status -s # short format M src/app.py # M = modified (staged) M src/utils.py # M = modified (unstaged) ?? temp.txt # ?? = untracked A new_feature.py # A = newly staged D old_file.py # D = deleted
bash
git diff # unstaged changes git diff --staged # staged changes vs last commit git diff main feature # diff between two branches git diff HEAD~3 # changes since 3 commits ago git diff --stat # summary only git diff --word-diff # word-level (great for docs)
.gitignore
# Python __pycache__/ *.pyc venv/ .env # Secrets .env.local config/secrets.json # OS files .DS_Store Thumbs.db # Build output dist/ build/
💡
TipUse gitignore.io to generate .gitignore files for any stack. Already committed a file? Run git rm --cached filename to stop tracking it.

Commits & History
Snapshot your progress like a pro

A commit is a permanent snapshot of your staged changes, along with a message, timestamp, and a pointer to its parent. Think of commits as diary entries — brief, clear, and specific.

bash — Committing
git commit # opens editor git commit -m "feat: add user auth module" # inline message git commit -am "fix: null pointer in login" # stage tracked + commit git commit --amend -m "new message" # fix last commit message # ⚠️ Don't amend commits you've already pushed!

Reading History

bash — git log
git log # full history git log --oneline # compact one-liner git log --oneline --graph --all # visual branch graph ★ git log --author="Shrinivasa" # filter by author git log --since="2 weeks ago" # filter by time git log -p src/models.py # diffs for a specific file git log --grep="auth" # search commit messages # Sample --oneline --graph output: * a3f9b12 (HEAD → main) feat: dashboard v2 * | c9d1e45 (feature/dark-mode) add theme toggle |/ * 1de0c92 fix: resolve login bug

The Anatomy of a Good Commit Message

✅
Conventional CommitsUse: type(scope): short description. Types: feat, fix, docs, style, refactor, test, chore. Keep under 72 chars. Use the body to explain why, not what.
✓ GOOD
commit message
feat(auth): add JWT refresh token Tokens auto-refresh 5min before expiry. Resolves flaky session timeouts in prod. Closes #142
✗ BAD
commit message
fixed stuff wip asdfgh update changes

Fun FactThere are entire websites dedicated to terrible commit messages. The classic: "initial commit" → "actually initial commit" → "ok NOW initial commit". We've all been there.

Branching
Parallel universes for your code

A branch is just a lightweight pointer to a commit — creating one is nearly instant. This lets you work on features, fixes, or experiments in total isolation. The HEAD pointer tells Git where you currently are.

# Your branch tree main ──A──B──C──────────────────M ← (HEAD, main) \ / feature D──E──F──────── ← merged in main ──A──B──C────────────────── ← production \ hotfix H ← (HEAD, hotfix)
bash — Branch Operations
git switch -c feature/user-auth # create + switch (modern) git switch main # switch branch git branch # list local branches git branch -a # list all (incl. remote) git branch -m old-name new-name # rename git branch -d feature/user-auth # delete (if merged) git branch -D feature/user-auth # force delete git branch --merged main # branches merged into main
🏷️
Branch NamingCommon patterns: feature/, fix/, hotfix/, release/, chore/. A good name tells you the purpose at a glance. Avoid test, my-branch, stuff.
🔀
Merging & Conflict Resolution
Bringing code streams together

Merging combines the history of two branches. Git handles most merges automatically. When it can't — because two people edited the same lines differently — you get a merge conflict to resolve manually.

A fast-forward merge happens when the target branch hasn't moved since the feature was created — Git just moves the pointer forward, no merge commit needed.

# Before: main ──A──B \ feature C──D ← HEAD # After merge: pointer just moves forward main ──A──B──C──D ← (HEAD, main, feature)
bash
git switch main git merge feature Fast-forward # Force a merge commit even when FF is possible: git merge --no-ff feature

When both branches have diverged, Git uses a 3-way merge (both tips + common ancestor), creating a merge commit with two parents.

main ──A──B──E──F────M ← (merge commit) \ / feature C──D────

When Git can't auto-merge, it marks the conflict in the file and pauses for you to resolve it.

conflict markers
<<<<<<< HEAD (your branch) return "Hello, World!" ======= return "Hello, Git!" >>>>>>> feature/greeting (incoming)
bash — Resolve
# 1. Edit the file — keep what you want, remove markers # 2. Stage the resolved file git add greeting.py # 3. Complete the merge git commit # OR abort and go back: git merge --abort
💡
VS Code TipVS Code shows inline "Accept Current / Accept Incoming / Accept Both" buttons inside conflicted files — no manual marker deletion needed.

Remote Repositories
Collaborate with the world

A remote is a version of your repository hosted elsewhere (GitHub, GitLab, Bitbucket). The conventional name for the primary remote is origin.

bash — Remote Operations
git remote -v # view remotes git remote add origin https://github.com/you/repo.git git remote add upstream https://github.com/original/repo.git git push -u origin main # first push + set upstream git push # subsequent pushes git push -u origin feature/user-auth # push a new branch git push origin --delete feature/x # delete remote branch git fetch origin # download changes, don't merge git pull # fetch + merge git pull --rebase # fetch + rebase (cleaner)
📥 git fetch
Downloads changes but does not touch your working directory. Safe to run anytime. Inspect before merging: git log origin/main..main
⚡ git pull
Does fetch + merge in one step. Use --rebase to keep history linear and avoid surprise merge commits.
🔐
SSH vs HTTPSSSH keys let you push without a password. Setup: ssh-keygen -t ed25519 -C "you@example.com", then add the public key to GitHub. Switch a repo to SSH: git remote set-url origin git@github.com:you/repo.git

Undoing Things
Git's many ways to say "oops"

One of Git's greatest superpowers: almost nothing is truly lost. But there are different levels of undoing, each with different consequences. Choose wisely.

🧹 git restore — discard uncommitted changes ⌄
bash
git restore file.py # discard working dir changes (⚠️ UNRECOVERABLE) git restore . # discard all working dir changes git restore --staged file.py # unstage (keeps changes)
⚠️
Warninggit restore file.py permanently deletes your working directory changes. No undo!
🔄 git revert — safely undo a commit (creates new commit) ⌄

The safe undo for shared history. Creates a new commit that reverses the target commit. History is preserved — perfect for commits already pushed.

bash
git revert a3f9b12 # revert by hash git revert HEAD # revert last commit git revert HEAD~3..HEAD # revert a range git revert --no-commit HEAD # stage revert, don't auto-commit
⏮️ git reset — move HEAD backward (use with care!) ⌄
ModeStagedWorking DirUse When
--softKept stagedKeptUndo commit, keep changes staged
--mixed (default)UnstagedKeptUndo commit + unstage, keep changes
--hardDiscardedDiscardedNuke everything — go back completely
bash
git reset --soft HEAD~1 # undo last commit, keep staged git reset HEAD~1 # undo last commit, unstage changes git reset --hard HEAD~3 # go back 3 commits, lose everything # Emergency recovery with reflog: git reflog # see EVERY HEAD movement ever git reset --hard HEAD@{4} # jump to 4 actions ago
🛟
Reflog is your safety net!git reflog records every HEAD movement — even after --hard reset. You have ~90 days to recover "lost" commits. It has saved many careers.
🔁
Rebase
Rewrite history for a cleaner story

Rebasing tells Git: "pretend my branch started from here instead of there." It replays your commits on top of another branch, creating a beautifully linear commit history with no merge commits.

# Before rebase: main ──A──B──C──D \ feature X──Y # After: git switch feature && git rebase main main ──A──B──C──D \ feature X'──Y' ← replayed on top of D
bash — Rebase
git switch feature git rebase main # replay feature on top of main # Interactive rebase — the ultimate history editor git rebase -i HEAD~5 # edit last 5 commits # pick — keep as-is # squash — combine with previous # reword — change message only # drop — delete commit entirely # edit — pause to amend git rebase --abort # bail out git rebase --continue # continue after resolving conflict
⚠️
The Golden Rule of RebaseNever rebase commits already pushed to a shared branch. Rebase rewrites commit hashes — if others have based work on those commits, you'll create a painful mess. Rebase only local, private branches.
🔀 Merge
Preserves full history. Creates a merge commit. Shows exactly when branches were integrated. Best for public branches.
🔁 Rebase
Linear history, no merge commits. Cleaner git log. Best for local feature branches before a PR.
🎲
Fun FactThe merge vs. rebase debate is one of the great holy wars of software development — right up there with tabs vs. spaces. Both are valid. Pick a team convention and stick with it.

Git Stash
Temporarily shelve your work

Mid-feature when an urgent bug report lands? You can't commit half-done work, but you need a clean slate. Stash is your desk drawer — save messy WIP changes, fix the bug, then come back.

bash — Stash Operations
git stash # save WIP git stash -m "wip: user auth refactor" # with a label git stash -u # include untracked files git stash list stash@{0}: On main: wip: user auth refactor stash@{1}: WIP on feature: some older work git stash pop # apply + remove most recent git stash apply stash@{1} # apply without removing git stash drop stash@{0} # delete a specific stash git stash clear # clear all stashes git stash branch feature/from-stash # create branch from stash
💡
pop vs applyUse pop when done with the stash. Use apply when you want to apply the same stash to multiple branches. If pop creates a conflict, the stash stays in the list automatically.

Git Hooks
Automate your workflow

Hooks are scripts that run automatically at specific Git lifecycle points. They live in .git/hooks/ and can be written in any scripting language. Perfect for enforcing quality gates.

HookWhen it firesCommon Use
pre-commitBefore commit is createdRun linters, tests, format code
commit-msgAfter message is writtenValidate conventional commits format
prepare-commit-msgBefore editor opensAuto-populate message with ticket ID
post-commitAfter commit completesNotifications, CI triggers
pre-pushBefore pushing to remoteRun full test suite
post-mergeAfter merge completesAuto-install dependencies
shell — .git/hooks/pre-commit
#!/bin/sh echo "🔍 Running flake8..." if flake8 .; then echo "✅ All good!" exit 0 else echo "❌ Linting failed. Fix errors before committing." exit 1 fi
bash — Enable & Skip
chmod +x .git/hooks/pre-commit # must be executable git commit --no-verify -m "wip" # skip hooks (use sparingly)
📦
Husky for Teams.git/hooks/ isn't committed to the repo, so teammates won't get your hooks. Use Husky (Node.js) or pre-commit (Python) to share hooks as part of the project.

Pro Tips & Power Features
Level up your Git game
🔍 git bisect — binary search for a bug⌄

git bisect does a binary search through your commit history to find the exact commit that introduced a bug. Incredibly powerful for large repos.

bash
git bisect start git bisect bad # current = broken git bisect good v1.2.0 # this version was fine # Git checks out midpoint. Test it, then: git bisect good # or: git bisect bad # Repeat until Git finds the culprit git bisect reset # Or fully automated: git bisect run ./run_tests.sh
🍒 git cherry-pick — grab one specific commit⌄

Selectively apply any commit onto your current branch — without merging the whole branch.

bash
git cherry-pick a3f9b12 # single commit git cherry-pick a3f9b12..e7c4d21 # range git cherry-pick -n a3f9b12 # stage only, don't commit
🏷️ git tag — mark release versions⌄
bash
git tag -a v1.0.0 -m "Release 1.0.0" # annotated tag git tag -l "v1.*" # list tags git push origin v1.0.0 # push a tag git push origin --tags # push ALL tags
⚡ Aliases — your personal Git shortcuts⌄
bash
git config --global alias.st status git config --global alias.lg "log --oneline --graph --all --decorate" git config --global alias.undo "reset HEAD~1 --soft" git config --global alias.unstage "restore --staged" # Usage: git st git lg git undo
🔎 git blame & log — find anything in history⌄
bash
git blame src/auth.py # who last changed each line git blame -L 40,60 src/auth.py # lines 40–60 only git log -S "password_hash" --oneline # when was this string added/removed? git log -L :get_user:src/users.py # commits that touched this function
