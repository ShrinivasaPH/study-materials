What is MCP?

Model Context Protocol (MCP) is an open protocol introduced by Anthropic in November 2024 that standardizes how large language models communicate with external tools, data sources, and services.

💡
The core idea: Before MCP, every AI integration required custom, one-off glue code. MCP defines a single, universal "language" that any AI model (client) can use to talk to any external system (server) — like USB for AI tools.

Think of MCP as the HTTP of AI tool use. Just as HTTP standardized how browsers and web servers communicate, MCP standardizes how AI models and their tool providers communicate.

🔌

Standardization

One protocol replaces countless custom integrations. Build once, connect everywhere — any MCP client works with any MCP server.

🔒

Security-first

Explicit capability negotiation, consent-based data access, and clear separation between the AI model and external systems.

⚡

Stateful connections

Long-lived sessions allow complex, multi-turn agentic workflows — not just single request-response tool calls.

🌍

Open standard

Open-source specification licensed under MIT. Supported by Anthropic, OpenAI, Google DeepMind, and hundreds of developers.

The Problem MCP Solves

❌ Before MCP
  • → N tools × M models = N×M integrations
  • → Each integration has its own API format
  • → Context lost between tool calls
  • → No standard for auth, errors, streaming
  • → Difficult to compose multiple tools
✓ With MCP
  • → N + M integrations (one per side)
  • → Universal message format (JSON-RPC 2.0)
  • → Stateful sessions preserve context
  • → Built-in capability negotiation
  • → Composable tool ecosystems

Core Primitives

MCP defines five core primitives — the fundamental building blocks for every server. Understanding these is essential to designing effective MCP integrations.

🔧
Tools
Model-controlled actions

Functions the AI can invoke to perform actions or retrieve computed information. The model decides when and how to call tools based on context.

Examples: run_query(), send_email(), create_file(), search_web()

server-exposed model-invoked
📚
Resources
Data & content exposure

Structured data (files, database rows, API responses) that servers expose for the AI to read. Application-controlled — the host decides what to include in context.

Identified by URIs: file:///path/to/doc, db://table/row

server-exposed app-controlled
💬
Prompts
Reusable prompt templates

Pre-built prompt templates and workflows that can accept dynamic arguments. User-controlled — surfaced in the UI as slash commands or menu options.

Examples: /summarize, /review-pr, /explain-code

server-defined user-invoked
🧠
Sampling
Server-initiated LLM calls

Allows servers to request LLM completions through the client — enabling agentic server behaviors. The host maintains control and user approval.

Server → Client → LLM → Client → Server pipeline

client-facilitated host-approved
🌲
Roots
Filesystem boundaries

URI boundaries that define the scope of server access. The host exposes a list of "roots" (directories/URIs) the server is allowed to operate within.

Enables servers to understand project scope without arbitrary filesystem access.

client-exposed server-informs

The Protocol

MCP uses JSON-RPC 2.0 as its message format — a lightweight, stateless remote procedure call protocol encoded in JSON.

JSON-RPC 2.0 Message Types

// Request (expects a Response)
"jsonrpc": "2.0"Always exactly "2.0"
"id": string | numberUnique request ID for matching responses
"method": stringNamespaced method: "tools/call", "resources/read"
"params": object?Optional method-specific parameters
// Response
"jsonrpc": "2.0"
"id": string | numberMatches the request id
"result": anyOn success — method-specific result data
OR "error": { code, message, data? }On failure — error details
// Notification (no Response expected)
"jsonrpc": "2.0"
"method": stringe.g. "notifications/tools/list_changed"
"params": object?No "id" field — this is what makes it a notification

Standard Error Codes

Code Name When to Use
-32700Parse errorInvalid JSON received
-32600Invalid RequestJSON-RPC structure is invalid
-32601Method not foundRequested method doesn't exist
-32602Invalid paramsInvalid method parameters
-32603Internal errorInternal server error

Transports

MCP defines two official transport mechanisms for carrying JSON-RPC messages. The choice depends on where your server runs and how it's deployed.

⌨️
stdio
Standard Input/Output
DeploymentLocal process
ProtocolNewline-delimited JSON
BidirectionalYes (stdin + stdout)
LatencyVery low
Auth neededNo (process isolation)
Best forCLI tools, local agents
stderrServer logs/debug output
🌐
HTTP + SSE
Server-Sent Events
DeploymentRemote server
ProtocolHTTP POST + EventSource
BidirectionalPOST→, SSE←
LatencyNetwork dependent
Auth neededYes (tokens/OAuth)
Best forCloud services, SaaS
Endpoint/sse + /messages
📡
SSE mechanics: Client subscribes to GET /sse (persistent EventSource), server sends an endpoint URL. Client then POSTs messages to that URL; server sends responses via the SSE stream. This enables bidirectional communication over standard HTTP.

stdio Transport Example

BASH How stdio works 
# Host spawns server as a subprocess
# Each JSON-RPC message is terminated by a newline character

# Client → Server (via stdin)
{"jsonrpc":"2.0","id":1,"method":"initialize","params":{...}}

# Server → Client (via stdout)
{"jsonrpc":"2.0","id":1,"result":{"capabilities":{...}}}

# Server logs go to stderr (never to stdout!)
[2024-11-05 10:23:01] Server started, listening on stdio

TypeScriptserver.ts
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

// Create the server
const server = new McpServer({
  name: "my-server",
  version: "1.0.0",
});

// Register a tool
server.tool(
  "get_word_count",
  "Count words in a given text string",
  {
    text: z.string().describe("The text to count words in"),
  },
  async ({ text }) => {
    const count = text.trim().split(/\s+/).filter(Boolean).length;
    return {
      content: [{ type: "text", text: `Word count: ${count}` }],
    };
  }
);

// Register a resource
server.resource(
  "config",
  "config://app",
  async (uri) => ({
    contents: [{
      uri: uri.href,
      mimeType: "application/json",
      text: JSON.stringify({ version: "1.0", debug: false }),
    }],
  })
);

// Register a prompt template
server.prompt(
  "code-review",
  "Generate a code review for the provided code",
  { code: z.string().describe("Code to review") },
  ({ code }) => ({
    messages: [{
      role: "user",
      content: { type: "text", text: `Please review this code:\n\n${code}` }
    }],
  })
);

// Start the server on stdio
const transport = new StdioServerTransport();
await server.connect(transport);
console.error("Server running on stdio"); // Use stderr for logs!

Pythonserver.py
from mcp.server.fastmcp import FastMCP
from mcp.types import TextContent

# FastMCP: high-level, Pythonic API
mcp = FastMCP("my-python-server")

@mcp.tool()
def get_word_count(text: str) -> str:
    """Count words in a given text string."""
    count = len(text.split())
    return f"Word count: {count}"

@mcp.resource("config://app")
def get_config() -> str:
    """Application configuration"""
    import json
    return json.dumps({ "version": "1.0", "debug": False })

@mcp.prompt()
def code_review(code: str) -> str:
    """Generate a code review prompt for the provided code."""
    return f"Please review this code:\n\n{code}"

# For advanced use, FastMCP also supports async
@mcp.tool()
async def fetch_url(url: str) -> str:
    """Fetch content from a URL"""
    import httpx
    async with httpx.AsyncClient() as client:
        response = await client.get(url)
        return response.text[:2000]  # limit output

# Run via stdio (default)
if __name__ == "__main__":
    mcp.run()

Security & Trust

Security is a first-class concern in MCP. The protocol defines clear trust boundaries and mandates explicit human oversight for sensitive operations.

👤
User
Highest trust. Installs servers, provides credentials, approves sensitive actions.
🖥️
Host
Trusted by user. Enforces policy, presents consent dialogs, controls what LLM sees.
🔌
Server
Least trusted. Isolated. Cannot access other servers, cannot force actions without consent.

✓ Capability negotiation

Servers declare what they can do upfront. Clients only grant the permissions they agree to. No hidden capabilities.

✓ Explicit consent

Sampling requests (server → LLM) require host approval before any data is sent. Users can inspect and reject.

✓ Process isolation

stdio servers run as separate processes. OS-level isolation prevents servers from accessing host memory.

⚠ Prompt injection

Malicious servers may embed instructions in tool results. Hosts must sanitize content before passing to LLMs.

⚠ Tool confirmation

Destructive tools (delete, send, execute) should require user confirmation. Never auto-approve irreversible actions.

⚠ Secret management

Never hardcode credentials in server code. Use environment variables. Claude config supports env fields for this.

✗ Confused deputy

A compromised server could try to invoke its own capabilities via the LLM. Hosts must enforce per-tool rate limits.

✗ Data exfiltration

Servers could encode sensitive data in outbound requests. Constrain what servers can access with Roots primitive.

✓ Roots scoping

Use the Roots primitive to limit server filesystem access to only the directories relevant to the task.
