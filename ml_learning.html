What is Machine Learning?

Programs that get better at tasks through experience — without you having to spell out every rule by hand.

Arthur Samuel coined the term back in 1959, and the core insight hasn't changed: instead of painstakingly writing if-this-then-that rules, you hand a computer a pile of examples, and it figures out the patterns itself. It's the difference between teaching someone to recognise a cat by listing every feline feature, versus just showing them a thousand pictures of cats.

The Core Idea

At its heart, ML is about finding a function f such that f(X) ≈ y — a mapping from input features X to outputs y, discovered from data rather than written by hand. The art lies in choosing how expressive that function should be, and how to search for it efficiently.

The ML Workflow

1
Collect & explore data
Garbage in, garbage out — quality data is everything. Gather labelled examples, then spend real time understanding them: check distributions, hunt for missing values, spot correlations, and look for surprises. This phase (EDA) usually takes longer than people expect, and saves enormous pain later.
2
Feature engineering
Raw data rarely speaks directly to models. Transform it into meaningful numerical representations — normalise scales, encode categories, create interaction features, extract useful signals (like "day of week" from a timestamp). Great features can make a simple model outperform a complex one.
3
Choose & train a model
Pick an algorithm that suits your data size, feature types, and interpretability needs. Define a loss function that captures what "wrong" means for your problem, then let an optimiser — usually gradient descent — tune the model's parameters on your training data.
4
Evaluate & tune
A model that memorises training data is useless. Measure how well it generalises on a held-out validation set it has never seen. Tune hyperparameters, watch for overfitting or underfitting, and use cross-validation to make your estimates robust.
5
Deploy & monitor
Shipping to production is a beginning, not an end. Real-world data drifts — user behaviour shifts, sensor calibrations change, distributions evolve. Set up monitoring, track key metrics over time, and plan to retrain as the world changes around your model.

Underfitting

Your model is too simple to capture the real patterns — think trying to fit a straight line through data that clearly curves. It performs poorly even on training data, let alone on new examples. This is called high bias.

Fix: add more features, switch to a more expressive model, or dial down regularisation.

Overfitting

Your model has memorised the training data — noise and all. It scores brilliantly on training examples but falls apart on anything new. This is called high variance, and it's the more common pitfall.

Fix: gather more data, apply regularisation (L1/L2/Dropout), simplify your model, or use cross-validation to catch it early.

Supervised Learning

Regression

When your output is a number on a continuous scale — house prices, temperatures, tomorrow's sales — regression is your starting point.

Linear Regression

The simplest model you can build: assume the relationship between your inputs and output can be described by a straight line (or a hyperplane when you have multiple features). You fit that line by finding the weights that minimise the total squared distance between your predictions and the true values. Surprisingly often, this simple idea works remarkably well.

Model hypothesis
ŷ = θ₀ + θ₁x₁ + θ₂x₂ + … + θₙxₙ = θᵀx
ŷ predicted value θ₀ bias / intercept θᵢ weight for feature i xᵢ feature i
Mean Squared Error — Loss function
J(θ) = (1/2m) Σᵢ (ŷ⁽ⁱ⁾ − y⁽ⁱ⁾)²
m number of examples ŷ⁽ⁱ⁾ prediction for example i y⁽ⁱ⁾ true label for example i
Gradient Descent Update Rule
θⱼ ← θⱼ − α · (∂J/∂θⱼ) = θⱼ − (α/m) Σᵢ (ŷ⁽ⁱ⁾ − y⁽ⁱ⁾) · xⱼ⁽ⁱ⁾
α learning rate ∂J/∂θⱼ gradient (slope of loss surface)

Regularisation

When a model has many parameters, it can start fitting noise in the training data rather than genuine patterns. Regularisation adds a penalty to the loss function that discourages large weights — essentially telling the model "keep it simple unless the data really demands otherwise."

Ridge (L2)

J + λ Σθⱼ²

Penalises the square of each weight, which gently nudges all weights towards zero but never quite reaches it. Every feature gets to contribute a little. A good default when you believe all your features are genuinely relevant.

Lasso (L1)

J + λ Σ|θⱼ|

Penalises the absolute value of each weight, which has a sharper corner at zero — so unimportant weights get pushed all the way to exactly zero. You get automatic feature selection for free. Excellent when you suspect most of your features are irrelevant.

Python Code

import numpy as np
from sklearn.linear_model import LinearRegression, Ridge, Lasso
from sklearn.preprocessing import PolynomialFeatures
from sklearn.pipeline import Pipeline

# --- Simple linear regression from scratch ---
class LinearReg:
    def __init__(self, lr=0.01, epochs=1000):
        self.lr, self.epochs = lr, epochs

    def fit(self, X, y):
        m, n = X.shape
        self.theta = np.zeros(n + 1)          # weights + bias
        Xb = np.c_[np.ones(m), X]              # add bias column
        for _ in range(self.epochs):
            y_hat = Xb @ self.theta
            grad = (Xb.T @ (y_hat - y)) / m   # ∂J/∂θ
            self.theta -= self.lr * grad        # gradient descent step
        return self

    def predict(self, X):
        return np.c_[np.ones(len(X)), X] @ self.theta

# --- Sklearn pipeline with polynomial features + Ridge ---
model = Pipeline([
    ('poly', PolynomialFeatures(degree=2)),
    ('ridge', Ridge(alpha=1.0))
])
model.fit(X_train, y_train)

Evaluation Metrics

Regression metrics all ask the same fundamental question — how far off are your predictions from reality? — but they answer it in subtly different ways. Your choice of metric should reflect what "wrong" actually costs in your problem. An error of 10 units might be catastrophic in one context and totally acceptable in another.

Core regression metrics
MAE = (1/m) Σ|ŷ⁽ⁱ⁾ − y⁽ⁱ⁾|
MSE = (1/m) Σ(ŷ⁽ⁱ⁾ − y⁽ⁱ⁾)²
RMSE = √MSE
MAPE = (1/m) Σ|ŷ−y|/|y| × 100%
R² = 1 − SS_res / SS_tot
Adj R² = 1 − (1−R²)(m−1)/(m−n−1)
MAE

Mean Absolute Error

Takes every prediction error, ignores the sign, and averages them. Because it doesn't square errors, a single catastrophic mistake doesn't dominate the score — each error contributes proportionally. The result is in the same units as your target, making it easy to explain to non-technical stakeholders ("our predictions are off by about $X on average").

Use when: outliers shouldn't be over-penalised
RMSE

Root Mean Squared Error

Squares each error before averaging, then takes the square root to return to the original units. Crucially, squaring means large errors hurt much more than small ones — a prediction that's off by 10 contributes 100× more than one off by 1. If big mistakes are unacceptable in your use case, RMSE will penalise your model accordingly.

Use when: large errors are especially costly
R²

Coefficient of Determination

Answers the question: "how much of the variance in the target did my model actually explain?" It's unitless and scale-free, so you can meaningfully compare models across completely different problems. R²=1 is perfect prediction; R²=0 means your model is no better than just predicting the mean every time; negative means it's actively worse — a red flag.

Use when: comparing models on different scales
Adj R²

Adjusted R²

A known flaw of R²: it can only go up when you add features, even completely random ones. Adjusted R² corrects for this by penalising model complexity. It only improves when a new feature genuinely earns its keep — contributing more signal than a random variable would. Essential when comparing models with different numbers of features.

Use when: comparing models with different feature counts
MAPE

Mean Absolute % Error

Expresses error as a percentage of the true value — so being off by 5 on a value of 10 is far worse than being off by 5 on a value of 10,000. Highly intuitive for business reporting ("our forecast was within 8% on average"). Watch out though: it breaks down completely when true values are zero or near-zero, and it systematically penalises under-forecasts more harshly than over-forecasts.

Use when: relative error matters more than absolute
Huber

Huber Loss

A clever hybrid: behaves like MSE for small errors (smooth and differentiable, good for training), but switches to MAE-like behaviour for large errors (less sensitive to outliers). The transition point δ is a hyperparameter you set. It's the go-to choice when your data has occasional extreme values that you don't want driving your training off a cliff.

L = ½r² if |r|≤δ, else δ(|r|−½δ)

Quick decision guide

Outliers in your data? Reach for MAE or Huber — they won't let a few bad rows dominate. Large errors are especially costly? Use RMSE to make the model sweat the big mistakes. Need to explain results in % terms? MAPE translates naturally for non-technical audiences. Comparing models trained on different datasets? R² gives a fair, scale-free comparison. One final tip: always report two or three metrics together — no single number ever tells the whole story.

Supervised Learning

Classification

When the answer is a category — spam or not, which digit, which disease — you need a classifier rather than a regressor.

Logistic Regression

Don't let the name fool you — logistic regression is a classifier, not a regressor. The trick: take the usual linear combination of features, then squash it through a sigmoid function that maps any real number to a value strictly between 0 and 1. That output is a probability — "how confident am I that this example belongs to class 1?" You then apply a threshold (usually 0.5) to get a hard decision.

Sigmoid activation
σ(z) = 1 / (1 + e⁻ᶻ)    where    z = θᵀx
σ(z) ∈ (0, 1) — probability of class 1 e Euler's number ≈ 2.718
Binary Cross-Entropy Loss
J(θ) = −(1/m) Σᵢ [ y⁽ⁱ⁾ log(ŷ⁽ⁱ⁾) + (1−y⁽ⁱ⁾) log(1−ŷ⁽ⁱ⁾) ]
log(ŷ) penalises confident wrong predictions harshly

Evaluation Metrics — deep dive

Here's a trap almost everyone falls into: reporting accuracy and calling it done. On an imbalanced dataset (say, 99% of cases are negative), a model that blindly predicts "negative" every single time scores 99% accuracy — yet it's completely useless. These metrics exist to expose exactly that kind of failure, giving you a far richer picture of where your model actually succeeds and where it breaks down.

Accuracy

Overall Correctness

The fraction of all predictions that were correct. Simple and intuitive, but dangerously misleading on imbalanced datasets. Always check what the "naive" baseline accuracy would be before trusting this number.

(TP+TN) / (TP+TN+FP+FN)
Precision

"When you say yes, are you right?"

Of everything your model flagged as positive, what fraction actually was? High precision means few false alarms. This is the metric to optimise when the cost of a false positive is high — for example, in a spam filter, incorrectly blocking a legitimate email is a real problem.

TP / (TP + FP)
Recall

"Do you catch everything positive?"

Of all the actual positives in your data, how many did you catch? High recall means few missed detections. Prioritise this when missing a positive is catastrophic — cancer screening is the classic example: you'd rather flag 10 healthy patients for further tests than miss one real case.

TP / (TP + FN)
F1 Score

Harmonic Mean of P & R

Since precision and recall trade off against each other, you often need a single number that captures both. The F1 score is their harmonic mean — and crucially, the harmonic mean punishes extreme imbalances: a model with perfect precision but zero recall still gets an F1 of 0. A solid default when classes are imbalanced.

2 · (P · R) / (P + R)
AUC-ROC

Area Under ROC Curve

Instead of evaluating at one threshold, AUC-ROC sweeps across all possible thresholds and asks: how well does my model rank positives above negatives? AUC of 1.0 means perfect separation; 0.5 is pure random chance. Because it's threshold-independent, it's ideal for comparing two models head-to-head without committing to a deployment threshold.

∫₀¹ TPR(FPR) d(FPR)
AUC-PR

Precision-Recall AUC

On heavily imbalanced datasets — fraud detection, rare disease diagnosis — the ROC curve can look deceptively optimistic because there are so many true negatives. The Precision-Recall curve ignores true negatives entirely, focusing only on how well you identify the rare positive class. If positives are rare, this is the curve you should be watching.

Preferred for: fraud detection, rare disease
Log Loss

Cross-Entropy Loss

Unlike accuracy (which only cares whether you're right or wrong), log loss cares about how confident you were. If you predicted 99% probability and got it wrong, you'll be punished far more than if you'd predicted 51%. This forces your model to be well-calibrated — not just correct, but appropriately certain.

−(1/m)Σ[y log ŷ + (1−y)log(1−ŷ)]
MCC

Matthews Correlation Coefficient

Widely considered the most honest single metric for binary classification, because it incorporates all four cells of the confusion matrix equally. It's essentially a correlation coefficient between your predictions and the ground truth — so 0 means no better than random, +1 means perfect, and −1 means perfectly inverted. It remains meaningful even when classes are wildly imbalanced.

(TP·TN − FP·FN) / √((TP+FP)(TP+FN)(TN+FP)(TN+FN))

Precision vs Recall trade-off

You control this trade-off with your decision threshold (default 0.5, but rarely optimal). Lower the threshold → your model flags more things as positive, catching more true cases but also generating more false alarms (↑ Recall, ↓ Precision). Raise it → fewer flags, fewer false alarms, but you'll miss more real cases (↑ Precision, ↓ Recall). The key insight: always tune your threshold after training, based on what your use case actually requires. Don't assume 0.5 is right.

Other Algorithms

Decision Tree

Learns a series of yes/no questions about features, splitting the data recursively until it can make confident predictions. The result is a flowchart you can literally draw and explain to anyone — a huge advantage when interpretability matters. The downside: a single tree tends to overfit and is sensitive to small changes in data.

Split criterion: max Information Gain or min Gini Impurity

SVM

Finds the decision boundary that maximises the gap (margin) between the two classes — specifically, it's determined by the hardest examples near the boundary, called support vectors. For non-linearly separable data, the kernel trick implicitly maps your features into a higher-dimensional space where a linear boundary works.

min ½‖w‖² s.t. yᵢ(wᵀxᵢ+b) ≥ 1

k-Nearest Neighbours

The simplest possible approach: to classify a new point, find the k training examples closest to it and take a majority vote. There's no training phase — you just store all your data and look up neighbours at prediction time. Intuitive and surprisingly effective for small datasets, but slow and memory-hungry at scale.

distance(a,b) = √Σ(aᵢ−bᵢ)²

Naive Bayes

Uses Bayes' theorem to compute the probability of each class given the features — with one simplifying assumption: that features are conditionally independent of each other given the class. The "naive" assumption is rarely true, but the algorithm works remarkably well in practice, particularly for text classification like spam filtering.

P(y|X) ∝ P(y) Πᵢ P(xᵢ|y)

Unsupervised Learning

Clustering

Discover the natural structure hiding in your data — without any labels to guide you. The algorithm finds it on its own.

k-Means Clustering

The workhorse of clustering: divide your data into k groups by repeatedly asking "which centroid is each point closest to?" and "where should each centroid be, given its assigned points?". It's fast, intuitive, and scales well. The catch: you have to decide in advance how many clusters you want — which isn't always obvious. That's what the Elbow Method below is for.

k-Means Objective (Inertia)
J = Σₖ Σᵢ∈Cₖ ‖xᵢ − μₖ‖²
Cₖ set of points in cluster k μₖ centroid (mean) of cluster k ‖·‖² squared Euclidean distance
1
Initialise
Randomly pick k data points as your starting centroids. (The smarter k-means++ variant spreads them apart, reducing the chance of a bad initialisation — highly recommended in practice.)
2
Assign
For every data point, calculate its distance to each centroid and assign it to the nearest one. This partitions your data into k clusters.
3
Update
Recompute each centroid as the mean (average position) of all points currently assigned to it. The centroids shift to the "centre of gravity" of their cluster.
4
Repeat
Alternate between Assign and Update until the centroids stop moving — or move less than some tiny threshold. The algorithm is guaranteed to converge in a finite number of steps, though it may find a local minimum rather than the global best.

Choosing k — the Elbow Method

Run k-means with k=1, 2, 3… and record the total inertia (within-cluster sum of squares) each time. More clusters always reduces inertia, but at some point you're just splitting real groups apart unnecessarily. The "elbow" is where adding another cluster stops giving you a meaningful improvement — that's typically the right k. It's not always perfectly sharp, so use it as a guide alongside domain knowledge.

Other Clustering Algorithms

DBSCAN

Rather than finding clusters by distance to centroids, DBSCAN looks for dense regions of points. It can discover clusters of any shape — rings, crescents, blobs — and anything that doesn't belong to a dense region gets labelled as noise/outlier. No need to specify k in advance, but you do need to tune the density radius ε and minimum points parameters.

Works with noiseArbitrary shapes

Hierarchical

Builds a tree-like structure (a dendrogram) showing how clusters merge or split at different scales. You don't need to commit to a number of clusters upfront — just cut the tree at whatever height suits your analysis. Excellent for exploring data structure, though it can be slow on very large datasets.

No k neededDendrogram

Gaussian Mixture

Instead of hard assignments (each point belongs to exactly one cluster), GMM gives every point a probability of belonging to each cluster. Clusters are modelled as Gaussian distributions, and the model is fit via the EM algorithm. Particularly useful when clusters overlap and hard boundaries feel artificial.

ProbabilisticSoft clustering
from sklearn.cluster import KMeans, DBSCAN
from sklearn.metrics import silhouette_score
import numpy as np

# k-Means with elbow method
inertias = []
K_range = range(1, 11)
for k in K_range:
    km = KMeans(n_clusters=k, init='k-means++', n_init=10, random_state=42)
    km.fit(X)
    inertias.append(km.inertia_)

# Best k via silhouette score (higher = better separated clusters)
best_k, best_s = 2, -1
for k in range(2, 11):
    labels = KMeans(n_clusters=k, random_state=42).fit_predict(X)
    s = silhouette_score(X, labels)
    if s > best_s: best_k, best_s = k, s

print(f"Best k = {best_k}, silhouette = {best_s:.3f}")

# DBSCAN — good for irregular shapes
db = DBSCAN(eps=0.5, min_samples=5).fit(X)
labels = db.labels_                    # -1 = noise
n_clusters = len(set(labels)) - (1 if -1 in labels else 0)

Evaluation Metrics

Evaluating clustering is genuinely tricky — unlike supervised learning, there's no ground truth to compare against in real-world use. Instead, metrics fall into two camps: internal metrics, which assess cluster quality using only the data itself, and external metrics, which require pre-labelled data and are mostly used for benchmarking on known datasets.

Internal metrics (no labels needed)

Silhouette Score

For each point, asks: "how similar am I to my own cluster compared to the nearest other cluster?" A score near +1 means you're well inside your cluster; near 0 means you're on a boundary; negative means you're probably in the wrong cluster. The average silhouette score across all points is a robust measure of how well your clusters are defined. It's the internal metric to reach for first.

s = (b − a) / max(a, b)
a = mean intra-cluster distance   b = mean nearest-cluster distance
Inertia (WCSS)

The total sum of squared distances from each point to its assigned centroid — a direct measure of how tight and compact your clusters are. Lower is better, but inertia always decreases as you add more clusters (even trivially), so you shouldn't compare raw inertia values across different k. Use it for the elbow plot, not as a standalone quality score.

Σₖ Σᵢ∈Cₖ ‖xᵢ − μₖ‖²
Davies-Bouldin Index

Computes a ratio of within-cluster scatter to between-cluster distance for every pair of clusters, then averages the worst cases. A lower score means clusters are compact and well-separated. Zero would be perfect. It's particularly useful for diagnosing which specific clusters are problematic — the ones with high DB contributions are the ones bleeding into each other.

DB = (1/k) Σᵢ maxⱼ≠ᵢ (σᵢ+σⱼ)/d(μᵢ,μⱼ)

External metrics (need ground truth)

Adjusted Rand Index

Looks at every pair of points and asks: did the clustering agree with the ground truth labels about whether they should be together or apart? Crucially, it's adjusted for chance — so randomly assigning clusters scores near 0, not some inflated value. Perfect agreement with the true labels gives +1.

ARI ∈ [−1, 1], chance-corrected
Normalised Mutual Info

An information-theoretic perspective: how much does knowing a point's cluster assignment tell you about its true label? Scaled to [0, 1] so it's easy to interpret, and it doesn't matter if you've permuted the cluster labels (cluster 1 vs. cluster 2 is arbitrary, after all). Useful when you want a single number to compare different clustering configurations against a known ground truth.

NMI = I(Y;C) / √[H(Y)·H(C)]
Homogeneity & Completeness

Homogeneity asks: does each cluster contain only members of a single class? Completeness asks: are all members of a given class placed into the same cluster? You'd want both — but they trade off like precision and recall do. V-measure takes their harmonic mean, giving you one balanced score that's zero if either property is completely absent.

V = 2 · hom · comp / (hom + comp)

When to use which metric

No ground truth available? Use Silhouette to find the best k, and Inertia for the elbow plot. Benchmarking against known labels? Use ARI or NMI. Diagnosing cluster quality? Davies-Bouldin pinpoints which clusters overlap.

Meta-Learning

Ensemble Methods

Why bet on one model when you can combine many? Ensemble methods are some of the most reliable and battle-tested algorithms in all of machine learning.

A single decision tree is brittle — small changes to the data can result in a completely different tree (high variance). A single linear model is too rigid to capture complex patterns (high bias). Ensemble methods escape both traps by training many models and combining their outputs. The magic: individual errors tend to cancel out when they're uncorrelated. A wrong guess from one model gets outvoted by correct guesses from the others.

Bias–Variance Decomposition

Expected Error = Bias² + Variance + Irreducible Noise

This decomposition explains why ensembles work so well. Bagging trains models on different data samples, so their errors are less correlated — averaging them out reduces variance without touching bias. Boosting takes the opposite approach: it trains models sequentially to correct each other's mistakes, reducing bias. Stacking is flexible enough to reduce both, by learning how to optimally combine diverse base models.

1. Bagging — Bootstrap Aggregating

Create B versions of your training dataset by sampling with replacement (bootstrapping), train an independent model on each, then average their predictions. Because each model sees slightly different data, they make different mistakes — and averaging washes those mistakes out. The models are trained in parallel, so bagging is fast even with large ensembles.

Bagging prediction
ŷ_bag = (1/B) Σᵦ hᵦ(x)   [regression]    or    majority vote   [classification]
B number of base models hᵦ b-th base learner trained on bootstrap sample b

Random Forest

Random Forest takes bagging one step further: not only does each tree see different data, but at each split it can only consider a random subset of features. This decorrelates the trees further — preventing them all from making the same mistake when one feature is particularly dominant. The result is one of the most reliable, low-effort algorithms in existence. It handles mixed data types, is robust to outliers, and gives you feature importances for free.

features per split ≈ √n (classification) or n/3 (regression)

2. Boosting — Sequential Error Correction

Where bagging trains models independently in parallel, boosting trains them one after another — each new model specifically targeting the cases the previous models got wrong. Start with a simple model, find what it struggles with, train the next model to focus on those hard examples, and keep going. The result is a powerful model built from many simple "stumps" that collectively cover each other's blind spots.

AdaBoost

After each training round, misclassified examples get their weights increased — so the next learner is forced to pay more attention to the hard cases. Final predictions are a weighted vote of all learners, where more accurate learners get higher say. Elegant and interpretable, though sensitive to noisy data and outliers.

ŷ = sign(Σᵦ αᵦ hᵦ(x))

Gradient Boosting

Each new tree is fit to the residuals of the current ensemble — specifically, the negative gradient of the loss function with respect to the current predictions. This generalises boosting to any differentiable loss, making it incredibly flexible. Slower to train than bagging (can't parallelise), but typically achieves better accuracy on tabular data.

Fₜ(x) = Fₜ₋₁(x) + α · hₜ(x)

XGBoost / LightGBM / CatBoost

These libraries took gradient boosting from a textbook algorithm to a competition-winning machine. Beyond the core boosting logic, they add L1/L2 regularisation on tree weights, second-order gradient information for faster convergence, built-in handling of missing values, GPU acceleration, and clever memory-efficient tree-growing strategies. XGBoost became legendary after dominating Kaggle competitions; LightGBM is the go-to when you have very large datasets and need speed; CatBoost handles categorical features natively without preprocessing.

3. Stacking

Rather than averaging or voting, stacking trains a second-level meta-learner whose job is to figure out how to best combine the base models. The key trick: base models make predictions on held-out folds (out-of-fold predictions), and the meta-learner is trained on those — not the base models' training data. This prevents leakage and teaches the meta-learner to compensate for each base model's specific weaknesses.

Stacking — two-level structure
Level 0: h₁(x), h₂(x), …, hₙ(x) — diverse base models

Sequential Data

Time Series

When your data has a time dimension, order matters — and the past contains real signal about the future. But that signal is rarely simple.

Components of a Time Series

Most real-world time series are a mix of several underlying patterns layered on top of each other. The first step in any time series analysis is to decompose the signal and understand what you're working with.

📈

Trend

The long-term direction the series is heading — steadily rising, falling, or roughly flat. Think of global average temperatures climbing over decades, or a company's user base growing year after year. Trend is the "big picture" movement that persists over a long time horizon.

🔄

Seasonality

Repeating patterns that follow a fixed, known calendar period — daily, weekly, monthly, or yearly. Retail sales reliably surge every December. Website traffic drops every weekend. Electricity demand peaks every summer evening. If you can set your watch to it, it's seasonality.

〰️

Cyclicality

Longer-term, irregular fluctuations that don't have a fixed period. Economic boom-and-bust cycles are the classic example — they recur, but not on a predictable schedule. Unlike seasonality, cycles can last for years and are much harder to forecast.

⚡

Residual / Noise

Everything left over after you've accounted for trend, seasonality, and cycles. In a well-specified model, this should look like random white noise — no remaining pattern. If it doesn't, your model is missing something systematic.

ARIMA — The Classical Model

ARIMA(p, d, q) is the foundational statistical model for time series forecasting. It weaves together three ideas: autoregression (the current value depends on its own recent past), integration (differencing to remove trend and make the series stationary), and moving average (the current value also depends on past forecast errors). Getting the stationarity step right is crucial — most ARIMA failures can be traced back to modelling a non-stationary series directly.

ARIMA(p, d, q) model
Φ(B)(1−B)ᵈ yₜ = Θ(B) εₜ
AR(p): yₜ = c + φ₁yₜ₋₁ + φ₂yₜ₋₂ + … + φₚyₜ₋ₚ + εₜ
MA(q): yₜ = μ + εₜ + θ₁εₜ₋₁ + θ₂εₜ₋₂ + … + θqεₜ₋q
I(d): difference the series d times to make it stationary
p AR order — lags of yₜ d differencing order q MA order — lags of εₜ B backshift operator: Byₜ = yₜ₋₁ εₜ white noise residual

Modern Approaches

Prophet (Meta)

An additive model designed for business forecasting, where non-statisticians need reliable results without deep expertise. It automatically handles missing data, outliers, and trend changepoints, and lets you add known holiday effects explicitly. Not the most accurate on every benchmark, but remarkably robust and easy to iterate on.

y(t) = g(t) + s(t) + h(t) + ε(t)

LSTM / GRU

Recurrent neural networks with gating mechanisms that selectively remember or forget past information across long sequences. They can theoretically capture patterns spanning hundreds of timesteps. Powerful when you have large datasets and complex long-range dependencies, but they're slow to train, sensitive to hyperparameters, and often outperformed by simpler models on short series.

Deep LearningLong horizon

Temporal Fusion Transformer

A transformer-based architecture built specifically for multi-horizon forecasting with mixed static and time-varying inputs. Its attention mechanisms are designed to be interpretable — you can inspect which past timesteps the model is paying attention to when making a prediction. Currently one of the strongest general-purpose deep learning forecasters.

TransformerMulti-horizon

N-BEATS / N-HiTS

Surprisingly effective architectures built entirely from MLPs (no recurrence, no attention). They use a basis expansion framework to explicitly decompose the forecast into interpretable trend and seasonality components. N-HiTS extends this with hierarchical interpolation for longer horizons. Both consistently rank near the top on standard forecasting benchmarks despite their architectural simplicity.
