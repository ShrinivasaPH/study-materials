What is Power BI?

The Data Model

The data model is the backbone of every Power BI report. It defines how your tables relate to each other — like a database schema.

Fact Tables vs. Dimension Tables

Fact Table
Contains measurable, quantitative data (numbers you want to sum/average). E.g., Sales, Orders, Transactions. Usually has many rows and foreign key columns pointing to dimension tables.
Dimension Table
Descriptive lookup data that provides context. E.g., Products, Customers, Date. Has fewer rows, unique keys, and descriptive text columns.

Relationship Types

One-to-Many (1:*)
Most common. One customer can place many orders. The "one" side is the dimension, the "many" side is the fact. DAX filters flow from the "one" side to the "many" side.
Many-to-Many (*:*)
More complex — avoid unless necessary. Example: a product can belong to multiple categories AND a category can have multiple products. Use bridge tables to resolve.
One-to-One (1:1)
Rare. Usually better to merge into a single table. Used for security row-level filtering in some scenarios.
Active vs Inactive
Only one active relationship can exist between two tables at a time. Use USERELATIONSHIP() in DAX to activate an inactive relationship temporarily.

The Star Schema

The recommended data model pattern in Power BI. One central fact table connected to multiple dimension tables — visually resembling a star.

📅 DimDate
👤 DimCustomer
⭐ FactSales
🏢 DimRegion
📦 DimProduct
✅
Star schema gives you the best query performance and the cleanest DAX. Always aim for this structure when designing your model.

Nomenclature & Glossary
Every term, symbol, and concept you'll encounter — clearly explained.

Core Terms

Measure
A dynamic calculation defined in DAX that is evaluated in the context of a visual or filter. Not stored as a column — recalculated each time a visual renders. Denoted with a small calculator icon (∑ or ⚙).
Calculated Column
A DAX expression evaluated row-by-row and stored in the model. Consumes more memory than measures. Use when you need to filter or slice by the result.
Filter Context
The set of active filters when a DAX expression is evaluated. Determined by slicers, page filters, visual-level filters, and row position. This is the most important concept in DAX!
Row Context
Exists inside iterators (SUMX, FILTER, etc.). Refers to the "current row" being processed in a loop. Calculated columns always have row context.
Context Transition
When a CALCULATE() function converts an existing row context into an equivalent filter context. Critical to understand for correct results in calculated columns inside CALCULATE.
Cardinality
The number of unique values in a column. High cardinality columns (like IDs) can slow performance. VertiPaq engine compresses low-cardinality columns better.
VertiPaq
Power BI's in-memory columnar storage engine. Compresses data and enables extremely fast queries on millions of rows.
PBIX
Power BI Desktop file format (.pbix). Contains the data model, queries, visuals, and report definitions in a single file.
Refresh
Process of re-importing data from the source into the model. Can be manual or scheduled (via Power BI Service). Incremental refresh only loads new/changed rows.
RLS (Row-Level Security)
Restricts which rows a user can see in the report based on their identity. Defined using DAX filter expressions on tables.

DAX Notation Guide

TableName[Column]
Full column reference syntax. Always preferred for clarity. E.g., Sales[Revenue]
[MeasureName]
Reference to a measure (no table prefix needed, but you can add one for clarity). E.g., [Total Sales]
BLANK()
DAX's equivalent of NULL. Different from 0 — BLANK propagates in calculations and is treated specially by visuals (hides from charts by default).
TRUE/FALSE
Boolean constants. Not quoted. = TRUE not = "TRUE"
"Text"
String literals in double quotes. Single quotes are for table names with spaces: 'My Sales Table'[Column]
VAR ... RETURN
Variable declarations in DAX. Variables are evaluated once and reused. Greatly improves performance and readability in complex measures.
Module 3
DAX Basics
Data Analysis Expressions — the formula language that makes Power BI powerful.
📌
DAX Philosophy: DAX functions operate on entire columns or tables, not individual cells like Excel. The magic is in how filter context shapes the data each function sees.

Aggregation Functions

Syntax
SUM(ColumnName)
Sums all values in a column, respecting the current filter context.
ColumnNameA numeric column to sum. Must be a column reference, not an expression.
DAX — Basic Aggregations
-- Basic SUM: Total Revenue Total Revenue = SUM(Sales[Revenue]) -- COUNT: Number of orders Order Count = COUNT(Sales[OrderID]) -- COUNTROWS: Rows in filtered table Total Transactions = COUNTROWS(Sales) -- AVERAGE: Avg order value Avg Order Value = AVERAGE(Sales[Revenue]) -- MIN / MAX Min Sale = MIN(Sales[Revenue]) Max Sale = MAX(Sales[Revenue]) -- DISTINCTCOUNT: Unique customers Unique Customers = DISTINCTCOUNT(Sales[CustomerID])

CALCULATE — The Most Important Function

CALCULATE modifies the filter context in which a measure is evaluated. It is the foundation of virtually all advanced DAX.

Syntax
CALCULATE(Expression, Filter1, Filter2, ...)
Evaluates Expression after applying or removing the specified filters.
ExpressionAny DAX expression (usually a measure) to evaluate in the modified context.
Filter1, Filter2Filter conditions: boolean expressions, table expressions, or modifier functions (ALL, KEEPFILTERS, REMOVEFILTERS…)
DAX — CALCULATE Examples
-- Sales for a specific region (ignoring any slicer) South Sales = CALCULATE( SUM(Sales[Revenue]), Sales[Region] = "South" ) -- Total sales ignoring ALL filters (grand total) Grand Total Sales = CALCULATE( SUM(Sales[Revenue]), ALL(Sales) ) -- % of Total (market share pattern) % of Total = DIVIDE( SUM(Sales[Revenue]), CALCULATE(SUM(Sales[Revenue]), ALL(Sales)), 0 )

Time Intelligence Functions

⚠️
Prerequisite: Time Intelligence requires a proper Date table marked as such in Power BI (right-click table → Mark as date table). The date column must be continuous with no gaps and of Date type.
DAX — Time Intelligence
-- Year-to-Date Sales Sales YTD = TOTALYTD( SUM(Sales[Revenue]), DimDate[Date] ) -- Month-over-Month growth Sales MoM Growth = VAR CurrentMonthSales = SUM(Sales[Revenue]) VAR PrevMonthSales = CALCULATE( SUM(Sales[Revenue]), PREVIOUSMONTH(DimDate[Date]) ) RETURN DIVIDE(CurrentMonthSales - PrevMonthSales, PrevMonthSales, 0) -- Same period last year Sales SPLY = CALCULATE( SUM(Sales[Revenue]), SAMEPERIODLASTYEAR(DimDate[Date]) ) -- Rolling 3-month average Rolling 3M Avg = AVERAGEX( DATESINPERIOD( DimDate[Date], LASTDATE(DimDate[Date]), -3, MONTH ), CALCULATE(SUM(Sales[Revenue])) )

Iterator Functions (X-Functions)

Iterator functions (SUMX, AVERAGEX, MAXX, etc.) loop row-by-row over a table, evaluate an expression for each row, then aggregate the results.

Pattern — SUMX
SUMX(Table, Expression)
For each row in Table, evaluates Expression, then sums all results.
TableCan be a physical table, a filtered table from FILTER(), or a virtual table from VALUES().
ExpressionAny DAX expression that returns a scalar (single number) evaluated in row context.
DAX — SUMX vs SUM (critical difference!)
-- WRONG: SUM can't multiply two columns directly Revenue (Wrong) = SUM(Sales[Quantity] * Sales[UnitPrice]) -- ERROR! -- CORRECT: SUMX iterates row-by-row Revenue = SUMX(Sales, Sales[Quantity] * Sales[UnitPrice]) -- Profit margin using SUMX Total Profit = SUMX( Sales, Sales[Revenue] - Sales[COGS] ) -- Weighted average price (not a simple AVERAGE!) Weighted Avg Price = DIVIDE( SUMX(Sales, Sales[Quantity] * Sales[UnitPrice]), SUM(Sales[Quantity]) )

Advanced DAX Patterns
Professional-grade patterns used in real enterprise dashboards.

FILTER + ALL — Context Control

DAX — Rank & Top N
-- Customer Sales Rank Customer Rank = RANKX( ALL(Customer[CustomerName]), CALCULATE(SUM(Sales[Revenue])), , DESC, DENSE ) -- Show only Top 5 products (use in visual filter) IsTop5 = IF( RANKX( ALL(Product[ProductName]), CALCULATE(SUM(Sales[Revenue])) ) <= 5, "Top 5", "Others" )

SWITCH — Readable Conditionals

DAX — SWITCH Pattern
-- Customer Segment based on spend Customer Tier = SWITCH( TRUE(), [Total Revenue] > 100000, "Platinum", [Total Revenue] > 50000, "Gold", [Total Revenue] > 10000, "Silver", "Bronze" ) -- Dynamic measure selector (with What-If parameter) Selected Metric = SWITCH( SELECTEDVALUE(MetricSelector[Metric]), "Revenue", [Total Revenue], "Profit", [Total Profit], "Orders", [Order Count], "Avg Value", [Avg Order Value] )

Running Total & Cumulative Patterns

DAX — Cumulative / Running Totals
-- Cumulative revenue over time Cumulative Revenue = CALCULATE( SUM(Sales[Revenue]), FILTER( ALL(DimDate[Date]), DimDate[Date] <= MAX(DimDate[Date]) ) )
Module 5
Measures vs Calculated Columns
When to use each — one of the most common sources of confusion for beginners.
⚡ Measure
Computed at query time. Lives in no table. Uses filter context. Stored as a formula, not data. Zero memory footprint. The preferred option for aggregations.
Dynamic No storage cost Best practice
📋 Calculated Column
Computed at data refresh time. Stored in the model (uses RAM). Uses row context. Necessary when you need to group/filter/slice by the result.
Static per row Uses memory Use for categories
DAX — Measure vs Calculated Column
-- ✅ MEASURE: Use when you want to sum/aggregate -- Create in: Home → New Measure Total Revenue = SUM(Sales[Revenue]) -- ✅ CALCULATED COLUMN: Use when you need to categorize rows -- Create in: Table → New Column Sales[PriceCategory] = IF(Sales[UnitPrice] > 1000, "Premium", "Standard") -- ✅ CALCULATED COLUMN: Full name (for display purposes) Customer[FullName] = Customer[FirstName] & " " & Customer[LastName] -- ✅ CALCULATED COLUMN: Days since order (row-level date math) Sales[DaysSinceOrder] = DATEDIFF(Sales[OrderDate], TODAY(), DAY)

Monthly Sales Performance Dashboard
Retail
🔴 Problem
A retail chain wants to track monthly revenue, compare it to the same month last year, and show which products are underperforming.
✅ Solution
Build a KPI dashboard with YoY comparison, conditional formatting, and a drill-through to product-level detail.
DAX — Sales Dashboard Measures
Monthly Sales = SUM(Sales[Revenue]) LY Sales = CALCULATE([Monthly Sales], SAMEPERIODLASTYEAR(DimDate[Date])) YoY Growth % = DIVIDE([Monthly Sales] - [LY Sales], [LY Sales], 0) -- Traffic light indicator for visual cues Performance Flag = SWITCH(TRUE(), [YoY Growth %] > 0.1, "🟢 Strong", [YoY Growth %] >= 0, "🟡 Stable", "🔴 Declining" )
Python — Test in Google Colab
# Simulate Power BI sales dashboard logic in Python (test in Colab) import pandas as pd import numpy as np # Create sample sales data np.random.seed(42) dates = pd.date_range("2023-01-01", periods=24, freq="ME") df = pd.DataFrame({ "Date": dates, "Revenue": np.random.randint(80000, 200000, 24) }) df["Year"] = df["Date"].dt.year df["Month"] = df["Date"].dt.month # YoY comparison (equivalent of SAMEPERIODLASTYEAR) ly = df[df["Year"] == 2023].set_index("Month")["Revenue"] cy = df[df["Year"] == 2024].set_index("Month")["Revenue"] result = pd.DataFrame({"CY Revenue": cy, "LY Revenue": ly}) result["YoY Growth %"] = ((cy - ly) / ly * 100).round(1) result["Flag"] = result["YoY Growth %"].apply( lambda x: "🟢 Strong" if x>10 else ("🟡 Stable" if x>=0 else "🔴 Declining") ) print(result)

HR Attrition Rate Analysis
HR / People Analytics
🔴 Problem
An HR team needs to identify departments with high employee turnover and predict which months see the most attrition.
✅ Solution
Calculate attrition rate per department and build a heatmap-style visual with risk flagging.
DAX — HR Attrition Measures
-- Total employees at start of period Headcount Start = CALCULATE(COUNTROWS(Employees), Employees[HireDate] <= STARTOFMONTH(DimDate[Date])) -- Employees who left this period Separations = CALCULATE(COUNTROWS(Employees), Employees[Status] = "Terminated") -- Monthly Attrition Rate % Attrition Rate = DIVIDE([Separations], [Headcount Start], 0) -- Risk label for conditional formatting Attrition Risk = IF([Attrition Rate] > 0.05, "HIGH RISK", "Normal")
Python — Test in Colab
import pandas as pd # Simulate HR attrition analysis data = { "Department": ["Engineering", "Sales", "HR", "Operations", "Finance"], "Headcount": [120, 85, 30, 200, 45], "Separations":[4, 9, 1, 18, 2] } df = pd.DataFrame(data) df["Attrition Rate %"] = (df["Separations"] / df["Headcount"] * 100).round(1) df["Risk"] = df["Attrition Rate %"].apply(lambda x: "🔴 HIGH" if x>5 else "✅ Normal") print(df.sort_values("Attrition Rate %", ascending=False).to_string(index=False))

Inventory Stockout Alerts
Supply Chain
🔴 Problem
A warehouse team needs to flag products that will stock out within 7 days based on current stock and average daily demand.
✅ Solution
Calculate Days of Supply per product and highlight items below the safety threshold.
DAX — Inventory Intelligence
-- Average daily units sold (last 30 days) Avg Daily Demand = DIVIDE( CALCULATE( SUM(Sales[UnitsSold]), DATESINPERIOD(DimDate[Date], TODAY(), -30, DAY) ), 30 ) -- Days of supply remaining Days of Supply = DIVIDE( SUM(Inventory[StockOnHand]), [Avg Daily Demand], 999 -- if no demand, 999 = no risk ) -- Stockout alert Stockout Alert = SWITCH(TRUE(), [Days of Supply] < 7, "🚨 CRITICAL", [Days of Supply] < 14, "⚠️ LOW", "✅ OK" )
Python — Inventory Simulation (Colab)
import pandas as pd inventory = pd.DataFrame({ "Product": ["Widget A", "Widget B", "Gadget X", "Gadget Y"], "Stock": [150, 20, 500, 8], "DailyDemand":[12, 5, 15, 4] }) inventory["Days of Supply"] = (inventory["Stock"] / inventory["DailyDemand"]).round(1) inventory["Alert"] = inventory["Days of Supply"].apply( lambda d: "🚨 CRITICAL" if d < 7 else ("⚠️ LOW" if d < 14 else "✅ OK") ) print(inventory.to_string(index=False))

Choosing the Right Visual
Picking the wrong chart type is the #1 mistake in dashboards.
Use when: Showing trends over time (days, months, years). Comparing two or more trend lines simultaneously.

Real example: Monthly Revenue vs Target line chart with a forecast band. Use reference lines to show average and targets.

Power BI tip: Enable "Forecast" in Analytics pane to add automatic time-series projection. Works best with daily or monthly granularity.
Use when: Comparing discrete categories. Bar charts (horizontal) work better for long category names. Column charts (vertical) work better for time periods.

Real example: Revenue by Product Category. Use conditional formatting to colour bars based on performance vs target.

Power BI tip: Sort by value (descending) to create a natural rank view. Use small multiples for "sparkline" style comparison across categories.
Use when: Explaining how a starting value changes through a series of positive and negative contributions to reach an ending value.

Real example: P&L bridge — starting with Gross Revenue, subtracting Returns, COGS, Operating Expenses, to arrive at Net Profit.

Power BI tip: Designate "Total" columns correctly in the formatting pane so they render as full bars, not increments.
Use when: Data has a geographic dimension and you want users to see spatial distribution (e.g. sales by city/state/country).

Real example: Filled Map showing sales intensity by Indian state, colour-coded from low (light) to high (dark amber).

Power BI tip: Always provide both city AND country/state to avoid geocoding ambiguity. Use ArcGIS Maps for advanced clustering.
Use when: Displaying a single important metric with its trend vs target.

Real example: Total Revenue: ₹48.2L | vs Target: ₹50L | -3.6% 🔴

Power BI tip: Use the new Card (v2) visual for richer formatting. Combine DAX measures for the value, target, and % deviation to show a complete KPI in a compact space.
