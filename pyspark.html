🎯 What is PySpark?

Imagine you have a billion rows of sales data. Your laptop fans spin up. Excel freezes. Even pandas starts sweating. This is where PySpark steps in — it laughs at a billion rows and asks for more.

PySpark is the Python API for Apache Spark, a distributed computing engine built to process massive datasets across many machines in parallel. You write Python; Spark handles the cluster magic behind the scenes.

🎉 Fun Fact — Where "Spark" Comes From:
Apache Spark was born in 2009 at UC Berkeley's AMPLab to fix what everyone hated about Hadoop MapReduce. Hadoop was slow because it wrote intermediate results to disk after every step. Spark said: what if we kept things in memory instead? The result was up to 100× faster for certain workloads. The name "Spark" literally comes from the idea of a spark over Hadoop's slow fire — faster ignition, brighter flame.

Why PySpark? Why Not Just Pandas?

Pandas is brilliant — and you should keep using it for data that fits in your laptop's RAM. But once you're dealing with gigabytes, terabytes, or petabytes, pandas starts gasping. PySpark was built for exactly that scale.

The Four Core Concepts

RDD (Resilient Distributed Dataset): Spark's original low-level data structure. An immutable, distributed collection of objects spread across the cluster. Think of it as a Python list that lives on 100 machines simultaneously.
DataFrame: A distributed table with named columns — like a pandas DataFrame, but spread across a cluster and optimized by Spark's Catalyst engine. This is what you'll use 90% of the time in real work.
Transformations (Lazy): Operations like filter, select, groupBy that describe what you want to do. Spark doesn't run them immediately — it builds a plan first. Nothing happens until you ask for results.
Actions: Operations like show(), count(), collect() that say "okay, now actually do it." This is when Spark executes the full optimized plan.
Why Lazy Evaluation? Because Spark can look at your entire chain of operations, reorder things, skip unnecessary steps, and find the most efficient execution path — all before touching a single row of data. It's like telling a chef your full order vs. interrupting them after every ingredient.

Spark Architecture

Spark follows a master-worker architecture:

  • Driver Program: Your Python script. Runs on one machine, creates the SparkContext, builds the execution plan, and coordinates everything.
  • Cluster Manager: The resource allocator — decides which worker gets which task. Can be YARN, Kubernetes, Mesos, or Spark Standalone.
  • Worker Nodes + Executors: The actual workhorses. Each worker runs Executor processes that handle the data crunching in parallel.
  • Tasks: The smallest unit of work. Each data partition gets processed by one task on one core.
🧠 The DAG — Spark's Secret Weapon:
When you chain transformations, Spark builds a Directed Acyclic Graph (DAG) — a visual map of all operations and their dependencies. The DAG Scheduler finds the most efficient execution order, grouping work into Stages. You can see this graph live at http://localhost:4040 while any job runs. Genuinely satisfying to watch.

⚙️ Installation & Setup

Good news: getting PySpark running locally is much easier than it used to be. A few years ago you'd spend half a day fighting Java environment variables. Now? One pip command and you're mostly there.

Prerequisites

  • Java JDK 8 or 11 — PySpark runs on the JVM because Spark itself is written in Scala. Check with java -version in your terminal.
  • Python 3.7+ — PySpark 3.x dropped Python 2 support (rightly so).
  • 8GB+ RAM recommended — Spark loves memory. Give it what it wants.
🤔 Why does Spark need Java?
Spark is written in Scala, which compiles to JVM bytecode. PySpark is a Python wrapper that communicates with the JVM backend through a library called Py4J. When you write df.filter(...), Python sends instructions to the JVM, which does the actual data crunching. This also explains why you occasionally see Java stack traces in your Python errors — a bit startling at first, but you get used to it.

Install PySpark

pip install pyspark # With optional extras (recommended for data science) pip install pyspark[sql] # SQL + DataFrame support pip install pyspark[ml] # MLlib for machine learning pip install pyspark[sql,ml] # Both -- the full toolkit
No cluster? No problem. For learning, run Spark in "local mode" on your laptop — it simulates a cluster using your CPU cores. You can also use Databricks Community Edition (free) or run !pip install pyspark in Google Colab — both work great for learning without any cluster setup.

Create Your First SparkSession

The SparkSession is your entry point to everything PySpark. It replaced the older trio of SparkContext + SQLContext + HiveContext in Spark 2.0. Think of it as the front door to the Spark universe.

from pyspark.sql import SparkSession # The builder pattern -- fluent and readable spark = SparkSession.builder \ .appName("MyFirstApp") \ # Name shown in the Spark UI .master("local[*]") \ # Use ALL available CPU cores locally .config("spark.sql.shuffle.partitions", "4") \ # Sensible for local dev .getOrCreate() # Get existing session or create new # Access SparkContext for RDD operations sc = spark.sparkContext print(f"Spark Version: {spark.version}")
What does local[*] mean? It tells Spark to run locally using all available CPU cores (* = all of them). Write local[4] to use exactly 4 cores, or just local for a single thread. In a real cluster, replace this with a YARN or Kubernetes master URL.

Test Your Installation

# Quick sanity check -- if this runs, you're good to go rdd = sc.parallelize([1, 2, 3, 4, 5]) result = rdd.map(lambda x: x * 2).collect() print(result) # [2, 4, 6, 8, 10] # DataFrame test data = [("PySpark", "Works!")] df = spark.createDataFrame(data, ["library", "status"]) df.show() # +-------+------+ # |library|status| # +-------+------+ # |PySpark|Works!| # +-------+------+

Useful Local Development Configs

# Tone down Spark's noisy INFO logs (highly recommended) spark.sparkContext.setLogLevel("WARN") # Pretty table rendering in Jupyter notebooks spark.conf.set("spark.sql.repl.eagerEval.enabled", True) # Give the driver more memory (set this BEFORE creating the session) spark = SparkSession.builder \ .config("spark.driver.memory", "4g") \ .getOrCreate()
⚠️ Always stop your session when done!
Leaving a SparkSession running keeps JVM processes alive and eats memory. End scripts with spark.stop(), or use a try/finally block. Your laptop will thank you.

📦 RDDs (Resilient Distributed Datasets)

RDDs are Spark's original data abstraction — the foundation everything else is built on. They're lower-level than DataFrames, but understanding them is the difference between a PySpark user who copy-pastes code and one who actually understands what's happening under the hood.

What Makes an RDD "Resilient"?

Every RDD knows its lineage — the exact chain of operations that created it. If a node dies and loses its portion of the data, Spark doesn't panic. It looks at the lineage and recomputes the missing partition from scratch.

🧬 The Recipe Analogy:
Instead of keeping a backup copy of every cake you bake, you keep the recipe. If a cake is ruined, you bake it again from the same recipe. Spark stores lineage graphs (recipes), not full data copies — making fault tolerance essentially free.

The three defining properties of every RDD:

  • Partitioned: Data is split into chunks spread across the cluster. More partitions = more parallelism. Default is 2× the number of CPU cores.
  • Immutable: You can never modify an RDD in place. Transformations always return a new RDD. This is what makes lineage tracking clean and reliable.
  • Lazy: Defining a transformation doesn't run anything. Spark waits until an action is called, then executes the entire optimized plan in one shot.

Creating RDDs

Method 1: From a Python collection

rdd = sc.parallelize([1, 2, 3, 4, 5]) # Control the number of partitions (affects parallelism) rdd = sc.parallelize(range(100), numSlices=4) print(rdd.getNumPartitions()) # 4 # Key-value pairs -- very common pattern kv = sc.parallelize([("Electronics", 1200), ("Clothing", 300)])

Method 2: From external storage

# Each line becomes one element rdd = sc.textFile("path/to/file.txt") # Glob patterns work -- read multiple files at once rdd = sc.textFile("data/logs/*.log")

Method 3: Transform an existing RDD

base = sc.parallelize([1, 2, 3, 4, 5]) doubled = base.map(lambda x: x * 2) # New RDD -- base unchanged

Core RDD Operations

rdd = sc.parallelize([1, 2, 3, 4, 5, 6]) # --- TRANSFORMATIONS (lazy, return new RDDs) --- squared = rdd.map(lambda x: x ** 2) # [1, 4, 9, 16, 25, 36] evens = rdd.filter(lambda x: x % 2 == 0) # [2, 4, 6] unique = sc.parallelize([1,1,2,3,3]).distinct() # [1, 2, 3] # flatMap vs map -- easy to confuse! sentences = sc.parallelize(["Hello World", "Big Data"]) words_map = sentences.map(lambda x: x.split()) # [["Hello","World"],["Big","Data"]] words_flat = sentences.flatMap(lambda x: x.split()) # ["Hello","World","Big","Data"] # --- ACTIONS (trigger execution) --- print(rdd.collect()) # [1,2,3,4,5,6] -- brings ALL data to driver! print(rdd.count()) # 6 print(rdd.first()) # 1 print(rdd.take(3)) # [1, 2, 3] print(rdd.sum()) # 21 print(rdd.mean()) # 3.5 print(rdd.max()) # 6

Key-Value RDDs (Pair RDDs) — The Power Move

When your RDD contains (key, value) tuples, you unlock a whole family of powerful aggregation operations. This is the MapReduce pattern made elegant.

sales = sc.parallelize([ ("Electronics", 1200), ("Clothing", 300), ("Electronics", 800), ("Clothing", 450), ("Electronics", 500), ]) # Sum values per key totals = sales.reduceByKey(lambda a, b: a + b) print(totals.collect()) # [('Electronics', 2500), ('Clothing', 750)] # Count items per key print(sales.countByKey()) # {'Electronics': 3, 'Clothing': 2} # Sort by value descending sorted_totals = totals.sortBy(lambda x: x[1], ascending=False)
reduceByKey vs groupByKey — a classic interview question:
Always prefer reduceByKey for aggregations. groupByKey shuffles all values across the network before reducing — very wasteful. reduceByKey partially reduces on each partition first, then shuffles much less data. The performance difference on large datasets is dramatic.
Classic: Word Count — The "Hello World" of big data
text = sc.parallelize(["spark is fast spark is fun", "spark uses memory"]) word_counts = ( text .flatMap(lambda line: line.split()) .map(lambda word: (word, 1)) .reduceByKey(lambda a, b: a + b) .sortBy(lambda x: x[1], ascending=False) ) word_counts.collect() # [('spark', 3), ('is', 2), ('fast', 1), ('fun', 1), ('uses', 1), ('memory', 1)]

📊 DataFrames

DataFrames are the modern face of PySpark — structured, optimized, and far more intuitive than raw RDDs for tabular data. Introduced in Spark 1.3, they brought named columns, schema awareness, and dramatic performance improvements via the Catalyst optimizer. For 95% of real-world work, DataFrames are your primary tool.

⚡ What is Catalyst?
Catalyst is Spark's query optimizer. When you write DataFrame operations, Catalyst rewrites your logical plan into an optimized physical plan — reordering filters, eliminating redundant steps, choosing efficient join strategies. It's like having a senior DBA silently optimize every query you write, automatically, every single time.

Creating DataFrames

Method 1: From Python data

data = [("Alice", 25, 55000), ("Bob", 30, 72000), ("Charlie", 35, 91000)] df = spark.createDataFrame(data, schema=["name", "age", "salary"]) df.show()

Method 2: Explicit schema — best practice

from pyspark.sql.types import StructType, StructField, StringType, IntegerType schema = StructType([ StructField("name", StringType(), nullable=True), StructField("age", IntegerType(), nullable=True), StructField("salary", IntegerType(), nullable=True), ]) df = spark.createDataFrame(data, schema=schema) df.printSchema()
Why define schema explicitly? When you use inferSchema=True, Spark scans a sample of your data to guess types — which costs time and can guess wrong (e.g., an integer column with some nulls may be inferred as string). Explicit schemas are faster, safer, and self-documenting.

Method 3: From files

# CSV df = spark.read \ .option("header", True) \ .option("inferSchema", True) \ .csv("employees.csv") # Parquet -- Spark's native columnar format, very fast df = spark.read.parquet("data/sales.parquet") # JSON (one JSON object per line) df = spark.read.json("events.json") # Convert from pandas (for small data) import pandas as pd pdf = pd.DataFrame({"x": [1, 2, 3], "y": ["a", "b", "c"]}) df = spark.createDataFrame(pdf)

Exploring Your DataFrame

df.show() # First 20 rows df.show(5, truncate=False) # 5 rows, full column values df.printSchema() # Column names and types df.dtypes # [(col_name, dtype), ...] df.columns # List of column names df.count() # Row count (triggers a job) df.describe().show() # Mean, stddev, min, max per column # Peek at the execution plan -- great for debugging slow queries df.filter(df.age > 25).explain(True)

Common DataFrame Manipulations

from pyspark.sql.functions import col, when, round, coalesce, lit # Add a new column df = df.withColumn("salary_in_lakhs", round(col("salary") / 100000, 2)) # Rename a column df = df.withColumnRenamed("salary", "annual_salary") # Drop a column df = df.drop("unwanted_col") # Conditional column (like SQL CASE WHEN) df = df.withColumn("grade", when(col("salary") > 80000, "Senior") .when(col("salary") > 50000, "Mid") .otherwise("Junior") ) # Cast to a different type df = df.withColumn("age", col("age").cast("double")) # Handle nulls df.na.drop() # Drop rows with ANY null df.na.drop(subset=["salary", "age"]) # Drop rows null in specific columns df.na.fill(0) # Fill numeric nulls with 0 df.na.fill({"salary": 50000, "name": "Unknown"}) # Fill per column

Joins

# Inner join (only matching rows on both sides) result = employees.join(departments, on="dept_id", how="inner") # Left join -- keep ALL employees, even if no matching dept result = employees.join(departments, on="dept_id", how="left") # Anti join -- employees with NO matching department (finding orphans) result = employees.join(departments, on="dept_id", how="left_anti") # Join on expression when column names differ result = employees.join(departments, employees.dept_id == departments.id, how="left")
⚠️ The Shuffle Join Warning:
Joining large DataFrames triggers a shuffle — data moves across the network so matching rows land on the same machine. This is expensive. If one DataFrame is small (<100MB), use a broadcast join: from pyspark.sql.functions import broadcast, then df1.join(broadcast(df2), ...). This sends the small DF to every worker, skipping the shuffle entirely.

🔍 SQL Operations

One of PySpark's killer features: you can write real, full SQL and execute it against distributed data. If you already know SQL — and most data people do — you're already halfway to being productive with PySpark.

Registering DataFrames as Views

To query a DataFrame with SQL, register it as a temporary view. This just gives it a name SQL can reference — the data doesn't move, the view is just an alias.

# Session-scoped view -- gone when the session ends df.createOrReplaceTempView("employees") # Global view -- accessible across sessions df.createGlobalTempView("employees_global") # Reference with: SELECT * FROM global_temp.employees_global result = spark.sql(""" SELECT name, age, salary FROM employees WHERE age > 25 ORDER BY salary DESC LIMIT 10 """) result.show()

Full SQL Power — More Than Just SELECT

# GROUP BY aggregation spark.sql(""" SELECT department, COUNT(*) AS headcount, AVG(salary) AS avg_salary, MAX(salary) AS top_salary FROM employees GROUP BY department ORDER BY avg_salary DESC """).show() # CASE WHEN spark.sql(""" SELECT name, salary, CASE WHEN salary > 80000 THEN 'Senior' WHEN salary > 50000 THEN 'Mid' ELSE 'Junior' END AS grade FROM employees """).show() # Subquery -- find everyone above company average spark.sql(""" SELECT name, salary FROM employees WHERE salary > (SELECT AVG(salary) FROM employees) """).show() # Window function in SQL spark.sql(""" SELECT name, department, salary, RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS dept_rank, salary - AVG(salary) OVER (PARTITION BY department) AS vs_dept_avg FROM employees """).show() # CTE (WITH clause) -- great for readable complex queries spark.sql(""" WITH dept_avg AS ( SELECT department, AVG(salary) AS avg_sal FROM employees GROUP BY department ) SELECT e.name, e.salary, da.avg_sal, e.salary - da.avg_sal AS diff FROM employees e JOIN dept_avg da ON e.department = da.department """).show()

SQL vs DataFrame API — They're Equal

Same query, two styles. Both produce identical results and run the same optimized physical plan:

from pyspark.sql.functions import avg, count, col # SQL style spark.sql(""" SELECT department, COUNT(*) AS hc, AVG(salary) AS avg_sal FROM employees WHERE age > 25 GROUP BY department ORDER BY avg_sal DESC """).show() # DataFrame API -- identical result (employees .filter(col("age") > 25) .groupBy("department") .agg(count("*").alias("hc"), avg("salary").alias("avg_sal")) .orderBy(col("avg_sal").desc()) .show() )
Tip: You can mix SQL and DataFrame API freely in the same script. Both are compiled by Catalyst to the same physical plan — there's no performance difference. Use SQL when the logic is naturally query-shaped or when non-programmers need to read it. Use the DataFrame API when you need programmatic control.

⚡ Transformations

Transformations are the "describe what you want" part of PySpark. They're lazy — Spark listens, builds a plan, and does nothing until you call an action. This laziness lets Spark optimize your entire pipeline before touching a single byte.

Narrow vs Wide Transformations — This Matters

  • Narrow transformations: Each output partition depends on only one input partition. No data moves between nodes. Fast! Examples: map, filter, flatMap, select, withColumn.
  • Wide transformations (shuffles): Output partitions depend on multiple input partitions — data must cross the network. Expensive. Examples: groupBy, join, distinct, orderBy, repartition.
🚚 The Shuffle Tax:
A shuffle is like asking everyone in a building to swap offices based on the first letter of their name — total movement, total overhead. In Spark, shuffles write intermediate data to disk and transfer it over the network. This is why minimizing unnecessary shuffles is a key optimization goal.

Common DataFrame Transformations

1. select

from pyspark.sql.functions import col, upper, concat_ws # Simple selection df.select("name", "age").show() # Derive new columns inline df.select( col("name"), upper(col("name")).alias("NAME_UPPER"), (col("salary") * 1.1).alias("with_bonus"), concat_ws(" - ", col("name"), col("department")).alias("name_dept") ).show()

2. filter / where

# All of these do the same thing df.filter(df.age > 25) df.filter("age > 25") # SQL string syntax works too! df.where(col("age") > 25) # Compound conditions -- use & | ~ (not Python 'and'/'or') df.filter((col("age") > 25) & (col("dept") == "Engineering")) df.filter((col("salary") > 80000) | (col("title") == "Manager")) df.filter(~col("is_contract")) # NOT # isin -- like SQL IN df.filter(col("dept").isin(["Engineering", "Product", "Design"])) # Filter nulls df.filter(col("email").isNull()) df.filter(col("email").isNotNull())

3. groupBy + agg

from pyspark.sql.functions import count, avg, sum, max, min, stddev, collect_list df.groupBy("department").agg( count("*").alias("headcount"), avg("salary").alias("avg_salary"), sum("salary").alias("total_payroll"), max("salary").alias("top_earner"), stddev("salary").alias("salary_spread"), collect_list("name").alias("all_members") ).show()

4. Window Functions — the underrated superpower

Window functions let you calculate aggregates without collapsing rows. Rank employees, compute running totals, compare each row to its group average — all without a groupBy.

from pyspark.sql.window import Window from pyspark.sql.functions import rank, dense_rank, row_number, lag, lead # Rank employees per department by salary window = Window.partitionBy("department").orderBy(col("salary").desc()) df = df.withColumn("dept_rank", rank().over(window)) # 1,2,2,4 (skips ties) df = df.withColumn("dense_rank", dense_rank().over(window)) # 1,2,2,3 (no skip) df = df.withColumn("row_num", row_number().over(window)) # 1,2,3,4 (always unique) # Running total (cumulative sum) win_run = Window.partitionBy("dept").orderBy("date") \ .rowsBetween(Window.unboundedPreceding, 0) df = df.withColumn("running_total", sum("sales").over(win_run)) # Lag/Lead -- compare with previous or next row win2 = Window.partitionBy("dept").orderBy("month") df = df.withColumn("prev_month", lag("sales", 1).over(win2)) df = df.withColumn("next_month", lead("sales", 1).over(win2))
Key Point: Transformations are lazy — they don't execute until you call an action like collect(), show(), count(), or write(). Spark builds the entire plan first, optimizes it, then executes everything in one sweep.

🎬 Actions

If transformations are writing your to-do list, actions are actually doing the things. They trigger execution, break the lazy spell, and make Spark run everything it's been quietly planning. Every Spark job ends with at least one action.

Viewing Actions

show() — your best friend during development

df.show() # 20 rows, truncate long strings df.show(5) # First 5 rows df.show(truncate=False) # Full column values, no cutting df.show(10, False) # 10 rows, no truncation

collect() — use with extreme caution

rows = df.collect() # Returns list of Row objects to the driver print(rows[0].name) # Access by column name print(rows[0]["age"]) # Or by key
⚠️ The collect() Trap:
collect() on a 100GB dataset tries to pull all of it into your driver's RAM. Best case: it crawls. Worst case: it crashes with OutOfMemoryError. Use limit(n).collect() or just show(n) when exploring. Reserve collect() for when you genuinely need all data locally — after heavy filtering that brings it to manageable size.

Other viewing actions

first_row = df.first() # One Row object top_3 = df.head(3) # List of 3 Row objects pdf = df.limit(1000).toPandas() # Convert to pandas (small data only!) df.select("department").distinct().show()

Counting & Stats

total = df.count() print(f"Total records: {total:,}") from pyspark.sql.functions import countDistinct df.select(countDistinct("department")).show() df.describe("salary", "age").show() # RDD stats actions rdd = sc.parallelize([10, 20, 30, 40, 50]) print(rdd.sum(), rdd.mean(), rdd.max(), rdd.min(), rdd.stdev())

Writing / Saving Actions

# Write modes: # "overwrite" -- delete existing and replace # "append" -- add to existing # "ignore" -- skip if already exists # "error" -- fail if already exists (default) # CSV df.write.mode("overwrite").option("header", True).csv("output/employees") # Parquet (recommended -- columnar, compressed, fast to query) df.write.mode("overwrite").parquet("output/employees.parquet") # Partition output -- massively speeds up downstream queries df.write.partitionBy("year", "month") \ .mode("overwrite") \ .parquet("output/sales_partitioned") # Creates: output/year=2024/month=01/part-*.parquet # Queries filtered by year/month only read relevant folders -- huge speedup # Write to database via JDBC df.write.format("jdbc") \ .option("url", "jdbc:postgresql://host:5432/mydb") \ .option("dbtable", "employees") \ .option("user", "user").option("password", "pass") \ .mode("append").save()

Caching — Making Repeated Actions Faster

df.cache() # Keep in memory between actions print(df.count()) # Computed and cached here df.show() # Uses cache -- no recomputation df.describe().show() # Still using cache df.unpersist() # Free the memory when done # Choose storage level explicitly from pyspark.storagelevel import StorageLevel df.persist(StorageLevel.MEMORY_AND_DISK) # Spills to disk if needed (safe default) df.persist(StorageLevel.MEMORY_ONLY) # Fastest, but can lose data if evicted df.persist(StorageLevel.DISK_ONLY) # For very large DFs that won't fit in RAM
Example: Action triggers all queued transformations
# Nothing runs yet -- just building the plan df2 = df.filter(df.age > 25) df3 = df2.select("name", "department") df4 = df3.groupBy("department").count() # THIS triggers all of the above in one optimized execution df4.show()

🧪 UDFs & Built-in Functions

PySpark ships with hundreds of built-in functions for strings, dates, math, arrays, and aggregations. Before you ever write a custom function, check if there's already a built-in. Built-ins run entirely in Spark's JVM engine — no Python overhead, maximum speed.

Built-in Functions — Use These First

from pyspark.sql.functions import ( # String upper, lower, trim, length, substring, split, concat, concat_ws, regexp_replace, regexp_extract, initcap, lpad, rpad, instr, locate, # Date / Time current_date, current_timestamp, date_add, date_sub, datediff, months_between, to_date, to_timestamp, year, month, dayofmonth, hour, minute, second, date_format, # Math round, floor, ceil, abs, sqrt, log, exp, pow, # Array array, array_contains, size, explode, array_distinct, sort_array, # Null handling coalesce, isnull, isnan, # Aggregate count, sum, avg, max, min, countDistinct, approx_count_distinct ) # Example: clean a messy dataset using only built-ins df = df.withColumn("clean_name", initcap(trim(col("name")))) df = df.withColumn("join_year", year(to_date(col("join_date"), "dd-MM-yyyy"))) df = df.withColumn("email", lower(regexp_replace(col("email"), r"\s+", "")))

UDFs — When Built-ins Aren't Enough

from pyspark.sql.functions import udf from pyspark.sql.types import StringType, IntegerType # Method 1: Decorator syntax (cleanest) @udf(returnType=StringType()) def classify_salary(salary): if salary is None: return "Unknown" elif salary >= 100000: return "Senior" elif salary >= 60000: return "Mid" else: return "Junior" df = df.withColumn("grade", classify_salary(col("salary"))) # Method 2: Register for SQL use as well def mask_email(email): if email: parts = email.split("@") return parts[0][:3] + "***@" + parts[1] return None mask_email_udf = udf(mask_email, StringType()) spark.udf.register("mask_email", mask_email, StringType()) df.withColumn("masked", mask_email_udf(col("email"))).show() spark.sql("SELECT name, mask_email(email) FROM users").show()
⚠️ UDF Performance Warning:
Python UDFs are slow. Each row crosses the Python↔JVM boundary: Spark serializes data to Python, your function runs, data is serialized back to JVM. For a billion rows, this overhead is severe. Always check built-ins first. If you must write custom logic, consider Pandas UDFs (vectorized) — they work on batches using Apache Arrow and can be 10–100× faster.

Pandas UDFs — The Fast Alternative

from pyspark.sql.functions import pandas_udf from pyspark.sql.types import DoubleType import pandas as pd # Receives a pandas Series, returns a pandas Series # Works on batches -- much faster than row-by-row UDFs @pandas_udf(DoubleType()) def normalize_salary(s: pd.Series) -> pd.Series: return (s - s.mean()) / s.std() df = df.withColumn("salary_norm", normalize_salary(col("salary")))

🔧 Optimization & Tuning

This is where "it works" becomes "it's fast." PySpark does a lot automatically via Catalyst, but knowing how to help the optimizer — and when to override it — separates good engineers from great ones.

1. Partitioning — The Most Important Knob

Partitions are the unit of parallelism. Each partition is processed by one task on one core. Too few: cores sit idle. Too many: excessive scheduler overhead.

# Check current partition count print(df.rdd.getNumPartitions()) # repartition: full shuffle, creates balanced partitions df = df.repartition(8) # 8 evenly balanced partitions df = df.repartition("department") # Co-locate rows by department # coalesce: reduce partitions WITHOUT a shuffle (merges locally) df = df.coalesce(1) # Single output file -- use before writing small results # Tune shuffle partitions (affects groupBy, join, distinct) spark.conf.set("spark.sql.shuffle.partitions", "4") # Great for local dev spark.conf.set("spark.sql.shuffle.partitions", "200") # Default for clusters
Rule of Thumb: Aim for partition sizes of 100–200MB each. For local development, 4–8 partitions is fine. The default of 200 shuffle partitions is overkill locally — set it to 4 and your local jobs noticeably speed up.

2. Broadcast Joins

from pyspark.sql.functions import broadcast # When one side is small (< ~100MB) big_df.join(broadcast(small_df), on="id") # Sends small_df to every executor in memory -- big_df never moves # Set auto-broadcast threshold spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "100mb")

3. Filter Early

# GOOD -- filter first, carry less data through expensive operations df.filter(col("year") == 2024) \ .filter(col("country") == "India") \ .groupBy("city").agg(sum("sales")) # AVOID -- grouping all data, then filtering (wastes resources) df.groupBy("year", "country", "city").agg(sum("sales")) \ .filter((col("year") == 2024) & (col("country") == "India"))

4. explain() — X-Ray Vision for Your Queries

result = df.filter(col("salary") > 70000).groupBy("dept").agg(avg("salary")) result.explain() # Physical plan only (quick check) result.explain(True) # Full: parsed -> analyzed -> optimized -> physical result.explain("cost") # With cost estimates (PySpark 3.x)

🌍 Real-World Project: E-Commerce Analytics Pipeline

Time to put it all together. This is the kind of pipeline you'd actually build on the job — ingesting messy data, cleaning it, enriching it, running multiple analyses, and writing structured output. Every concept from the previous lessons appears here.

The Business Problem

You're a data engineer at an e-commerce company. The product team wants a daily report: Which categories are growing? Who are the top customers? How is each city performing? You need to build the pipeline that powers it.

Step 1: Setup

from pyspark.sql import SparkSession from pyspark.sql.functions import ( col, when, round, year, month, to_date, count, sum, avg, max, coalesce, lit, rank ) from pyspark.sql.window import Window spark = SparkSession.builder \ .appName("EcommerceAnalytics") \ .master("local[*]") \ .config("spark.sql.shuffle.partitions", "4") \ .getOrCreate() spark.sparkContext.setLogLevel("WARN")

Step 2: Load and Clean Data

# In real life: spark.read.csv("orders.csv", header=True, inferSchema=True) orders_data = [ ("O001", "C101", "Electronics", "Laptop", 75000, 2, "2024-01-15", "Mumbai"), ("O002", "C102", "Clothing", "T-Shirt", 799, 3, "2024-01-16", "Delhi"), ("O003", "C101", "Electronics", "Phone", 45000, 1, "2024-02-01", "Mumbai"), ("O004", "C103", "Books", "Python Book", 599, 2, "2024-02-10", "Bengaluru"), ("O005", "C104", "Electronics", "Headphones", 3500, 1, "2024-02-14", "Chennai"), ("O006", "C102", "Clothing", "Jeans", 1999, 2, "2024-03-01", "Delhi"), ("O007", "C105", "Books", "Spark Guide", 799, 1, "2024-03-05", "Pune"), ("O008", "C103", "Electronics", "Tablet", 28000, 1, "2024-03-20", "Bengaluru"), ("O009", None, "Electronics", "Keyboard", 2500, None, "2024-03-22", "Hyderabad"), ("O010", "C101", "Clothing", "Jacket", 4500, 1, "2024-04-01", "Mumbai"), ] cols = ["order_id", "customer_id", "category", "product", "unit_price", "quantity", "order_date", "city"] raw_df = spark.createDataFrame(orders_data, schema=cols) # Clean nulls, cast types, derive new columns cleaned_df = raw_df \ .withColumn("quantity", coalesce(col("quantity"), lit(1))) \ .withColumn("customer_id", coalesce(col("customer_id"), lit("GUEST"))) \ .withColumn("order_date", to_date(col("order_date"))) \ .withColumn("order_year", year(col("order_date"))) \ .withColumn("order_month", month(col("order_date"))) \ .withColumn("revenue", round(col("unit_price") * col("quantity"), 2)) \ .withColumn("revenue_tier", when(col("revenue") >= 50000, "High") .when(col("revenue") >= 5000, "Mid") .otherwise("Low") ) cleaned_df.cache() # Cache -- multiple analyses will reuse this print(f"Records loaded: {cleaned_df.count()}") cleaned_df.show()

Step 3: Analysis 1 — Category Performance

category_stats = cleaned_df.groupBy("category").agg( count("*").alias("order_count"), round(sum("revenue"), 2).alias("total_revenue"), round(avg("revenue"), 2).alias("avg_order_value"), max("revenue").alias("max_order"), sum("quantity").alias("units_sold") ).orderBy(col("total_revenue").desc()) category_stats.show()

Step 4: Analysis 2 — Top Customers via Window Function

customer_totals = cleaned_df.groupBy("customer_id", "city").agg( count("*").alias("order_count"), round(sum("revenue"), 2).alias("lifetime_value") ) window = Window.orderBy(col("lifetime_value").desc()) top_customers = customer_totals \ .withColumn("rank", rank().over(window)) \ .filter(col("rank") <= 5) top_customers.show()

Step 5: Analysis 3 — Monthly Trends via SQL

cleaned_df.createOrReplaceTempView("orders") monthly_trends = spark.sql(""" SELECT order_year, order_month, category, COUNT(*) AS order_count, ROUND(SUM(revenue), 2) AS monthly_revenue, ROUND(AVG(revenue), 2) AS avg_order_value, ROUND( SUM(revenue) - LAG(SUM(revenue), 1) OVER (PARTITION BY category ORDER BY order_year, order_month), 2 ) AS mom_change FROM orders GROUP BY order_year, order_month, category ORDER BY order_year, order_month, monthly_revenue DESC """) monthly_trends.show()

Step 6: Write Outputs

# Category summary as CSV (for stakeholders) category_stats \ .coalesce(1) \ .write.mode("overwrite") \ .option("header", True) \ .csv("output/category_summary") # Partitioned Parquet for downstream Spark jobs cleaned_df \ .write.mode("overwrite") \ .partitionBy("order_year", "order_month") \ .parquet("output/cleaned_orders") cleaned_df.unpersist() spark.stop() print("Pipeline complete!")
