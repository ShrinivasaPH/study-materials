What is SQL?

INTRO

SQL (Structured Query Language) is the standard language for communicating with relational databases. Every time you search products on Amazon, book a flight, or log into an app — SQL is quietly working behind the scenes.

📊

Query Data

Retrieve exactly the rows and columns you need from millions of records

✏️

Modify Data

Insert, update, or delete records with precise control

🏗️

Define Structure

Create tables, define data types, set constraints and relationships

🔒

Control Access

Grant and revoke permissions on tables and operations

SQL Sub-Languages

Sub-languageStands ForPurposeCommands
DDLData Definition LanguageDefine structureCREATE, ALTER, DROP, TRUNCATE
DMLData Manipulation LanguageModify data (CRUD)INSERT, SELECT, UPDATE, DELETE
DCLData Control LanguageAccess permissionsGRANT, REVOKE
TCLTransaction Control LanguageManage transactionsCOMMIT, ROLLBACK, SAVEPOINT
DQLData Query LanguageQuery/retrieve dataSELECT
💡

Key Mental Model: Think of a relational database as a collection of interconnected spreadsheets (tables). SQL is the language you use to ask questions about those spreadsheets — and modify them.

Nomenclature Reference

TABLE
A structured grid of rows and columns storing related data. e.g., employees, orders, products
ROW / RECORD
A single entry in a table representing one entity instance. e.g., one employee
COLUMN / FIELD
A named attribute for all rows in a table. e.g., employee_name, salary
PRIMARY KEY
A column (or set) that uniquely identifies each row. Cannot be NULL or duplicate. e.g., employee_id
FOREIGN KEY
A column referencing the PRIMARY KEY of another table to enforce relationships. e.g., dept_id in employees
INDEX
A performance structure that speeds up data retrieval (like a book index). Trades write speed for read speed
NULL
Absence of a value — NOT zero, NOT empty string. Requires IS NULL / IS NOT NULL to check.
CONSTRAINT
A rule enforced on a column. Types: PRIMARY KEY, FOREIGN KEY, UNIQUE, NOT NULL, CHECK, DEFAULT
SCHEMA
The blueprint of the database — its tables, columns, types, and relationships.
QUERY
An instruction to the database engine written in SQL. Queries are declarative — you say what you want, not how to get it.

Adds one or more new rows to a table.

INSERT INTO table_name (col1, col2, ...) VALUES (val1, val2, ...);

1.2 READ (SELECT)

The most-used SQL command. Retrieves rows matching your criteria.

SELECT {* | col1, col2, ...} FROM table_name [WHERE condition] [ORDER BY column ASC|DESC] [LIMIT n];
🔍

Use Case: Find all laptops priced under $1500

SQL — SELECT
-- Retrieve all columns for all products
SELECT * FROM products;

-- Retrieve specific columns with a filter
SELECT name, price, stock
FROM   products
WHERE  category = 'Laptop'
  AND  price < 1500
ORDER BY price ASC;

1.3 UPDATE

Modifies the values of existing rows. Always use WHERE — without it, every row gets updated!

UPDATE table_name SET col1 = new_val1, col2 = new_val2 WHERE condition;
⚠️

Golden Rule: Always write and test the SELECT query first using the same WHERE clause to see which rows will be affected before running UPDATE or DELETE.

📉

Use Case: Apply a 10% discount to all headphones

SQL — UPDATE
-- First: CHECK what will be affected
SELECT name, price FROM products
WHERE category = 'Headphones';

-- Then: Apply the 10% discount
UPDATE products
SET   price = price * 0.90,
      updated_at = CURRENT_TIMESTAMP
WHERE category = 'Headphones';

1.4 DELETE

Removes rows from a table. The table structure remains intact. Use TRUNCATE to delete all rows faster.

DELETE FROM table_name WHERE condition; -- Remove ALL rows (no WHERE needed, but use with caution): TRUNCATE TABLE table_name;
🗑️

Use Case: Remove discontinued products (stock = 0)

SQL — DELETE
-- Delete out-of-stock discontinued products
DELETE FROM products
WHERE  stock = 0
  AND  is_discontinued = TRUE;

GROUP BY & Aggregates

ANALYTICS

GROUP BY collapses rows with the same value in a column into a single summary row. Paired with aggregate functions, it's how you compute totals, averages, counts per group.

Aggregate Functions

FunctionPurposeNote
COUNT(*)Count all rows in groupIncludes NULLs
COUNT(col)Count non-null valuesIgnores NULLs
SUM(col)Total of numeric valuesIgnores NULLs
AVG(col)Mean valueIgnores NULLs
MIN(col)Smallest valueWorks on text too
MAX(col)Largest valueWorks on text too
GROUP_CONCAT(col)Concatenate values in groupSQLite / MySQL
SELECT group_col, AGG_FUNC(metric_col) AS alias FROM table [WHERE row_filter] ← filters rows BEFORE grouping GROUP BY group_col [HAVING group_filter] ← filters groups AFTER aggregation [ORDER BY alias DESC];
⚡

WHERE vs HAVING: Use WHERE to filter individual rows before grouping. Use HAVING to filter groups after aggregation. You cannot use aggregate functions in WHERE.

📊

Use Case: Sales report by product category

SQL — GROUP BY
-- Total revenue per category (only delivered orders)
SELECT
  p.category,
  COUNT(o.order_id)              AS total_orders,
  SUM(o.quantity)               AS units_sold,
  SUM(o.quantity * p.price)     AS revenue,
  ROUND(AVG(p.price), 2)        AS avg_price
FROM        orders o
INNER JOIN  products p ON o.product_id = p.product_id
WHERE  o.status = 'delivered'            -- filter rows first
GROUP BY p.category
HAVING SUM(o.quantity * p.price) > 100  -- only categories with >$100 revenue
ORDER BY revenue DESC;

JOINs

RELATIONS

JOINs combine rows from two or more tables based on a related column. They're how you query data spread across normalized tables.

INNER JOIN

Only matching rows in both tables

LEFT JOIN

All left rows + matching right rows (NULLs if no match)

RIGHT JOIN

All right rows + matching left rows (NULLs if no match)

FULL OUTER JOIN

All rows from both tables, NULLs for non-matches

SELF JOIN

Join a table with itself (for hierarchies, pairs)

CROSS JOIN

Every row × every row (Cartesian product)

SELECT columns FROM table_A AS a {INNER | LEFT | RIGHT | FULL OUTER} JOIN table_B AS b ON a.key = b.key WHERE condition;
🔗

Use Case: Full order report with customer & product details

SQL — Multi-table JOIN
-- INNER JOIN: only orders that have matching customer and product
SELECT
  o.order_id,
  c.full_name         AS customer,
  c.city,
  p.name              AS product,
  p.category,
  o.quantity,
  p.price,
  (o.quantity * p.price) AS line_total,
  o.status,
  o.order_date
FROM       orders    o
INNER JOIN customers c ON o.customer_id = c.customer_id
INNER JOIN products  p ON o.product_id  = p.product_id
ORDER BY o.order_date DESC;

-- LEFT JOIN: find customers who have NEVER placed an order
SELECT
  c.full_name,
  c.email,
  o.order_id   -- NULL for customers with no orders
FROM      customers c
LEFT JOIN orders    o ON c.customer_id = o.customer_id
WHERE o.order_id IS NULL;  -- the NULL trick to find non-matching left rows

Sub-Queries

NESTING

A sub-query (or nested query) is a SELECT statement embedded inside another SQL statement. The inner query runs first and its result is used by the outer query.

Scalar Sub-query

Returns a single value (one row, one column). Used in SELECT or WHERE.

Row Sub-query

Returns a single row with multiple columns. Used in WHERE with row constructors.

Table Sub-query

Returns multiple rows/columns. Used in FROM (inline view) or with IN/EXISTS.

SELECT col FROM table WHERE col IN ( SELECT col ← inner query runs first FROM other_table WHERE condition );
🔍

Use Case: Find products that cost more than the average price

SQL — Sub-queries
-- Scalar sub-query in WHERE
SELECT name, price
FROM  products
WHERE price > (
  SELECT AVG(price) FROM products
);

-- Sub-query in SELECT (show avg alongside each product)
SELECT
  name,
  price,
  ROUND((SELECT AVG(price) FROM products), 2) AS avg_price,
  price - (SELECT AVG(price) FROM products)    AS vs_average
FROM products;

-- Correlated sub-query: reference the outer query (runs once per row)
-- Find customers who spent more than $1000 total
SELECT c.full_name, c.city
FROM   customers c
WHERE (SELECT SUM(o.quantity * p.price)
       FROM       orders   o
       INNER JOIN products p ON o.product_id = p.product_id
       WHERE o.customer_id = c.customer_id    ← references outer c
      ) > 1000;

-- EXISTS: find customers who have placed at least one order
SELECT full_name, email
FROM   customers c
WHERE EXISTS (
  SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id
);

-- Derived table (sub-query in FROM)
SELECT category, ROUND(avg_price, 2) AS avg_price
FROM (
  SELECT category, AVG(price) AS avg_price
  FROM   products
  GROUP BY category
) cat_summary
WHERE avg_price > 300;
💡

EXISTS vs IN: Use EXISTS when the sub-query returns many rows (it short-circuits on first match). Use IN for small, static lists. For correlated queries, EXISTS is typically faster.

7.2 Correlated Sub-queries — Deep Dive

A correlated sub-query is a sub-query that references a column from the outer query. Unlike a regular sub-query (which runs once and hands its result to the outer query), a correlated sub-query re-executes once for every row the outer query processes.

Regular Sub-query

Runs once. Result is a fixed set that the outer query uses.

① Inner query executes → produces result set
② Outer query filters using that fixed result

Correlated Sub-query

Runs N times — once per outer row. Each run sees a different outer value.

① Outer query fetches row 1
② Inner query runs with row 1's value
③ Outer query fetches row 2 → inner reruns…
SELECT outer_col FROM outer_table AS o ← alias the outer table WHERE expression = ( SELECT AGG(inner_col) FROM inner_table AS i WHERE i.key = o.key ← reference to outer alias — this is the correlation );
🔑

The correlation clause is the WHERE inside the sub-query that ties it to the outer row — e.g. WHERE o.customer_id = c.customer_id. Remove it and the sub-query becomes independent (non-correlated). That one line is what makes it correlated.

📍

Use Case 1 — WHERE: Customers whose total spend exceeds their city's average

For every customer row, the inner query calculates the average spend only for customers in that same city — a different calculation for each row.

SQL — Correlated sub-query in WHERE
-- Step 1: helper CTE to pre-compute each customer's total spend
WITH spend AS (
  SELECT
    o.customer_id,
    SUM(o.quantity * p.price) AS total
  FROM       orders   o
  INNER JOIN products p ON o.product_id = p.product_id
  GROUP BY   o.customer_id
)
-- Step 2: correlated sub-query compares each customer's spend
-- to the average spend of all customers in the SAME city
SELECT
  c.full_name,
  c.city,
  s.total                        AS my_spend,
  ROUND((SELECT AVG(s2.total)   -- ← city average (inner query)
         FROM       spend     s2
         INNER JOIN customers c2 ON s2.customer_id = c2.customer_id
         WHERE      c2.city = c.city  -- ← correlation: same city as outer
        ), 2)                     AS city_avg
FROM       spend     s
INNER JOIN customers c ON s.customer_id = c.customer_id
WHERE s.total > (
  SELECT AVG(s2.total)
  FROM       spend     s2
  INNER JOIN customers c2 ON s2.customer_id = c2.customer_id
  WHERE      c2.city = c.city  -- ← re-evaluated for every outer row
)
ORDER BY s.total DESC;
📊

Use Case 2 — SELECT: Show each product's price vs its category average

The correlated sub-query sits in the SELECT list and is evaluated per row to produce an inline derived column — no JOIN needed.

SQL — Correlated sub-query in SELECT
SELECT
  p.name,
  p.category,
  p.price,
  ROUND((SELECT AVG(p2.price)    -- inner query: avg FOR THIS category
         FROM  products p2
         WHERE p2.category = p.category  -- ← correlation
        ), 2)                           AS category_avg,
  ROUND(p.price - (SELECT AVG(p2.price)
                   FROM  products p2
                   WHERE p2.category = p.category), 2)
                                        AS vs_cat_avg,
  CASE
    WHEN p.price > (SELECT AVG(p2.price) FROM products p2
                    WHERE p2.category = p.category)
    THEN 'Above avg'
    ELSE 'Below avg'
  END                                  AS position
FROM products p
ORDER BY p.category, p.price DESC;

Use Case 3 — UPDATE: Set a flag column based on correlated logic

Correlated sub-queries work inside UPDATE SET too — each row's update value is computed relative to other rows.

SQL — Correlated sub-query in UPDATE
-- Assume we added a column: ALTER TABLE products ADD COLUMN is_priciest_in_cat INTEGER DEFAULT 0;
-- Mark each product that is the most expensive in its category
UPDATE products p
SET    is_priciest_in_cat = (
  CASE WHEN p.price = (
    SELECT MAX(p2.price)
    FROM   products p2
    WHERE  p2.category = p.category  -- ← correlation
  ) THEN 1 ELSE 0 END
);

7.3 Execution Model — How the Engine Runs a Correlated Sub-query

  1. Outer query fetches the first row from customers (e.g., customer_id = 1, Priya Sharma).
  2. The engine substitutes c.customer_id = 1 into the inner query and executes it, returning Priya's total spend.
  3. The outer WHERE evaluates the result (e.g., 2099.96 > 1000 → TRUE → row included).
  4. Outer query moves to the next row (customer_id = 2, Rahul). Inner query re-executes with the new value.
  5. This repeats for every outer row. Total inner executions = number of outer rows.
⚡

Performance: Because the inner query runs N times, correlated sub-queries on large tables can be slow. Always check if an equivalent JOIN + GROUP BY or a CTE + JOIN would be faster. Correlated sub-queries in SELECT are especially costly — consider LEFT JOIN instead.

🎯

Rule of thumb: If you find yourself writing a sub-query that references the outer table, ask — "Can I express this as a JOIN with a GROUP BY or a Window Function?" If yes, prefer that. Use correlated sub-queries when the logic is clearest written that way and the table is small-to-medium sized.

EXISTS, ALL & ANY

SET OPS

These three operators work alongside sub-queries to compare a value against a set of results. They let you express powerful "does something exist?" or "is this value bigger than everything in that set?" logic in a single readable clause.

✅

EXISTS

Returns TRUE if the sub-query produces at least one row. Stops scanning the moment it finds a match.

🌐

ALL

Returns TRUE only if the condition holds for every single row in the sub-query result set.

⚡

ANY / SOME

Returns TRUE if the condition holds for at least one row in the sub-query result. SOME is an alias for ANY.

8.1 EXISTS

EXISTS checks whether the correlated sub-query returns any rows at all. It doesn't care about the actual values — just presence or absence. This makes it ideal for membership checks and anti-joins.

SELECT col FROM table_A AS a WHERE [NOT] EXISTS ( SELECT 1 ← convention: SELECT 1, value irrelevant FROM table_B AS b WHERE b.key = a.key ← correlated to outer row );
⚡

Performance tip: EXISTS short-circuits — as soon as the inner query finds one matching row, it stops and returns TRUE. With IN, the engine must materialise the entire sub-query result first. For large datasets, EXISTS is almost always faster for membership tests.

🛍️

Use Case: Find customers who have placed at least one order vs. those who haven't

SQL — EXISTS / NOT EXISTS
-- Customers who HAVE placed at least one order
SELECT c.full_name, c.city
FROM   customers c
WHERE EXISTS (
  SELECT 1
  FROM   orders o
  WHERE  o.customer_id = c.customer_id   -- correlated to outer c
);

-- Customers who have NEVER placed an order (anti-join via NOT EXISTS)
SELECT c.full_name, c.city
FROM   customers c
WHERE NOT EXISTS (
  SELECT 1
  FROM   orders o
  WHERE  o.customer_id = c.customer_id
);

-- EXISTS with multiple conditions:
-- Customers who ordered a Laptop AND had it delivered
SELECT c.full_name
FROM   customers c
WHERE EXISTS (
  SELECT 1
  FROM        orders   o
  INNER JOIN  products p ON o.product_id = p.product_id
  WHERE  o.customer_id = c.customer_id
    AND  p.category     = 'Laptop'
    AND  o.status       = 'delivered'
);

8.2 ALL

ALL compares a value against every row returned by the sub-query. The condition must hold for every row. Think of it as "is X greater than the maximum?" expressed differently.

SELECT col FROM table WHERE value {> | < | >= | <= | = | <>} ALL ( SELECT col FROM other_table WHERE ... );
ExpressionEquivalent toMeaning
x > ALL (subq)x > MAX(subq)x is greater than every value
x < ALL (subq)x < MIN(subq)x is less than every value
x = ALL (subq)all values equal xx equals every value (rare)
x <> ALL (subq)x NOT IN (subq)x matches none of the values
💰

Use Case: Find the most expensive product — more expensive than all others

SQL — ALL
-- Products whose price is greater than ALL other products
-- i.e., the single most expensive product
SELECT name, price
FROM   products
WHERE  price > ALL (
  SELECT price
  FROM   products p2
  WHERE  p2.product_id <> products.product_id  -- exclude self
);

-- Products more expensive than ALL products in the 'Keyboard' category
SELECT name, price, category
FROM   products
WHERE  price > ALL (
  SELECT price
  FROM   products
  WHERE  category = 'Keyboard'
)
ORDER BY price ASC;

-- Equivalent using MAX (same result, often more readable)
SELECT name, price, category
FROM   products
WHERE  price > (SELECT MAX(price) FROM products WHERE category = 'Keyboard')
ORDER BY price ASC;

⚠️

NULL trap: If the sub-query returns even one NULL, > ALL evaluates to UNKNOWN (not TRUE) for every comparison — and the outer row is excluded. Always filter NULLs in the sub-query: add WHERE col IS NOT NULL.

8.3 ANY / SOME

ANY (alias SOME) passes if the condition is true for at least one row in the sub-query. It's logically equivalent to EXISTS for equality checks, but more expressive with <, >, etc.

SELECT col FROM table WHERE value {> | < | >= | <= | = | <>} ANY ( SELECT col FROM other_table WHERE ... );
ExpressionEquivalent toMeaning
x > ANY (subq)x > MIN(subq)x is greater than at least one value
x < ANY (subq)x < MAX(subq)x is less than at least one value
x = ANY (subq)x IN (subq)x matches at least one value
x <> ANY (subq)not all values equal xx differs from at least one value
📦

Use Case: Find products cheaper than at least one product in a given category

SQL — ANY / SOME
-- Products cheaper than AT LEAST ONE laptop
-- i.e., not more expensive than the most affordable laptop
SELECT name, price, category
FROM   products
WHERE  price < ANY (
  SELECT price
  FROM   products
  WHERE  category = 'Laptop'
)
ORDER BY price DESC;

-- = ANY is identical to IN (choose whichever reads better)
-- Find products that have been ordered
SELECT name, price
FROM   products
WHERE  product_id = ANY (
  SELECT product_id FROM orders
);
-- Identical result using IN:
SELECT name, price
FROM   products
WHERE  product_id IN (
  SELECT product_id FROM orders
);

-- Practical: find customers who spent MORE than at least one
-- other customer's total spend (not the bottom spender)
WITH spend AS (
  SELECT o.customer_id, SUM(o.quantity * p.price) AS total
  FROM       orders   o
  INNER JOIN products p ON o.product_id = p.product_id
  GROUP BY   o.customer_id
)
SELECT c.full_name, s.total
FROM       spend     s
INNER JOIN customers c ON s.customer_id = c.customer_id
WHERE s.total > ANY (
  SELECT total FROM spend s2
  WHERE  s2.customer_id <> s.customer_id
)
ORDER BY s.total DESC;

Window Functions

ADVANCED

Window functions perform calculations across a set of rows related to the current row — without collapsing rows like GROUP BY does. They're one of SQL's most powerful analytical features.

🪟

Key Insight: Unlike aggregates with GROUP BY, window functions keep every original row in the output. Each row retains its individual identity and gets a calculated value from its "window" of peers.

SELECT col, WINDOW_FUNC(col) OVER ( [PARTITION BY partition_col] ← divide rows into groups [ORDER BY order_col] ← order within group [ROWS BETWEEN frame_start AND frame_end] ) AS alias FROM table;

Use Case: Rank customers by spend + running revenue total

SQL — Window Functions
-- Customer spend ranking
SELECT
  c.full_name,
  c.city,
  SUM(o.quantity * p.price)              AS total_spend,
  RANK() OVER (ORDER BY SUM(o.quantity * p.price) DESC)
                                         AS spend_rank,
  DENSE_RANK() OVER (ORDER BY SUM(o.quantity * p.price) DESC)
                                         AS dense_rank
FROM       orders    o
INNER JOIN customers c ON o.customer_id = c.customer_id
INNER JOIN products  p ON o.product_id  = p.product_id
GROUP BY c.customer_id, c.full_name, c.city;

-- Running revenue total (cumulative sum ordered by date)
SELECT
  o.order_date,
  p.name,
  (o.quantity * p.price)                                     AS order_value,
  SUM(o.quantity * p.price)
    OVER (ORDER BY o.order_date
          ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)  AS running_total
FROM       orders   o
INNER JOIN products p ON o.product_id = p.product_id
ORDER BY o.order_date;

-- LAG: compare each order to the previous one from same customer
SELECT
  c.full_name,
  o.order_date,
  p.price                                              AS current_price,
  LAG(p.price) OVER
    (PARTITION BY o.customer_id ORDER BY o.order_date) AS prev_price,
  p.price - LAG(p.price) OVER
    (PARTITION BY o.customer_id ORDER BY o.order_date) AS price_change
FROM       orders    o
INNER JOIN customers c ON o.customer_id = c.customer_id
INNER JOIN products  p ON o.product_id  = p.product_id;

CTEs — Common Table Expressions

ADVANCED

A CTE defines a named, temporary result set that you can reference like a table within a single query. They make complex queries readable by breaking them into named, logical steps.

WITH cte_name (optional_col_names) AS ( SELECT ... ← define the CTE ), cte_two AS ( SELECT ... FROM cte_name ← chain CTEs ) SELECT ... FROM cte_two; ← final query uses CTEs

✅ CTE Advantages

Readable, modular, reusable within query, easier to debug step by step

🌀 Recursive CTEs

Can reference themselves — perfect for org charts, category trees, path finding

📈

Use Case: Multi-step customer segmentation pipeline

SQL — CTE Chain
-- Step 1: Calculate total spend per customer
WITH customer_spend AS (
  SELECT
    o.customer_id,
    COUNT(o.order_id)              AS order_count,
    SUM(o.quantity * p.price)     AS total_spend
  FROM       orders   o
  INNER JOIN products p ON o.product_id = p.product_id
  GROUP BY o.customer_id
),

-- Step 2: Segment customers by spend tier
customer_segments AS (
  SELECT
    cs.customer_id,
    c.full_name,
    c.city,
    cs.order_count,
    ROUND(cs.total_spend, 2)    AS total_spend,
    CASE
      WHEN cs.total_spend >= 2000 THEN 'VIP'
      WHEN cs.total_spend >= 500  THEN 'Regular'
      ELSE                          'New'
    END                             AS segment
  FROM       customer_spend cs
  INNER JOIN customers    c ON cs.customer_id = c.customer_id
)

-- Step 3: Final aggregation — segment summary
SELECT
  segment,
  COUNT(*)                AS customer_count,
  ROUND(AVG(total_spend),2) AS avg_spend,
  SUM(total_spend)         AS segment_revenue,
  GROUP_CONCAT(full_name)  AS customers
FROM  customer_segments
GROUP BY segment
ORDER BY avg_spend DESC;

Recursive CTE Example

SQL — Recursive CTE (Number sequence / hierarchy)
-- Generate a sequence of numbers 1–10 without a table
WITH RECURSIVE numbers(n) AS (
  SELECT 1                -- anchor: starting value
  UNION ALL
  SELECT n + 1          -- recursive: each row adds 1
  FROM  numbers
  WHERE n < 10          -- termination condition
)
SELECT n FROM numbers;

Stored Procedures

PROGRAMMATIC

A stored procedure is a named, saved block of SQL code stored in the database itself. Call it by name to re-execute it — like a function in programming. It can accept parameters, include logic (IF/ELSE, loops), and perform multiple operations.

⚡

Performance

Pre-compiled and cached by the DB engine — faster than sending fresh SQL

🔒

Security

Grant EXECUTE permission without exposing underlying tables

♻️

Reusability

Write once, call many times from any app or session

🧩

Modularity

Encapsulate business logic (refunds, approvals) in the database layer

⚠️

Note: SQLite does not support stored procedures. The examples below use MySQL and PostgreSQL syntax. You can test MySQL procedures free at db-fiddle.com or on a local XAMPP install.

-- MySQL Syntax DELIMITER $$ CREATE PROCEDURE proc_name( IN param1 datatype, OUT param2 datatype ) BEGIN -- SQL statements and logic DECLARE var_name datatype; SELECT ...; IF condition THEN ... END IF; END$$ DELIMITER ;
📦

Use Case: Place an order — validate stock, insert order, update inventory

SQL — Stored Procedure (MySQL)
DELIMITER $$

CREATE PROCEDURE place_order(
  IN  p_customer_id INT,
  IN  p_product_id  INT,
  IN  p_quantity    INT,
  OUT p_result      VARCHAR(100)
)
BEGIN
  DECLARE v_stock   INT;
  DECLARE v_price   DECIMAL(10,2);
  DECLARE v_orderid INT;

  -- Check current stock
  SELECT stock, price
  INTO   v_stock, v_price
  FROM   products
  WHERE  product_id = p_product_id;

  IF v_stock IS NULL THEN
    SET p_result = 'ERROR: Product not found';
  ELSEIF v_stock < p_quantity THEN
    SET p_result = CONCAT('ERROR: Only ', v_stock, ' units available');
  ELSE
    -- Get next order ID
    SELECT COALESCE(MAX(order_id),0)+1 INTO v_orderid FROM orders;

    -- Insert the order
    INSERT INTO orders (order_id, customer_id, product_id, quantity, status)
    VALUES (v_orderid, p_customer_id, p_product_id, p_quantity, 'pending');

    -- Deduct stock
    UPDATE products
    SET    stock = stock - p_quantity
    WHERE  product_id = p_product_id;

    SET p_result = CONCAT('OK: Order #', v_orderid, ' placed. Total: $',
                           p_quantity * v_price);
  END IF;
END$$

DELIMITER ;

-- Call the procedure
CALL place_order(1, 103, 2, @result);
SELECT @result;
-- Output: 'OK: Order #8 placed. Total: $199.98'

-- Drop when no longer needed
DROP PROCEDURE IF EXISTS place_order;
