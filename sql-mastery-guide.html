What is SQL?

INTRO

SQL — Structured Query Language — is the standard language for interacting with relational databases. It was developed at IBM in the 1970s, standardised by ANSI/ISO, and remains the most widely used data language in the world, 50+ years later. Every time you search products on Amazon, check a bank balance, log into an app, or book a flight — dozens of SQL queries fire behind the scenes.

What makes SQL powerful is that it is declarative: you describe what data you want, not how to retrieve it. The database engine's query planner figures out the most efficient execution path. This is in contrast to procedural languages (Python, Java) where you specify step-by-step instructions.

🧠

Mental model: Imagine a relational database as a collection of interconnected spreadsheets (tables). Each spreadsheet has named columns and typed rows. SQL is the language you use to ask questions, combine sheets, and modify data — all in a single, standardised syntax understood by every major database.

SQL Sub-Languages — The Big Picture

SQL is not just one thing. It is composed of five sub-languages, each governing a different aspect of database interaction. Most SQL you write day-to-day is DML — but understanding all five builds a complete mental map.

How the Database Engine Processes a Query

When you run a SQL statement, it doesn't just "run". It passes through a multi-stage pipeline inside the database engine:

  1. Parsing — The SQL text is checked for syntax errors and tokenised into an internal representation (AST — Abstract Syntax Tree).
  2. Semantic Analysis — The engine checks that all table and column names exist, that data types are compatible, and that permissions allow the operation.
  3. Query Optimisation — The query planner generates multiple possible execution plans and picks the cheapest one based on table statistics (row counts, index cardinality, etc.). This is why the same logical query can perform very differently on different data distributions.
  4. Execution — The chosen plan is executed by the storage engine, which reads/writes pages of data from disk or cache.
  5. Result Return — Rows are streamed or buffered back to the client.

Complete Nomenclature Reference

DATABASE
A named container holding a set of related tables, views, functions, and other objects. One server can host hundreds of databases. e.g., ecommerce_db, analytics_db
TABLE
A structured grid of rows and columns, the fundamental storage unit. Each table represents one entity type. e.g., customers, orders, products
ROW / RECORD / TUPLE
A single entry in a table — one instance of the entity. A table with 10,000 customers has 10,000 rows. All three terms mean the same thing.
COLUMN / FIELD / ATTRIBUTE
A named property shared by all rows. Every column has a fixed data type that all values in it must conform to. e.g., customer_name VARCHAR(100)
PRIMARY KEY (PK)
One column (or a combination of columns) that uniquely identifies every row. Cannot be NULL. Cannot have duplicates. The DB creates a unique index on it automatically. e.g., customer_id INT PRIMARY KEY
FOREIGN KEY (FK)
A column whose values must match an existing value in another table's primary key. This enforces referential integrity — you can't have an order for a customer that doesn't exist. e.g., orders.customer_id REFERENCES customers.customer_id
INDEX
A performance data structure (usually a B-tree) maintained alongside a table to speed up lookups on specific columns. Trades faster reads for slower writes and extra storage. Think of it as the book index at the back — you don't read every page to find a topic.
NULL
The explicit absence of a value — distinct from 0, empty string, or FALSE. NULL is not equal to anything, including itself (NULL = NULL is UNKNOWN). Always use IS NULL or IS NOT NULL to test for it, never = NULL.
CONSTRAINT
A rule the DB engine enforces on every INSERT and UPDATE. If a row would violate the constraint, the operation is rejected with an error. Types: PRIMARY KEY, FOREIGN KEY, UNIQUE, NOT NULL, CHECK, DEFAULT.
SCHEMA
In the broad sense: the full blueprint of the database — all tables, columns, types, and relationships. In PostgreSQL specifically, a schema is also a namespace within a database (like a folder), e.g., public.customers vs analytics.customers.
VIEW
A saved SELECT query that behaves like a virtual table. No data is stored — querying a view runs the underlying SELECT in real time. Useful for simplifying complex joins and restricting column access.
TRANSACTION
A group of SQL statements executed as one atomic unit. Either all succeed (COMMIT) or all are undone (ROLLBACK). Guarantees ACID properties: Atomicity, Consistency, Isolation, Durability.
QUERY PLAN
The execution strategy chosen by the query optimiser. Use EXPLAIN (or EXPLAIN ANALYZE in PostgreSQL) to inspect it — the single most important tool for diagnosing slow queries.
CARDINALITY
The number of distinct values in a column. High cardinality (e.g., email — nearly every row different) = index very useful. Low cardinality (e.g., gender — few distinct values) = index less useful.
💡

ACID — the four guarantees of a transaction: Atomicity (all or nothing), Consistency (data rules are always preserved), Isolation (concurrent transactions don't interfere), Durability (committed data survives crashes). Every production database you'll work with honours ACID.

CRUD Operations

DML

CRUD — Create, Read, Update, Delete — are the four fundamental operations on any data store. In SQL they map directly to four DML commands: INSERT, SELECT, UPDATE, DELETE. Every data-driven application you'll ever build is ultimately just variations of these four operations.

➕

Create → INSERT

Add new rows into a table. The table structure must already exist.

🔍

Read → SELECT

Retrieve rows. The most frequent operation — never modifies data.

✏️

Update → UPDATE

Modify existing row values. Always pair with WHERE to avoid full-table updates.

🗑️

Delete → DELETE

Remove rows. Keeps the table structure. Use WHERE or lose everything.

🏪

Running Example: All examples use an e-commerce store with customers, products, and orders tables. The Colab boilerplate in Section 3 creates and seeds all three — paste it once and run every snippet below.

1.1 INSERT — Creating Rows

INSERT adds one or more new rows to a table. You can insert by specifying column names explicitly (recommended — order-independent and future-safe) or by position (fragile — breaks if columns are added/reordered).

-- Single row, named columns (preferred) INSERT INTO table_name (col1, col2, col3) VALUES (val1, val2, val3); -- Multiple rows in one statement (more efficient than N separate INSERTs) INSERT INTO table_name (col1, col2) VALUES (val1a, val2a), (val1b, val2b); -- Insert from a SELECT result (INSERT … SELECT) INSERT INTO target_table (col1, col2) SELECT col1, col2 FROM source_table WHERE condition;
🛍️

Use Case: Register new customers and stock new products

A new customer signs up and three new products arrive in inventory. We also copy last year's delivered orders into an archive table.

SQL — INSERT (all forms)
-- Single customer registration
INSERT INTO customers (customer_id, full_name, email, city)
VALUES (5, 'Meera Nair', 'meera@email.com', 'Chennai');

-- Columns with DEFAULT values can be omitted entirely;
-- joined_date will use CURRENT_DATE automatically
INSERT INTO customers (customer_id, full_name, email)
VALUES (6, 'Arjun Menon', 'arjun@email.com');

-- Insert multiple products in one round-trip (3x faster than 3 INSERTs)
INSERT INTO products (product_id, name, category, price, stock)
VALUES
  (106, 'Anker PowerBank 26800', 'Accessories', 59.99,  300),
  (107, 'Razer DeathAdder V3',  'Mouse',       89.99,  150),
  (108, 'LG 34" UltraWide',     'Monitor',    699.99,  40);

-- INSERT … SELECT: archive all delivered orders from Q1 2024
INSERT INTO orders_archive (order_id, customer_id, product_id, quantity, order_date, status)
SELECT                      order_id, customer_id, product_id, quantity, order_date, status
FROM   orders
WHERE  status     = 'delivered'
  AND  order_date < '2024-04-01';

-- INSERT OR IGNORE (SQLite) / INSERT IGNORE (MySQL):
-- skip the row if a UNIQUE constraint would be violated
INSERT OR IGNORE INTO customers (customer_id, full_name, email)
VALUES (1, 'Priya Sharma', 'priya@email.com');  -- already exists, silently skipped

-- UPSERT (INSERT … ON CONFLICT): insert if new, update if exists (PostgreSQL / SQLite)
-- If customer_id already exists, update their city instead
INSERT INTO customers (customer_id, full_name, email, city)
VALUES (1, 'Priya Sharma', 'priya@email.com', 'Pune')
ON CONFLICT (customer_id) DO UPDATE SET
  city = EXCLUDED.city;
⚠️

Common mistake — inserting without column names: INSERT INTO products VALUES (109, 'Item', 'Cat', 49.99, 10) works today, but breaks silently if a column is added or reordered later. Always name your columns explicitly.

1.2 SELECT — Reading Data

SELECT is the most-used SQL command by a large margin. It never modifies data — it only reads. A SELECT statement describes the shape of the result set you want: which table, which columns, which rows, in what order, how many.

SELECT {* | col1 [AS alias], col2, expression, ...} FROM table_name [AS alias] [WHERE condition] [GROUP BY col] [HAVING group_condition] [ORDER BY col ASC|DESC] [LIMIT n OFFSET m];
🔑

SELECT * in production: Avoid SELECT * in application code. It returns all columns even if you only need two, wasting network bandwidth and memory. It also breaks silently when columns are added or reordered. Always name the columns you need.

🔍

Use Case: Find and sort products — basic to advanced filters

SQL — SELECT (all forms)
-- All columns, all rows (use only for quick inspection)
SELECT * FROM products;

-- Specific columns only
SELECT name, category, price FROM products;

-- Computed column with alias
SELECT
  name,

1.3 UPDATE — Modifying Rows

UPDATE changes the column values of existing rows. The WHERE clause filters which rows get changed. Without a WHERE clause, every single row in the table is updated — a mistake that can be catastrophic in production.

UPDATE table_name SET col1 = new_val1, col2 = new_val2, col3 = col3 + expression ← can reference the current value WHERE condition; ← ALWAYS include this
🛡️

The safety habit: Before running any UPDATE or DELETE, first run the equivalent SELECT with the same WHERE clause to see exactly which rows will be affected. Only then convert SELECT to UPDATE/DELETE. Many engineers keep this as a permanent rule.

✏️

Use Case: Price changes, stock adjustments, order status updates

SQL — UPDATE (all forms)
-- Step 1: always check first — which rows will be affected?
SELECT name, price FROM products WHERE category = 'Headphones';

-- Step 2: apply 10% discount to all headphones
UPDATE products
SET   price = ROUND(price * 0.90, 2)  -- reduce by 10%
WHERE category = 'Headphones';

-- Update multiple columns at once (comma-separate in SET)
UPDATE products
SET
  price = 1199.99,
  stock = stock + 25  -- increment current stock by 25 units
WHERE product_id = 101;

-- Conditional update using CASE WHEN:
-- give different discounts by category
UPDATE products
SET    price = CASE
  WHEN category = 'Laptop'     THEN ROUND(price * 0.95, 2)  -- 5% off
  WHEN category = 'Monitor'   THEN ROUND(price * 0.90, 2)  -- 10% off
  WHEN category = 'Keyboard'  THEN ROUND(price * 0.85, 2)  -- 15% off
  ELSE price  -- no change for other categories
END;

-- Update from another table (UPDATE … JOIN in MySQL)
UPDATE orders o
INNER JOIN customers c ON o.customer_id = c.customer_id
SET    o.status = 'priority'
WHERE  c.city = 'Bengaluru'
  AND  o.status = 'pending';  -- only pending Bengaluru orders

-- Equivalent in PostgreSQL (uses FROM clause)
UPDATE orders o
SET    status = 'priority'
FROM   customers c
WHERE  o.customer_id = c.customer_id
  AND  c.city = 'Bengaluru'
  AND  o.status = 'pending';

1.4 DELETE — Removing Rows

DELETE removes rows from a table. The table structure, columns, and indexes remain intact. A DELETE without WHERE removes every row (equivalent to TRUNCATE but slower and transactional — see Section 2 for the comparison).

DELETE FROM table_name WHERE condition; ← ALWAYS include this -- Delete with a JOIN (MySQL) — delete rows from table A based on table B DELETE a FROM table_A AS a INNER JOIN table_B AS b ON a.id = b.id WHERE condition;
🗑️

Use Case: Remove old data, duplicates, and orphaned records

SQL — DELETE (all forms)
-- Delete a single specific row by PK (safest)
DELETE FROM products WHERE product_id = 108;

-- Delete all out-of-stock AND discontinued products
DELETE FROM products
WHERE  stock = 0
  AND  is_discontinued = TRUE;

-- Delete old cancelled orders (data retention cleanup)
DELETE FROM orders
WHERE  status     = 'cancelled'
  AND  order_date < '2023-01-01';

-- Delete duplicate emails — keep the row with the lowest customer_id
DELETE FROM customers
WHERE customer_id NOT IN (
  SELECT MIN(customer_id)
  FROM   customers
  GROUP BY email           -- one MIN per unique email
);

-- Soft delete pattern: mark as deleted instead of removing
-- (preferred in production — preserves audit trail)
ALTER TABLE customers ADD COLUMN deleted_at TIMESTAMP;

UPDATE customers
SET    deleted_at = CURRENT_TIMESTAMP
WHERE  customer_id = 6;  -- "deleted" but row still exists

-- Then exclude soft-deleted rows in every query
SELECT * FROM customers WHERE deleted_at IS NULL;
💡

Soft delete vs hard delete: In most production systems, data is never truly deleted — it's marked with a deleted_at timestamp. This preserves audit history, supports undo, and avoids FK violations. Hard DELETE is reserved for GDPR/compliance erasure requests or genuine test-data cleanup.

1.5 Transactions — Wrapping CRUD in Safety Nets

A transaction groups multiple DML statements into one atomic unit. Either all succeed, or if anything fails, all are rolled back. Without transactions, a crash halfway through a multi-step operation leaves data in an inconsistent state.

SQL — Transaction (COMMIT / ROLLBACK)
-- Transfer an order from pending to shipped + deduct stock atomically
BEGIN;  -- start the transaction

UPDATE orders
SET    status = 'shipped'
WHERE  order_id = 3;

UPDATE products
SET    stock = stock - 2
WHERE  product_id = 103;

-- If both succeed, commit. Both changes are now permanent.
COMMIT;

-- If anything goes wrong (e.g. stock check failed), undo EVERYTHING
-- ROLLBACK;  ← run this instead of COMMIT if there is an error

-- SAVEPOINT: partial rollback (rollback to a checkpoint, not the full BEGIN)
BEGIN;
INSERT INTO orders (order_id, customer_id, product_id, quantity)
VALUES (8, 2, 106, 1);
SAVEPOINT after_insert;

UPDATE products SET stock = stock - 1 WHERE product_id = 106;
-- something went wrong with the stock update only
ROLLBACK TO after_insert;  -- undo only the UPDATE, keep the INSERT
COMMIT;

Schema & DDL — CREATE, ALTER, DROP, TRUNCATE

DDL

DDL — Data Definition Language — is the set of SQL commands that define, modify, and destroy the structure of your database. While DML (INSERT/UPDATE/DELETE) works on rows of data, DDL works on the containers that hold that data: tables, columns, indexes, views, constraints. Most DDL in traditional databases is auto-committed and cannot be rolled back, so it demands extra care.

🏗️

CREATE

Build new tables, indexes, views, sequences, databases

✏️

ALTER

Evolve existing table structure — add/drop/rename columns, change types, manage constraints

🗑️

DROP

Permanently delete an object and all of its data. No undo.

⚡

TRUNCATE

Wipe all rows instantly, keep the table shell. Far faster than DELETE.

2.1 Data Types — Choosing the Right One

Choosing the right data type is the most consequential schema decision you make. The wrong type wastes storage, silently truncates data, causes comparison bugs (comparing number stored as text), and prevents valid range indexes. The right type lets the engine validate data automatically and optimise storage and lookups.

Use Case: Build the complete e-commerce schema with indexes and a view

A well-designed schema anticipates query patterns. We create indexes on the columns that will appear in WHERE clauses and JOIN conditions most often, and a view that abstracts the common join for the reporting team.

SQL — CREATE (full schema)
-- Database
CREATE DATABASE IF NOT EXISTS ecommerce
  CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;  -- MySQL: full Unicode support

-- 1. Customers
CREATE TABLE IF NOT EXISTS customers (
  customer_id   INT           PRIMARY KEY,
  full_name     VARCHAR(100)  NOT NULL,
  email         VARCHAR(150)  NOT NULL UNIQUE,
  phone         VARCHAR(20),                      -- nullable: optional
  city          VARCHAR(50),
  joined_date   DATE          NOT NULL DEFAULT (CURRENT_DATE),
  is_active     BOOLEAN       NOT NULL DEFAULT TRUE
);

-- 2. Products with a CHECK on price and stock
CREATE TABLE IF NOT EXISTS products (
  product_id    INT            PRIMARY KEY,
  name          VARCHAR(200)  NOT NULL,
  category      VARCHAR(50)   NOT NULL,
  price         DECIMAL(10,2) NOT NULL,
  stock         INT            NOT NULL DEFAULT 0,
  CONSTRAINT chk_price CHECK (price > 0),          -- named constraint
  CONSTRAINT chk_stock CHECK (stock >= 0)
);

-- 3. Orders — child of both customers and products
CREATE TABLE IF NOT EXISTS orders (
  order_id      INT           PRIMARY KEY,
  customer_id   INT           NOT NULL,
  product_id    INT           NOT NULL,
  quantity      INT           NOT NULL DEFAULT 1,
  order_date    TIMESTAMP     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  status        VARCHAR(20)  NOT NULL DEFAULT 'pending',
  CONSTRAINT chk_qty    CHECK (quantity > 0),
  CONSTRAINT chk_status CHECK (status IN ('pending','shipped','delivered','cancelled')),
  CONSTRAINT fk_cust FOREIGN KEY (customer_id)
      REFERENCES customers(customer_id) ON DELETE RESTRICT ON UPDATE CASCADE,
  CONSTRAINT fk_prod FOREIGN KEY (product_id)
      REFERENCES products(product_id)  ON DELETE RESTRICT ON UPDATE CASCADE
);

-- 4. Indexes on FK columns (DB doesn't auto-create these for FK targets)
CREATE INDEX idx_orders_customer ON orders (customer_id);
CREATE INDEX idx_orders_product  ON orders (product_id);
CREATE INDEX idx_orders_date     ON orders (order_date);   -- for date-range queries
CREATE INDEX idx_products_cat    ON products (category);   -- for category filters

-- 5. View: join orders + customers + products — reporting team uses this
CREATE OR REPLACE VIEW v_order_detail AS
  SELECT
    o.order_id,
    o.order_date,
    c.full_name      AS customer,
    c.city,
    p.name           AS product,
    p.category,
    o.quantity,
    p.price,
    (o.quantity * p.price) AS line_total,
    o.status
  FROM       orders    o
  INNER JOIN customers c ON o.customer_id = c.customer_id
  INNER JOIN products  p ON o.product_id  = p.product_id;

-- Now the reporting team can just write:
SELECT * FROM v_order_detail WHERE city = 'Bengaluru';

-- 6. CREATE TABLE AS SELECT — snapshot of premium products for a campaign
CREATE TABLE campaign_premium_products AS
  SELECT product_id, name, category, price
  FROM   products
  WHERE  price > 500;
-- Note: CTAS copies data at this moment but does NOT copy indexes or constraints

2.4 ALTER — Evolving Table Structure

Schemas evolve. Business requirements change. ALTER TABLE modifies an existing table without losing its data. It is how you add new features (new columns), fix mistakes (rename a column, change its type), improve performance (add a constraint or index), or clean up (drop unused columns).

In production, ALTER TABLE on large tables can be dangerous — adding a column on a 500-million-row table can lock the table for hours in older MySQL. Modern databases (PostgreSQL 11+, MySQL 8+, cloud databases) handle most ALTERs online (without locks), but always test first in a staging environment.

ALTER TABLE table_name -- Column operations ADD COLUMN col type [constraint] | DROP COLUMN [IF EXISTS] col | RENAME COLUMN old_col TO new_col | MODIFY COLUMN col new_type [constraint] ← MySQL | ALTER COLUMN col TYPE new_type ← PostgreSQL | -- Constraint operations ADD CONSTRAINT name constraint_def | DROP CONSTRAINT [IF EXISTS] name | -- Table-level RENAME TO new_table_name;
✏️

Use Case: Six months of schema evolution on the customers table

Real schemas evolve iteratively. Here's a timeline of changes a growing e-commerce company might make — and the exact ALTER statements that implement them.

SQL — ALTER TABLE (complete evolution timeline)
-- Month 1: Add loyalty programme columns
ALTER TABLE customers
  ADD COLUMN loyalty_points INT          NOT NULL DEFAULT 0,
  ADD COLUMN tier           VARCHAR(20) NOT NULL DEFAULT 'Bronze';
-- All existing rows instantly get loyalty_points=0, tier='Bronze'

-- Month 2: Enforce tier values with a CHECK constraint
ALTER TABLE customers
  ADD CONSTRAINT chk_tier
    CHECK (tier IN ('Bronze', 'Silver', 'Gold', 'Platinum'));

-- Month 3: "phone" is always a mobile now — rename it for clarity
ALTER TABLE customers
  RENAME COLUMN phone TO mobile;

-- Month 4: mobile numbers are longer internationally — widen the type
ALTER TABLE customers
  MODIFY COLUMN mobile VARCHAR(20);  -- MySQL
-- ALTER TABLE customers ALTER COLUMN mobile TYPE VARCHAR(20);  -- PostgreSQL

-- Month 5: Add an audit timestamp — when was the record last changed?
ALTER TABLE customers
  ADD COLUMN updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
             ON UPDATE CURRENT_TIMESTAMP;  -- MySQL auto-updates on any UPDATE

-- Month 6: Referral system — a customer can refer another customer
-- Self-referential FK: referred_by points to another customer_id
ALTER TABLE customers
  ADD COLUMN referred_by INT,
  ADD CONSTRAINT fk_referral
    FOREIGN KEY (referred_by) REFERENCES customers(customer_id)
    ON DELETE SET NULL;  -- if the referrer is deleted, just set to NULL

-- Remove the loyalty constraint (business decided tier is free-form text)
ALTER TABLE customers
  DROP CONSTRAINT chk_tier;  -- PostgreSQL
-- ALTER TABLE customers DROP CHECK chk_tier;  -- MySQL

-- Rename the table (company re-brand)
ALTER TABLE customers
  RENAME TO members;

-- Set a new default on an existing column (PostgreSQL)
ALTER TABLE members
  ALTER COLUMN tier SET DEFAULT 'Silver';

-- Drop a column that's no longer used
ALTER TABLE members
  DROP COLUMN IF EXISTS loyalty_points;

Zero-downtime ALTER strategy (expand-contract): On large production tables, instead of ALTER TABLE MODIFY COLUMN (which can lock the table): (1) ADD the new column alongside the old one, (2) backfill it in batches, (3) deploy app code to write to both, (4) verify consistency, (5) switch app to read the new column, (6) DROP the old column. This is how big companies run schema migrations without downtime.

2.6 DROP — Deleting Objects Permanently

DROP permanently removes a database object — its structure and all data stored in it. Unlike DELETE (which removes rows), DROP removes the table itself. There is no UNDO, no recycle bin, no rollback in most databases. Run it only when you are absolutely certain.

DROP TABLE [IF EXISTS] table_name [CASCADE | RESTRICT]; DROP DATABASE [IF EXISTS] db_name; DROP INDEX [IF EXISTS] idx_name [ON table]; ← ON table: MySQL only DROP VIEW [IF EXISTS] view_name; DROP PROCEDURE [IF EXISTS] proc_name;
🔴

CASCADE vs RESTRICT: CASCADE drops the table AND all objects that depend on it (FK constraints from other tables, views, etc.). RESTRICT (the default) refuses to drop if any dependents exist — forcing you to manually clean them up first. In production, always use RESTRICT; use CASCADE only in dev/test teardowns where you know what you're doing.

🗑️

2.7 TRUNCATE — Fast Row Wipeout

TRUNCATE removes all rows from a table in one fast operation, but keeps the table structure, columns, indexes, and constraints intact. Instead of deleting rows one by one (like DELETE does), TRUNCATE deallocates the data pages directly — making it orders of magnitude faster on large tables. A table with 100 million rows: DELETE takes minutes; TRUNCATE takes milliseconds.

The trade-off: TRUNCATE cannot be filtered (no WHERE clause), and in most databases it cannot be rolled back because it's DDL (auto-committed). It also does not fire row-level DELETE triggers.

TRUNCATE TABLE table_name; -- PostgreSQL: truncate multiple tables + reset auto-increment ID counter TRUNCATE TABLE table1, table2 RESTART IDENTITY ← reset sequence/serial back to 1 CASCADE; ← also truncate FK child tables -- MySQL: disable FK checks to truncate parent before child SET FOREIGN_KEY_CHECKS = 0; TRUNCATE TABLE parent_table; SET FOREIGN_KEY_CHECKS = 1;
⚡

Use Case: ETL pipeline — wipe staging tables before each load

In data engineering, staging tables hold raw data for the current ETL batch. Before each run you must wipe them clean. TRUNCATE is the right tool: it's fast, keeps the structure (so you don't have to recreate it), and resets the ID counter so IDs are predictable.

SQL — TRUNCATE (all variants)
-- Basic: wipe the staging table for a fresh load
TRUNCATE TABLE staging_orders;

-- PostgreSQL: wipe + reset the SERIAL/BIGSERIAL ID back to 1
-- Without RESTART IDENTITY, IDs continue from where they left off
TRUNCATE TABLE staging_orders RESTART IDENTITY;

-- PostgreSQL: truncate a parent and all its FK children in one statement
TRUNCATE TABLE customers
  RESTART IDENTITY CASCADE;
-- This also truncates orders (which has FK to customers) automatically

-- MySQL: orders must be truncated before customers (FK child before parent)
-- OR disable FK checks temporarily
SET FOREIGN_KEY_CHECKS = 0;
TRUNCATE TABLE orders;
TRUNCATE TABLE customers;
TRUNCATE TABLE products;
SET FOREIGN_KEY_CHECKS = 1;

-- Practical ETL script pattern (PostgreSQL)
BEGIN;                             -- PostgreSQL allows TRUNCATE in a transaction
TRUNCATE TABLE staging_orders RESTART IDENTITY;
-- ... load new data via COPY or INSERT ...
COMMIT;                            -- commit both the truncate and the new data
-- if loading fails, ROLLBACK restores everything

Syntax & Clauses — Deep Reference

CORE

The SELECT statement is the most complex statement in SQL. It has up to eight distinct clauses, each serving a specific role in filtering, grouping, or ordering the result. Understanding the interaction between clauses — especially their execution order — is the key to writing correct, performant queries.

4.1 SELECT Execution Order vs. Writing Order

This is one of the most important SQL concepts and one of the most commonly misunderstood. The order you write clauses is fixed by SQL syntax. But the order the database executes them is completely different. Failing to understand this leads to confusing errors like "column alias not found in WHERE".

Writing Order (How you type it)

SELECT col, AGG()
FROM table
JOIN other ON …
WHERE row_filter
GROUP BY col
HAVING group_filter
ORDER BY col
LIMIT n OFFSET m

Execution Order (How the engine runs it)

① FROM — identify source table(s)
② JOIN — combine tables
③ WHERE — filter rows
④ GROUP BY — group filtered rows
⑤ HAVING — filter groups
⑥ SELECT — compute output columns
⑦ ORDER BY — sort results
⑧ LIMIT/OFFSET — paginate
❗

Alias visibility rules from the execution order:
❌ WHERE total > 100 — fails if total is a SELECT alias. WHERE runs before SELECT.
✅ HAVING total > 100 — works in PostgreSQL (HAVING can see SELECT aliases in some DBs).
✅ ORDER BY total — works. ORDER BY runs after SELECT, so aliases are visible.
✅ WHERE price * quantity > 100 — always works. Reference source columns, not aliases, in WHERE.

4.3 CASE WHEN — Conditional Logic Inside SQL

CASE WHEN is SQL's if-else. It can appear in SELECT (to compute derived columns), WHERE (to conditionally filter), ORDER BY (to sort with custom logic), and UPDATE SET (to update different rows differently). There are two forms:

-- Searched CASE (most flexible — each WHEN has its own condition) CASE WHEN condition1 THEN result1 WHEN condition2 THEN result2 ELSE default_result ← if omitted, NULL is returned END -- Simple CASE (compares one expression against a list of values) CASE column WHEN 'val1' THEN result1 WHEN 'val2' THEN result2 ELSE default_result END
SQL — CASE WHEN in all positions
-- In SELECT: create price tier label
SELECT name, price,
  CASE
    WHEN price <  100  THEN 'Budget'
    WHEN price <  500  THEN 'Mid-range'
    WHEN price < 1000  THEN 'Premium'
    ELSE                    'Luxury'
  END AS price_tier
FROM products;

-- In ORDER BY: custom sort logic (pending first, then shipped, then delivered)
SELECT order_id, status
FROM   orders
ORDER BY
  CASE status
    WHEN 'pending'   THEN 1
    WHEN 'shipped'   THEN 2
    WHEN 'delivered' THEN 3
    ELSE                  4
  END;

-- In aggregate: conditional COUNT (pivot-like pattern)
SELECT
  category,
  COUNT(*)                                             AS total_products,
  COUNT(CASE WHEN price < 500  THEN 1 END)            AS affordable,
  COUNT(CASE WHEN price >= 500 THEN 1 END)            AS premium,
  SUM(CASE WHEN stock = 0     THEN 1 ELSE 0 END)    AS out_of_stock
FROM products
GROUP BY category;

Sub-Queries

NESTING

A sub-query (or nested query) is a SELECT statement embedded inside another SQL statement. The inner query runs first and its result is used by the outer query.

Scalar Sub-query

Returns a single value (one row, one column). Used in SELECT or WHERE.

Row Sub-query

Returns a single row with multiple columns. Used in WHERE with row constructors.

Table Sub-query

Returns multiple rows/columns. Used in FROM (inline view) or with IN/EXISTS.

SELECT col FROM table WHERE col IN ( SELECT col ← inner query runs first FROM other_table WHERE condition );
🔍

Use Case: Find products that cost more than the average price

SQL — Sub-queries
-- Scalar sub-query in WHERE
SELECT name, price
FROM  products
WHERE price > (
  SELECT AVG(price) FROM products
);

-- Sub-query in SELECT (show avg alongside each product)
SELECT
  name,
  price,
  ROUND((SELECT AVG(price) FROM products), 2) AS avg_price,
  price - (SELECT AVG(price) FROM products)    AS vs_average
FROM products;

-- Correlated sub-query: reference the outer query (runs once per row)
-- Find customers who spent more than $1000 total
SELECT c.full_name, c.city
FROM   customers c
WHERE (SELECT SUM(o.quantity * p.price)
       FROM       orders   o
       INNER JOIN products p ON o.product_id = p.product_id
       WHERE o.customer_id = c.customer_id    ← references outer c
      ) > 1000;

-- EXISTS: find customers who have placed at least one order
SELECT full_name, email
FROM   customers c
WHERE EXISTS (
  SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id
);

-- Derived table (sub-query in FROM)
SELECT category, ROUND(avg_price, 2) AS avg_price
FROM (
  SELECT category, AVG(price) AS avg_price
  FROM   products
  GROUP BY category
) cat_summary
WHERE avg_price > 300;
💡

EXISTS vs IN: Use EXISTS when the sub-query returns many rows (it short-circuits on first match). Use IN for small, static lists. For correlated queries, EXISTS is typically faster.

7.2 Correlated Sub-queries — Deep Dive

A correlated sub-query is a sub-query that references a column from the outer query. Unlike a regular sub-query (which runs once and hands its result to the outer query), a correlated sub-query re-executes once for every row the outer query processes.

Regular Sub-query

Runs once. Result is a fixed set that the outer query uses.

① Inner query executes → produces result set
② Outer query filters using that fixed result

Correlated Sub-query

Runs N times — once per outer row. Each run sees a different outer value.

① Outer query fetches row 1
② Inner query runs with row 1's value
③ Outer query fetches row 2 → inner reruns…
SELECT outer_col FROM outer_table AS o ← alias the outer table WHERE expression = ( SELECT AGG(inner_col) FROM inner_table AS i WHERE i.key = o.key ← reference to outer alias — this is the correlation );
🔑

The correlation clause is the WHERE inside the sub-query that ties it to the outer row — e.g. WHERE o.customer_id = c.customer_id. Remove it and the sub-query becomes independent (non-correlated). That one line is what makes it correlated.

📍

Use Case 1 — WHERE: Customers whose total spend exceeds their city's average

For every customer row, the inner query calculates the average spend only for customers in that same city — a different calculation for each row.

SQL — Correlated sub-query in WHERE
-- Step 1: helper CTE to pre-compute each customer's total spend
WITH spend AS (
  SELECT
    o.customer_id,
    SUM(o.quantity * p.price) AS total
  FROM       orders   o
  INNER JOIN products p ON o.product_id = p.product_id
  GROUP BY   o.customer_id
)
-- Step 2: correlated sub-query compares each customer's spend
-- to the average spend of all customers in the SAME city
SELECT
  c.full_name,
  c.city,
  s.total                        AS my_spend,
  ROUND((SELECT AVG(s2.total)   -- ← city average (inner query)
         FROM       spend     s2
         INNER JOIN customers c2 ON s2.customer_id = c2.customer_id
         WHERE      c2.city = c.city  -- ← correlation: same city as outer
        ), 2)                     AS city_avg
FROM       spend     s
INNER JOIN customers c ON s.customer_id = c.customer_id
WHERE s.total > (
  SELECT AVG(s2.total)
  FROM       spend     s2
  INNER JOIN customers c2 ON s2.customer_id = c2.customer_id
  WHERE      c2.city = c.city  -- ← re-evaluated for every outer row
)
ORDER BY s.total DESC;
📊

Use Case 2 — SELECT: Show each product's price vs its category average

The correlated sub-query sits in the SELECT list and is evaluated per row to produce an inline derived column — no JOIN needed.

SQL — Correlated sub-query in SELECT
SELECT
  p.name,
  p.category,
  p.price,
  ROUND((SELECT AVG(p2.price)    -- inner query: avg FOR THIS category
         FROM  products p2
         WHERE p2.category = p.category  -- ← correlation
        ), 2)                           AS category_avg,
  ROUND(p.price - (SELECT AVG(p2.price)
                   FROM  products p2
                   WHERE p2.category = p.category), 2)
                                        AS vs_cat_avg,
  CASE
    WHEN p.price > (SELECT AVG(p2.price) FROM products p2
                    WHERE p2.category = p.category)
    THEN 'Above avg'
    ELSE 'Below avg'
  END                                  AS position
FROM products p
ORDER BY p.category, p.price DESC;

Correlated sub-queries work inside UPDATE SET too — each row's update value is computed relative to other rows.

SQL — Correlated sub-query in UPDATE
-- Assume we added a column: ALTER TABLE products ADD COLUMN is_priciest_in_cat INTEGER DEFAULT 0;
-- Mark each product that is the most expensive in its category
UPDATE products p
SET    is_priciest_in_cat = (
  CASE WHEN p.price = (
    SELECT MAX(p2.price)
    FROM   products p2
    WHERE  p2.category = p.category  -- ← correlation
  ) THEN 1 ELSE 0 END
);

7.3 Execution Model — How the Engine Runs a Correlated Sub-query

  1. Outer query fetches the first row from customers (e.g., customer_id = 1, Priya Sharma).
  2. The engine substitutes c.customer_id = 1 into the inner query and executes it, returning Priya's total spend.
  3. The outer WHERE evaluates the result (e.g., 2099.96 > 1000 → TRUE → row included).
  4. Outer query moves to the next row (customer_id = 2, Rahul). Inner query re-executes with the new value.
  5. This repeats for every outer row. Total inner executions = number of outer rows.
⚡

Performance: Because the inner query runs N times, correlated sub-queries on large tables can be slow. Always check if an equivalent JOIN + GROUP BY or a CTE + JOIN would be faster. Correlated sub-queries in SELECT are especially costly — consider LEFT JOIN instead.

EXISTS, ALL & ANY

SET OPS

These three operators work alongside sub-queries to compare a value against a set of results. They let you express powerful "does something exist?" or "is this value bigger than everything in that set?" logic in a single readable clause.

✅

EXISTS

Returns TRUE if the sub-query produces at least one row. Stops scanning the moment it finds a match.

🌐

ALL

Returns TRUE only if the condition holds for every single row in the sub-query result set.

⚡

ANY / SOME

Returns TRUE if the condition holds for at least one row in the sub-query result. SOME is an alias for ANY.

8.1 EXISTS

EXISTS checks whether the correlated sub-query returns any rows at all. It doesn't care about the actual values — just presence or absence. This makes it ideal for membership checks and anti-joins.

SELECT col FROM table_A AS a WHERE [NOT] EXISTS ( SELECT 1 ← convention: SELECT 1, value irrelevant FROM table_B AS b WHERE b.key = a.key ← correlated to outer row );
⚡

Performance tip: EXISTS short-circuits — as soon as the inner query finds one matching row, it stops and returns TRUE. With IN, the engine must materialise the entire sub-query result first. For large datasets, EXISTS is almost always faster for membership tests.

🛍️

Use Case: Find customers who have placed at least one order vs. those who haven't

SQL — EXISTS / NOT EXISTS
-- Customers who HAVE placed at least one order
SELECT c.full_name, c.city
FROM   customers c
WHERE EXISTS (
  SELECT 1
  FROM   orders o
  WHERE  o.customer_id = c.customer_id   -- correlated to outer c
);

-- Customers who have NEVER placed an order (anti-join via NOT EXISTS)
SELECT c.full_name, c.city
FROM   customers c
WHERE NOT EXISTS (
  SELECT 1
  FROM   orders o
  WHERE  o.customer_id = c.customer_id
);

-- EXISTS with multiple conditions:
-- Customers who ordered a Laptop AND had it delivered
SELECT c.full_name
FROM   customers c
WHERE EXISTS (
  SELECT 1
  FROM        orders   o
  INNER JOIN  products p ON o.product_id = p.product_id
  WHERE  o.customer_id = c.customer_id
    AND  p.category     = 'Laptop'
    AND  o.status       = 'delivered'

Window Functions

ADVANCED

Window functions perform calculations across a set of rows related to the current row — without collapsing rows like GROUP BY does. They're one of SQL's most powerful analytical features.

🪟

Key Insight: Unlike aggregates with GROUP BY, window functions keep every original row in the output. Each row retains its individual identity and gets a calculated value from its "window" of peers.

SELECT col, WINDOW_FUNC(col) OVER ( [PARTITION BY partition_col] ← divide rows into groups [ORDER BY order_col] ← order within group [ROWS BETWEEN frame_start AND frame_end] ) AS alias FROM table;

Use Case: Rank customers by spend + running revenue total

SQL — Window Functions
-- Customer spend ranking
SELECT
  c.full_name,
  c.city,
  SUM(o.quantity * p.price)              AS total_spend,
  RANK() OVER (ORDER BY SUM(o.quantity * p.price) DESC)
                                         AS spend_rank,
  DENSE_RANK() OVER (ORDER BY SUM(o.quantity * p.price) DESC)
                                         AS dense_rank
FROM       orders    o
INNER JOIN customers c ON o.customer_id = c.customer_id
INNER JOIN products  p ON o.product_id  = p.product_id
GROUP BY c.customer_id, c.full_name, c.city;

-- Running revenue total (cumulative sum ordered by date)
SELECT
  o.order_date,
  p.name,
  (o.quantity * p.price)                                     AS order_value,
  SUM(o.quantity * p.price)
    OVER (ORDER BY o.order_date
          ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)  AS running_total
FROM       orders   o
INNER JOIN products p ON o.product_id = p.product_id
ORDER BY o.order_date;

-- LAG: compare each order to the previous one from same customer
SELECT
  c.full_name,
  o.order_date,
  p.price                                              AS current_price,
  LAG(p.price) OVER
    (PARTITION BY o.customer_id ORDER BY o.order_date) AS prev_price,
  p.price - LAG(p.price) OVER
    (PARTITION BY o.customer_id ORDER BY o.order_date) AS price_change
FROM       orders    o
INNER JOIN customers c ON o.customer_id = c.customer_id
INNER JOIN products  p ON o.product_id  = p.product_id;

CTEs — Common Table Expressions

ADVANCED

A CTE defines a named, temporary result set that you can reference like a table within a single query. They make complex queries readable by breaking them into named, logical steps.

WITH cte_name (optional_col_names) AS ( SELECT ... ← define the CTE ), cte_two AS ( SELECT ... FROM cte_name ← chain CTEs ) SELECT ... FROM cte_two; ← final query uses CTEs

✅ CTE Advantages

Readable, modular, reusable within query, easier to debug step by step

🌀 Recursive CTEs

Can reference themselves — perfect for org charts, category trees, path finding

📈

Use Case: Multi-step customer segmentation pipeline

SQL — CTE Chain
-- Step 1: Calculate total spend per customer
WITH customer_spend AS (
  SELECT
    o.customer_id,
    COUNT(o.order_id)              AS order_count,
    SUM(o.quantity * p.price)     AS total_spend
  FROM       orders   o
  INNER JOIN products p ON o.product_id = p.product_id
  GROUP BY o.customer_id
),

-- Step 2: Segment customers by spend tier
customer_segments AS (
  SELECT
    cs.customer_id,
    c.full_name,
    c.city,
    cs.order_count,
    ROUND(cs.total_spend, 2)    AS total_spend,
    CASE
      WHEN cs.total_spend >= 2000 THEN 'VIP'
      WHEN cs.total_spend >= 500  THEN 'Regular'
      ELSE                          'New'
    END                             AS segment
  FROM       customer_spend cs
  INNER JOIN customers    c ON cs.customer_id = c.customer_id
)

-- Step 3: Final aggregation — segment summary
SELECT
  segment,
  COUNT(*)                AS customer_count,
  ROUND(AVG(total_spend),2) AS avg_spend,
  SUM(total_spend)         AS segment_revenue,
  GROUP_CONCAT(full_name)  AS customers
FROM  customer_segments
GROUP BY segment
ORDER BY avg_spend DESC;

Recursive CTE Example

SQL — Recursive CTE (Number sequence / hierarchy)
-- Generate a sequence of numbers 1–10 without a table
WITH RECURSIVE numbers(n) AS (
  SELECT 1                -- anchor: starting value
  UNION ALL
  SELECT n + 1          -- recursive: each row adds 1

  WHERE n < 10          -- termination condition

Stored Procedures

PROGRAMMATIC

A stored procedure is a named, saved block of SQL code stored in the database itself. Call it by name to re-execute it — like a function in programming. It can accept parameters, include logic (IF/ELSE, loops), and perform multiple operations.

⚡

Performance

Pre-compiled and cached by the DB engine — faster than sending fresh SQL

🔒

Security

Grant EXECUTE permission without exposing underlying tables

♻️

Reusability

Write once, call many times from any app or session

🧩

Modularity

Encapsulate business logic (refunds, approvals) in the database layer

⚠️

Note: SQLite does not support stored procedures. The examples below use MySQL and PostgreSQL syntax. You can test MySQL procedures free at db-fiddle.com or on a local XAMPP install.

-- MySQL Syntax DELIMITER $$ CREATE PROCEDURE proc_name( IN param1 datatype, OUT param2 datatype ) BEGIN -- SQL statements and logic DECLARE var_name datatype; SELECT ...; IF condition THEN ... END IF; END$$ DELIMITER ;
📦

Use Case: Place an order — validate stock, insert order, update inventory

SQL — Stored Procedure (MySQL)
DELIMITER $$

CREATE PROCEDURE place_order(
  IN  p_customer_id INT,
  IN  p_product_id  INT,
  IN  p_quantity    INT,
  OUT p_result      VARCHAR(100)
)
BEGIN
  DECLARE v_stock   INT;
  DECLARE v_price   DECIMAL(10,2);
  DECLARE v_orderid INT;

  -- Check current stock
  SELECT stock, price
  INTO   v_stock, v_price
  FROM   products
  WHERE  product_id = p_product_id;

  IF v_stock IS NULL THEN
    SET p_result = 'ERROR: Product not found';
  ELSEIF v_stock < p_quantity THEN
    SET p_result = CONCAT('ERROR: Only ', v_stock, ' units available');
  ELSE
    -- Get next order ID
    SELECT COALESCE(MAX(order_id),0)+1 INTO v_orderid FROM orders;

    -- Insert the order
    INSERT INTO orders (order_id, customer_id, product_id, quantity, status)
    VALUES (v_orderid, p_customer_id, p_product_id, p_quantity, 'pending');

    -- Deduct stock
    UPDATE products
    SET    stock = stock - p_quantity
    WHERE  product_id = p_product_id;

    SET p_result = CONCAT('OK: Order #', v_orderid, ' placed. Total: $',
                           p_quantity * v_price);
  END IF;
END$$

DELIMITER ;

-- Call the procedure
CALL place_order(1, 103, 2, @result);
SELECT @result;
-- Output: 'OK: Order #8 placed. Total: $199.98'

-- Drop when no longer needed
DROP PROCEDURE IF EXISTS place_order;
